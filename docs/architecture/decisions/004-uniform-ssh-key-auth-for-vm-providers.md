# ADR-004: Uniform Key-Based SSH Authentication for VM Providers

Date: 2026-06-15
Status: Accepted
Deciders: Stefan (Product Owner)

## Context and Problem Statement

`create-vm.yml` provisions disposable machines through pluggable providers:
`docker` and `tart` (laptop-local) and `hcloud` and `aws` (cloud). Each provider
writes the new host's connection details into an autogenerated inventory, and
`configure-profile.yml` then connects as the fresh-system bootstrap user
(`admin_user_on_fresh_system`) to set up the steady-state account.

The bootstrap authentication method diverged by provider:

- **Cloud (`hcloud`, `aws`)** registered the operator's SSH key with the
  provider, which injected the **public** key into the bootstrap user's
  `authorized_keys` at boot. The inventory carried no password; connections used
  key authentication via the operator's ambient SSH agent / `~/.ssh`.
- **Local (`docker`, `tart`)** bootstrapped with a **password**: the Docker
  image set a root password, the Tart VM used the cirruslabs `admin` password,
  and the autogenerated inventory carried `ansible_ssh_pass` (driven by
  `sshpass`). No SSH key was ever placed for the bootstrap user.

This divergence produced a defect. `setup-users.yml` writes
`PasswordAuthentication no` and restarts SSH on the first run. For the local
providers, whose bootstrap user authenticated only by password and never
received a key, the **second** run of `configure-profile.yml` could no longer
log in:

```text
root@127.0.0.1: Permission denied (publickey).
fatal: [tatooine]: UNREACHABLE!
```

Cloud hosts were unaffected because their bootstrap user retained the
provider-injected key. The re-run is a normal operation and must succeed.

Decision: **should the bootstrap SSH authentication method be uniform across all
providers, and if so, which method?**

### Scope

The bootstrap-user authentication method for every `create-vm.yml` provider and
the connection details written into the autogenerated inventory. This ADR
narrows the "auth method out of scope / decided per provider" statement in
ADR-003, which it supersedes on that point. ADR-003's host-key **verification**
policy is unchanged.

Out of scope: the host-key verification policy (ADR-003), and the steady-state
`my_ansible_user` account created by `setup-users.yml`.

## Decision Drivers

- **Idempotency (Constitution I).** Re-running a configure on an existing host
  must succeed; the connection identity must be run-invariant.
- **Uniformity / DRY (Constitution XI).** One authentication model across
  providers removes a class of per-provider special-casing and the bugs it
  breeds.
- **Simplicity / YAGNI (Constitution IV).** Prefer the fix that removes the
  divergence over conditional "is this host already configured?" detection.
- **Fail loud (Constitution XII).** A missing or mismatched key must fail at
  create time with an actionable message, not silently at the next configure
  run.

## Considered Options

1. **Uniform key-based bootstrap auth.** Place the operator's public key into the
   local providers' bootstrap user at create time and connect by key, matching
   cloud.
2. **Re-run as the steady-state user.** Detect already-configured hosts and
   connect as `my_ansible_user` instead of the bootstrap user.
3. **Add a key inside `setup-users.yml` only.** Keep password bootstrap, but add
   the key to the bootstrap user before disabling password auth.

## Decision Outcome

Chosen: **Option 1 — uniform key-based bootstrap auth.**

Local providers install `my_ssh_public_key` into the bootstrap user's
`authorized_keys` at create time, over the one-time password channel that already
exists for first contact, using the idempotent `ansible.builtin.authorized_key`
module. The autogenerated inventory then drops `ansible_ssh_pass`; connections
resolve the matching private key from the operator's ambient SSH agent /
`~/.ssh`, exactly as cloud providers already do.

Because public-key placement alone does not prove the controller can offer the
matching **private** key, create runs a key-only login probe
(`PreferredAuthentications=publickey`, `PasswordAuthentication=no`,
`BatchMode=yes`) before writing the inventory. A failed probe aborts create with
an actionable message naming `my_ssh_public_key` (Fail Loud), rather than
deferring the failure to the next configure run.

The Docker image keeps its root password and the `sshpass` preflight remains:
both are needed for the single first-contact connection that delivers the key.
"Uniform key auth" applies to the **configure** path; create still bootstraps
over password for exactly one connection.

### Why not the alternatives

- **Option 2** requires fragile "fresh vs configured" detection or a two-phase
  bootstrap/steady-state split, adds conditional logic, and leaves docker/tart
  on password auth — the divergence stays.
- **Option 3** is the smallest diff but keeps the password bootstrap and
  `sshpass` on the create path, so it fixes the symptom without removing the
  divergence this decision targets. Retained only as a fallback if uniform auth
  is ever de-scoped.

### Consequences

- One authentication model across all four providers; the configure path is
  key-only everywhere.
- The Docker image and Tart VM carry the operator's **public** key (safe to
  expose) instead of relying on a password for configure.
- Pre-existing docker/tart hosts created under the old code still carry
  `ansible_ssh_pass` in the autogenerated inventory until destroyed and
  recreated; they do not migrate in place.
- A create now fails loudly if the controller cannot offer the private key
  matching `my_ssh_public_key`, surfacing key-setup problems early.
- ADR-003's host-key verification policy is unaffected; only its
  auth-method-out-of-scope note is superseded here.
