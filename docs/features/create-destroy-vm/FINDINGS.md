## docs/features/create-destroy-vm/README.md

**Finding:** Can we generalize the documentation so that it covers both
providers equally? Alternatively, document the provider that is easier to use
and add a note whenever the other provider deviates. I would assume that both
providers are handled similarly, isn't that true?

**Reply:** Not quite — the providers share the same high-level shape (pool
based hostname assignment, create → wait-for-SSH → write autogenerated
inventory, destroy → stop/remove → clean inventory), but diverge in several
details: requirements (macOS+`tart` CLI vs any Docker host), base OS image
(`ghcr.io/cirruslabs/ubuntu:24.04` vs `ubuntu:24.10` container), resource
variables (`vm_cpus`/`vm_memory_mb`/`vm_disk_size_gb` vs
`docker_vm_cpus`/`docker_vm_memory`, different defaults), credentials
(`admin`/`admin` vs `root`/`docker`), hostname pools (separate files/themes),
and connection details (VM IP:22 vs `127.0.0.1`:dynamic-port).

Given that, I'd favor your second option: keep `tart` as the primary
documented provider (current default), and fold `quickstart.md`'s Docker
content into the README as short "Docker differs: ..." notes per section
(Requirements, defaults, Credentials, Hostname Pool, Connection) instead of a
fully separate doc. A single doc trying to explain both providers
symmetrically for every section would add noise for the common case (tart,
default).

## playbooks/tasks/create/docker.yml

### Structure parity, sshpass check, shared code, podman

**Finding:** Shouldn't the playbook contain similar tasks and structure as
playbooks/tasks/create/tart.yml? I would expect that `sshpass` is checked. I
wonder what an expert ansible developer would think of extracting the shared
code (duplicates) into a separate file - consider pros and cons carefully.
Think hard.

The next feature coming into my mind is adding `podman` as a platform. Think
hard to judge the extra effort - is this trivial? What caveats do exist?

**Reply:**

**sshpass check** — real gap. `tart.yml` checks for `sshpass` and fails with
an actionable message before `wait_for_connection` (which needs it for
`ansible_ssh_pass`). `docker.yml` skips this and would fail with a less clear
error if `sshpass` is missing. Should be added to `docker.yml` for parity.

**Shared-code extraction** — both files share roughly 60-70% of their
structure: load pool, load+validate credentials, load/init autogenerated
inventory, compute used hostnames, fail-if-exhausted, pick next hostname,
build/write updated inventory, and a rescue block that tears down on failure.
The differing 30-40% is the provider-specific "create the thing" step (tart
clone/set/run + poll IP vs docker build-once-guard + run + discover published
port) and the per-host inventory fields (`ansible_host`/IP vs
`127.0.0.1`/dynamic port, user `admin` vs `root`).

Pros of extracting: single source for bug fixes (e.g. the sshpass check
above would only need writing once), and it lowers the bar for a third
provider.

Cons: the shared steps aren't parameter-free — each would need 3-6 inputs
(var-file path, inventory group name, connection fields, header comment
text), so a shared task file becomes a small DSL rather than a readable
recipe. The current "Algorithm" comment + linear task list is valuable
documentation for a single-maintainer infra repo.

My take: don't extract yet for just these two call sites — it risks shaping
the abstraction around two examples before a third (podman) reveals what's
actually invariant (YAGNI / Principle IV). I'd extract only once podman lands
and the same fragments are needed a third time — most likely candidates would
be "load/init autogenerated inventory" and "build + write updated inventory",
each needing only path/group-name/header parameters.

__Reviewer:__ I agree, let's wait for the `podman` provider to land.

**Podman as a third provider** — not trivial, but comparable in size to this
docker provider, not a new category of work. It would largely reuse
`docker.yml`'s skeleton, since podman's CLI is close to docker's (`podman
build`, `podman run -d --privileged`, `podman port`). Caveats to verify before
committing:

- **Rootless vs rootful**: systemd-in-container with `--privileged` behaves
  differently under rootless podman (cgroup v2 delegation, subuid/subgid
  mapping). This is the main risk — needs a smoke test on the actual target
  host's podman setup.
- `podman port <name> 22/tcp` output format should be checked against the
  regex used to extract `docker_ssh_port` — likely compatible but unverified.
- Same Dockerfile should build via `podman build` (OCI-compatible), but
  podman's native systemd support (`--systemd=true`) may simplify or change
  the cgroup setup compared to docker's privileged-systemd approach.
- New hostname pool file + new inventory group (`podman`), same pattern as
  the docker provider.

Recommend a short spike against the real podman host before estimating
further.

__Reviewer:__ After reconsidering, I think that a `podman` provider is not that important for me yet. I think the possible unknowns (possible complications) would cause effort which is inappropriate for the benefit for now. I don't need a Linux box to provision VMs today and docker engine without docker desktop is still free to use. Thus, this could be a low priority feature request for the future. Add it to the @docs/feature-requests/feat.create-destroy-vm/roadmap.md document in a new section "Outlook" at the bottom.

`docker build`, etc. in `docker.yml` to the one-token-per-line style shown.

### "Discover published SSH port" task / docker rm vs docker container rm

**Finding:** Explain how the task `Discover published SSH port` works. Start
with a high level, short explanation and give details when I ask for them.

You are using `docker rm` to remove the container on failure. I am used to
using `docker container rm`. Please explain the difference shortly.

**Reply:**

*Discover published SSH port* — the container is started with
`-p 127.0.0.1::22` (no host port given), so Docker picks a free host port and
maps it to the container's port 22. `docker port <name> 22/tcp` asks Docker
which host port that is, returning something like `127.0.0.1:54321`. The task
strips everything up to the last `:` to get `54321`, validates it's numeric,
and uses it as `ansible_port` for the SSH connection to `127.0.0.1`.

__Reviewer:__ Nice. Please add a short comment above the corresponding task so that I remember that when reading the file in the future.

*`docker rm` vs `docker container rm`* — aliases, functionally identical.
`docker rm` is the original top-level command; `docker container rm` is the
newer namespaced form (introduced when Docker grouped commands under
`docker container`/`image`/`volume`/...). Neither is deprecated.
`destroy/docker.yml` already uses the namespaced `docker container inspect`
for the existence check but top-level `docker stop`/`docker rm` for the
actions — a minor style inconsistency, not a functional issue.

## playbooks/vars/docker_credentials.yml

**Finding:** Would it be reasonable to use the same user / password
combination for the initial root user as for the tart provisioner
(admin/admin)?

**Reply:** I'd keep them as-is (`root`/`docker`). The container's primary
account is `root` — the Dockerfile doesn't create an `admin` user, so
"admin/admin" wouldn't map to an existing account without adding a
user-creation step. What matters for consistency with the tart provider is
the *pattern* — a documented, non-rotated, build-baked default credential for
a disposable local environment — and that pattern already matches. Reusing
the literal string "admin" as the root password would be more confusing than
helpful here.
