# Spike: Faster Tart VM Boot for create-vm Playbook

> Timeboxed investigation. Deliverable is a findings + recommendation section
> appended to `specs/009-create-destroy-vm/research.md` — NOT a full
> implementation, and NOT a numbered decision record. Tracker: r5vs (transient
> ref; do not inline into durable artefacts per Constitution Principle X).

## Context

`vagrant up` boot time in the create-vm flow is unacceptably slow. The current
implementation (`playbooks/tasks/create/tart.yml`) does:

1. Create per-VM dir + render `Vagrantfile.j2`.
2. `vagrant up` (vagrant-tart provider, image `ghcr.io/cirruslabs/ubuntu:24.04`).
3. Poll `tart ip <hostname>` — 30 retries x 10s (`tart.yml:99-117`).
4. Assert IP is non-empty, then `wait_for` SSH port 22, timeout 300s
   (`tart.yml:108-124`).
5. Build the autogenerated inventory (`set_fact` at `tart.yml:126-148`) and write
   it.

On any failure after VM-directory creation, a `rescue` block
(`tart.yml:161-178`) runs `vagrant destroy -f`, removes the VM directory, and
fails loud with a rollback message. This rescue/cleanup/fail-loud behaviour is a
hard requirement that any faster option must preserve or explicitly cost to
rebuild (see Acceptance Criteria).

`Vagrantfile.j2` sets `config.vm.boot_timeout = 0` (`Vagrantfile.j2:5`) — vagrant's
own SSH wait is disabled, a prior fix because vagrant-tart rejects
`communicator: none` — and `config.ssh.insert_key = false` (`Vagrantfile.j2:9`),
using password auth via sshpass, and pins the tart image.

Spike candidates from the issue:

- **A.** Call `tart` CLI directly (`tart clone` + `tart run`), bypassing vagrant.
- **B.** Pre-built / pre-cached images with SSH pre-enabled.
- **C.** Specific vagrant-tart config tuning (enumerated knobs — see Experiment C).

Platform constraint: macOS Apple Silicon only — tart is ARM64/macOS-only, so
**no Molecule test is possible** (Constitution Principle II/III exception, already
documented for feature 009).

## Pre-Declared Boot-Time Target (spike INPUT, not output)

- **Target: SSH-ready within 300 seconds (5 minutes)** on a cold boot, matching
  the Vagrant default `boot_timeout` and the existing `wait_for` timeout
  (`tart.yml:123`).
- This 300s target is an **input** to the spike: it defines candidacy, not the
  result the spike discovers.
- **Candidacy + ranking rule** (also in Acceptance Criteria): "Any option that
  achieves SSH-ready within 300s cold-boot is a candidate. Among candidates, the
  safest (highest failure-handling fidelity) wins — not merely the fastest."

## RALPLAN-DR Summary

Mode: **SHORT** (default). Timeboxed spike, low blast radius (no production
playbook change in this work item).

### Principles (governing this spike)

1. **YAGNI (Principle IV)** — measure first; build the minimum experiment needed
   to decide. No speculative abstraction, no rewrite during the spike.
2. **Fail Loud (Principle XII)** — every experiment must surface VM-creation
   failure explicitly; a "fast" path that silently leaves a half-booted VM is a
   failed experiment, not a faster one. Failure-handling fidelity is a ranking
   dimension, not a footnote.
3. **Evidence over assumption** — a recommendation is only valid with timed,
   reproducible numbers (min/median over >=3 runs) for baseline and each candidate.
4. **Self-contained findings (Principle X)** — the findings section must inline the
   substance; no dangling tracker IDs in `specs/**`.
5. **No untracked debt (Principle VIII)** — any follow-up work the spike reveals
   is filed as a tracked issue blocking the recommendation's adoption.

### Decision Drivers (top 3)

1. **Boot wall-clock time** (control-node start to SSH-ready) — the metric the
   spike exists to reduce, judged against the 300s cold-boot target.
2. **Failure-handling fidelity** — can the option replicate the create-time
   rescue/cleanup and fail-loud guarantees of the `rescue` block at
   `tart.yml:161-178`? An option that cannot is downgraded unless it costs the
   re-implementation of those guarantees.
3. **SSH-readiness reliability / fit with existing pure-Ansible flow** — the chosen
   path must deterministically produce a reachable, authenticable VM, and the cost
   of re-implementing the vagrant machinery (inventory write, idempotent `creates:`)
   must be accounted for.

### Viable Options (with bounded pros/cons)

**Option A — Direct `tart` CLI (`tart clone` + `tart run`)**

- Pros: removes the vagrant + ruby layer entirely; `tart clone` from a local
  cached image is typically the fastest path; full control over IP/SSH polling.
- Cons: must re-implement everything vagrant gave us — VM dir lifecycle, the
  rescue/teardown-on-failure + fail-loud block (`tart.yml:161-178`), SSH/IP
  discovery, idempotent re-run guard; higher maintenance surface; SSH enablement
  on a stock cloud image must be solved manually (cloud-init / image choice).

**Option B — Pre-built / pre-cached image with SSH pre-enabled**

- Pros: removes per-create image pull/setup cost; `tart clone` of a warm local
  image is fast and deterministic; orthogonal — composes with A or C.
- Cons: requires building/maintaining a custom tart image (a new artefact to
  store, version, and refresh — touches version-update mechanism if pinned);
  the cirruslabs image may already cache after first pull, so the marginal win
  needs measuring before committing.

**Option C — vagrant-tart config tuning (keep vagrant)**

- Pros: smallest change; keeps the proven inventory-write, rescue, and idempotency
  machinery — highest failure-handling fidelity by default; lowest maintenance.
- Cons: limited ceiling — much of the slowness may be image pull and guest boot,
  not vagrant overhead; some knobs (`boot_timeout`, `insert_key`) already tuned.

> At least two options are viable (A, B, C all survive). If baseline data shows
> vagrant overhead is negligible (most time in image pull + guest boot), Options
> A and C collapse toward B; the findings section must then record the explicit
> invalidation rationale for the dropped options.

## SSH-Readiness — Precise Definition

For **every** experiment, SSH-readiness means BOTH of the following are confirmed:

1. `tart ip <hostname>` returns a non-empty IP, AND
2. Port 22 is accepting connections on that IP.

Both must be confirmed for each experiment, matching the bar set by
`tart.yml:99-124` (the `tart ip` poll, the non-empty-IP assert, and the
`wait_for` on port 22). A run that satisfies only one is not SSH-ready.

## Timing & Hardware Method

- **Hardware binding**: the spike MUST run on macOS Apple Silicon (tart is
  ARM64/macOS-only). Timings are host-specific; record the test machine's specs
  (chip, core count, RAM, macOS version, tart version, vagrant + vagrant-tart
  versions, network state).
- **Cold vs warm**: measure each experiment cold (image not in tart's local cache)
  and warm (image already cached).
- **Sample size**: >=3 runs per cold/warm cell. Report **min and median** (not a
  single number) to capture variance.
- **Clock boundary**: start = control-node command issue; stop = SSH-readiness
  confirmed per the definition above.
