# Implementation Plan: Create and Destroy VM Playbooks

**Branch**: `009-create-destroy-vm` | **Date**: 2026-06-06 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/009-create-destroy-vm/spec.md`

## Summary

Replace ad-hoc Vagrant invocations with two focused playbooks — `create-vm.yml`
and `destroy-vm.yml` — that manage the full lifecycle of local Tart-backed VMs
on macOS ARM64. Hostnames are drawn sequentially from a shared Star Trek TNG
planet name pool; inventory is updated atomically on every create and destroy.

## Technical Context

**Language/Version**: Ansible (YAML), ansible-core (project default)
**Primary Dependencies**: `tart` CLI (macOS ARM64); `sshpass` for password auth
**Storage**: Autogenerated YAML inventory `inventories/tart_autogenerated.yml`;
tart VMs at `~/.tart/vms/<hostname>/`
**Testing**: Manual validation on macOS ARM64 (cannot containerise
— Tart requires Apple Silicon hardware); follows CONTRIBUTING.md procedure
**Target Platform**: macOS ARM64 (control node only; no managed remote hosts)
**Project Type**: Ansible playbooks (localhost CLI tools)
**Performance Goals**: N/A — single-operator, interactive use
**Constraints**: No locking; single-operator project (YAGNI, SC-004)
**Scale/Scope**: Up to 10 VMs (hostname pool size); one VM per invocation

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Idempotency | PASS (constrained) | Shell tasks in `tasks/create/tart.yml` and `tasks/destroy/tart.yml` MUST use `creates:` or `changed_when:` guards. Inventory update via load→modify→write is inherently idempotent. Pool check via inventory membership is idempotent. |
| II. Role-First | JUSTIFIED EXCEPTION | Runs entirely on `localhost` against the macOS Tart provider — not against managed hosts. Role model assumes tasks execute on managed nodes; wrapping these shell-based Vagrant invocations in a role misrepresents the execution model and adds no value. See Complexity Tracking. Molecule scenario: not possible (Tart requires Apple Silicon hardware); validate via Vagrant procedure in CONTRIBUTING.md. No versioned tool pins → version-update registration N/A. |
| III. Test Locally Before Cloud | PASS | Feature is local-VM management only; cloud targets not applicable. Validate by running both playbooks on macOS ARM64 host before any cloud work. |
| IV. Simplicity (YAGNI) | PASS | Single provider (tart) only. No locking. No multi-provider abstraction. Minimal Vagrantfile template. |
| XII. Fail Loud | PASS (constrained) | Pool exhaustion MUST assert before any VM/infra action (FR-007). Unknown hostname MUST fail with clear error before any action (FR-010). Platform unavailable MUST surface as explicit failure. No silent skips or empty fallbacks. |
| X. No External-System References | PASS | No beads IDs or tracker references in any durable artefact. |
| XI. Avoid Duplication | PASS | Pool-check logic lives in one task file. Inventory update logic is in dedicated task files, not duplicated between create and destroy. |
| XIII. No Empty Artefacts | PASS | No placeholder files. Every file created carries real content. |

### Source Code (repository root)

```text
playbooks/
├── create-vm.yml                  # Orchestrates pool-check → create → inventory update
├── destroy-vm.yml                 # Orchestrates inventory-check → destroy → inventory update
├── vars/
│   └── hostname_pool_tart.yml          # Ordered TNG planet name list (10 entries)
├── templates/
│   └── Vagrantfile.j2             # Jinja2 template rendered per-VM at create time
└── tasks/
    ├── create/
    │   └── tart.yml               # Vagrant+Tart VM creation tasks
    └── destroy/
        └── tart.yml               # Vagrant+Tart VM destruction tasks

inventories/
└── vagrant_tart.yml               # Static YAML inventory (updated by playbooks)

~/.local/share/ansible-vms/
└── <hostname>/
    ├── Vagrantfile                 # Per-VM; templated from playbooks at create time
    └── .vagrant/
        └── machines/<hostname>/tart/private_key
```

**Structure Decision**: Playbook-centric layout (playbooks/tasks/ includes) following
the existing project pattern (see `playbooks/tasks/setup-vscode.yml` et al.). Per-VM
Vagrantfiles live outside the repo under XDG-compatible `~/.local/share/ansible-vms/`
to keep the git tree clean and allow `.vagrant/` state to persist between runs.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| Principle II: implementation logic in `playbooks/tasks/create/tart.yml` and `playbooks/tasks/destroy/tart.yml` instead of roles | Runs on localhost against macOS Vagrant/Tart provider; no managed host involved. Role model (become, remote tasks, molecule) does not apply to control-node shell orchestration. | A role wrapping `vagrant up` on localhost would have no `molecule/default/` scenario (hardware constraint), no remote execution, and no reuse potential beyond this single use case — all the costs of a role with none of the benefits. The project already uses `playbooks/tasks/` for control-node operations. |
