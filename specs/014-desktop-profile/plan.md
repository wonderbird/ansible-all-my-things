# Implementation Plan: Desktop Profile for Create and Destroy VM Playbooks

**Branch**: `014-desktop-profile` | **Date**: 2026-06-20 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/014-desktop-profile/spec.md`

## Summary

Introduce a `profile` extra-var on `create-vm.yml` (values `basic` default,
`desktop`) where none exists today. Every provider's create/destroy task file
(`tasks/{create,destroy}/{tart,docker,hcloud,aws}.yml`) gains a `profile`
inventory-group key (`basic` or `desktop`) alongside its existing `all`/
`linux`/`<provider>` groups. `configure-profile-roles.yml` is scoped to
`hosts: basic` (replacing today's `hosts: linux`) and gains a sibling
`desktop`-targeted play that imports the legacy `configure-linux-roles.yml`
role list verbatim. `configure-profile.yml` gains a `desktop` branch
importing `setup-desktop.yml`, `setup-keyring.yml`, `setup-desktop-apps.yml`
verbatim. `create-vm.yml` rejects `provider=docker` with `profile=desktop`
before any infrastructure action. `tasks/create/aws.yml`'s security-group
rule for TCP 3389 is conditioned on `profile==desktop`. The legacy
verbatim-import of `setup-desktop.yml`/`setup-keyring.yml`/
`setup-desktop-apps.yml`/`configure-linux-roles.yml`'s role list — instead of
extracting them into roles first — is an explicitly accepted Principle II
exception, logged in Complexity Tracking below per the Governance clause.

## Technical Context

**Language/Version**: Ansible YAML (playbooks/tasks), consistent with the
existing `tart`/`docker`/`hcloud`/`aws` provider implementations and the
`basic`-profile orchestration playbooks in this repo.

**Primary Dependencies**: None new. Reuses existing roles already applied by
`configure-profile-roles.yml` (`podman`, `tmux`, `ruby`, `python`,
`dolt_sql_server`, `claude_code`) and existing legacy playbooks
(`configure-linux-roles.yml`, `setup-desktop.yml`, `setup-keyring.yml`,
`setup-desktop-apps.yml`) unchanged.

**Storage**: Flat YAML files only — the four gitignored, generated
`inventories/{tart,docker,hcloud,aws}_autogenerated.yml` files each gain one
new group key (`basic` or `desktop`) per host entry, alongside their
existing `all`/`linux`/`<provider>` keys. No new hostname pool, no new
committed inventory file, no new group_vars directory (the `desktop` group's
behavior comes entirely from which playbooks/roles target it, not from new
default variables).

**Testing**: No Molecule scenario — `configure-profile-roles.yml` and
`configure-profile.yml` are orchestration playbooks targeting dynamically
created remote hosts, not a fresh container (same precedent as
`specs/010-configure-basic-profile`, validated against a real tart VM per
Constitution Principle III). The `tart`/`hcloud` providers can validate the
full `profile=desktop` path (group plumbing, role list, legacy playbook
import) locally/cheaply; the AWS RDP rule (FR-007/FR-008) additionally needs
one real `provider=aws profile=desktop` and one `provider=aws profile=basic`
run to verify the security-group exposure boundary, since AWS security
groups have no local equivalent (same reasoning the AWS provider plan used
for Principle III).

**Target Platform**: Controller = `localhost`. Managed nodes = any
`tart`/`hcloud`/`aws` VM (Linux); `docker` is rejected for `profile=desktop`
(FR-006) and otherwise unaffected.

**Project Type**: Single Ansible project — task-file and playbook
modifications to the existing `playbooks/` tree. No new roles (the
verbatim-import exception, logged below, is specifically what avoids
introducing role work for `setup-desktop.yml`/`setup-keyring.yml`/
`setup-desktop-apps.yml`/the legacy desktop role list in this feature).

**Performance Goals**: N/A (one-shot provisioning/configuration commands;
existing timeout conventions apply unchanged).

**Constraints**: Constitution Principles I (idempotency — group-key
additions use the same `combine()`/`dict2items|selectattr` merge pattern
already used for `all`/`linux`/`<provider>`, which is already idempotent),
II (Role-First Organisation — exception logged in Complexity Tracking), IV
(YAGNI — `profile` carries exactly two live values, no speculative third),
XI (DRY — reuses `desktop_users`/`inventories/group_vars/all/vars.yml`
unchanged, no duplicated user-list logic), XII (Fail Loud —
`provider=docker profile=desktop` rejected before any infrastructure action,
mirroring the existing `assert-provider.yml` pattern). FR-001 through FR-012
in [spec.md](./spec.md) are the binding requirements.

**Scale/Scope**: 8 existing task files modified (`tasks/{create,destroy}/
{tart,docker,hcloud,aws}.yml` — add one inventory-group key each),
`create-vm.yml`/`destroy-vm.yml` (add `profile` var + validation),
`tasks/assert-provider.yml` or a new sibling assertion task (validate
`profile` value and reject the `docker`+`desktop` combination),
`configure-profile-roles.yml` (re-scope existing play to `hosts: basic`, add
a `desktop`-targeted play), `configure-profile.yml` (add `desktop` branch),
`tasks/create/aws.yml` (condition the 3389 rule). Zero new roles, zero new
top-level playbooks beyond what already exists.

**Structure Decision**: Single Ansible project, no new top-level
directories, roles, or playbooks. The feature is a coordinated, mechanical
extension of the existing per-provider create/destroy task-file pattern (one
more inventory-group key, mirroring how each provider already tracks its own
group) plus two new branches in the existing profile-configuration
playbooks. The legacy desktop playbooks/role-list are imported by reference
(`import_playbook` / role-list inclusion), not copied or rewritten.

## Complexity Tracking

> Constitution Check found one exception. Logged per the Governance clause
> requiring exceptions be recorded before implementation begins.

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|---------------------------------------|
| Principle II (Role-First Organisation): `configure-profile.yml`'s `desktop` branch verbatim-imports `setup-desktop.yml`, `setup-keyring.yml`, and `setup-desktop-apps.yml` (non-role playbooks with inline `tasks:`), and `configure-profile-roles.yml`'s `desktop` play verbatim-imports `configure-linux-roles.yml`'s role list, instead of every capability being a standalone role with a Molecule scenario. | None of this capability exists as a role today — extracting `setup-desktop.yml`/`setup-keyring.yml`/`setup-desktop-apps.yml` into proper roles (with Molecule coverage, since most of their tasks are container-testable) is itself a substantial body of work, separable from "can an engineer get a desktop VM through `create-vm.yml` at all." Blocking the desktop profile's delivery on that extraction first would leave desktop VMs reachable only through the legacy `provision.yml` → `configure-linux.yml` path this whole epic exists to replace. | Doing the full role extraction first was considered and rejected: it violates the roadmap's "vertical slices" guiding principle (a horizontal "extract all roles" pass delivers no demoable capability on its own) and is large enough to risk stalling the desktop profile indefinitely. The verbatim-import path delivers the user-visible capability (User Story 1) now, while the extraction is tracked as a separate, explicitly deferred follow-up (see spec.md Assumptions) rather than silently dropped. |
