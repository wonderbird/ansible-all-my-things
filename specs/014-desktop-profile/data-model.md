# Phase 1 Data Model: Desktop Profile for Create and Destroy VM Playbooks

This feature has no database or application data model — it extends the
existing flat-YAML inventory entities used by `create-vm.yml`/
`destroy-vm.yml`. The entities below are inventory/configuration concepts,
not application records.

## Profile

The `profile` extra-var on `create-vm.yml`.

| Field | Type | Values | Notes |
|---|---|---|---|
| `profile` | string | `basic` (default), `desktop` | Selects both the inventory group a new VM joins and the role/playbook set `configure-profile-roles.yml`/`configure-profile.yml` apply to it. |

**Validation rules**:

- MUST be one of `basic`, `desktop` (FR-001).
- `profile=desktop` combined with `provider=docker` MUST be rejected before
  any infrastructure action (FR-006).

**Relationships**: Determines which of two inventory groups (`basic` or
`desktop`) a VM is registered into (see Profile Inventory Group below);
orthogonal to `provider` (which selects the per-provider group: `tart`,
`docker`, `hcloud`, `aws`) except for the one disallowed combination above.

## Profile Inventory Group

A new dict key (`basic` or `desktop`) written into each provider's
`inventories/<provider>_autogenerated.yml`, alongside the existing `all`/
`linux`/`<provider>` keys.

| Field | Type | Notes |
|---|---|---|
| group name | string | `basic` or `desktop`, selected by the `profile` extra-var at create time. |
| `hosts` | dict (hostname → host vars) | Same hostname keys as the host's other group memberships (`all`, `linux`, `<provider>`); no additional per-host vars introduced by this group. |

**State transitions**:

- **Create**: host added under exactly one of `basic`/`desktop` (matching
  the `profile` value used at creation), in addition to its existing
  `all`/`linux`/`<provider>` memberships.
- **Destroy**: host removed from whichever of `basic`/`desktop` it belonged
  to, symmetrically with its removal from `all`/`linux`/`<provider>`.
- A host's profile group membership is fixed at create time and not
  changed by any other operation (no "re-profile an existing VM" capability
  in this feature).

## AWS Desktop Security Rule

A conditional rule on the existing, shared `ansible-sg` AWS security group.

| Field | Type | Notes |
|---|---|---|
| `proto` | string | `tcp` |
| `ports` | list[int] | `[3389]` |
| `cidr_ip` | string | `0.0.0.0/0` (matching the existing 22 rule's exposure) |
| condition | — | Applied only when `profile == 'desktop'` (FR-007/FR-008); never applied for `profile == 'basic'`. |

**Relationships**: Lives on the same `ansible-sg` group already used by
every AWS VM regardless of profile (no new security group entity).
