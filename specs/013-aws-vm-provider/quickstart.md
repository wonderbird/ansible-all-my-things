# Quickstart: aws Provider for create-vm.yml / destroy-vm.yml

## Prerequisites

- AWS credentials configured via the standard AWS SDK credential chain
  (environment variables, shared credentials file, or instance profile — the
  same convention used by `provisioners/aws-linux.yml`).
- Your SSH private key matching `my_ssh_key_name` (vault-configured in
  `inventories/group_vars/all/vault.yml`) loaded in your SSH agent — this key
  must already be registered with AWS as a key pair in the target region.
- `requirements.yml` collections installed (`amazon.aws` already declared).
- Vault password available via `ANSIBLE_VAULT_PASSWORD_FILE`.

## Create a disposable EC2 Linux instance

```bash
ansible-playbook playbooks/create-vm.yml -e provider=aws
```

- Picks the next unused hostname from `playbooks/vars/hostname_pool_aws.yml`
  (`arrakis`, `caladan`, `giedi-prime`, `kaitain`, `ix`, `chusuk`, `tleilax`,
  `salusa-secundus`, `wallach-ix`, `ginaz`).
- Ensures the shared `ansible-sg` security group exists (SSH/22 inbound).
- Creates the instance with the default AMI/instance-type/region (overridable
  via `-e aws_ami_id=...`, `-e aws_instance_type=...`, `-e aws_region=...`).
- On success, prints `VM created with hostname: <name>` and adds an entry to
  `inventories/aws_autogenerated.yml`.

Verify it's reachable:

```bash
ansible -i inventories/aws_autogenerated.yml <name> -m ping
```

## Destroy it

```bash
ansible-playbook playbooks/destroy-vm.yml -e provider=aws -e hostname=<name>
```

- Terminates the EC2 instance.
- Removes its entry from `inventories/aws_autogenerated.yml`.
- Removes its SSH host key from `inventories/aws_known_hosts`.
- Leaves the shared `ansible-sg` security group in place.

## Existing providers — unaffected

```bash
ansible-playbook playbooks/create-vm.yml                     # provider=tart (default), unchanged
ansible-playbook playbooks/create-vm.yml -e provider=docker  # unchanged
ansible-playbook playbooks/create-vm.yml -e provider=hcloud  # unchanged
```

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| AWS SDK authentication/credentials error on the security-group step | AWS credentials unset/invalid | Configure AWS credentials (env vars, `~/.aws/credentials`, or instance profile) |
| `AWS hostname pool exhausted...playbooks/vars/hostname_pool_aws.yml` (before any API call) | All 10 pool names already in `inventories/aws_autogenerated.yml` | Destroy unused servers, or append names to the pool file |
| `Hostname '<x>' is not present in inventories/aws_autogenerated.yml` | Destroying a hostname that was never created (or already destroyed) | Use a valid hostname from `inventories/aws_autogenerated.yml` |
| `REMOTE HOST IDENTIFICATION HAS CHANGED` on create/connect | A *different* instance now answers at a previously-known IP, and the stale key was never removed (e.g. a create that bypassed `destroy-vm.yml`) | This is the intended fail-loud MITM protection (ADR-003) — manually remove the stale line from `inventories/aws_known_hosts` only if you can confirm the IP reassignment is legitimate |
