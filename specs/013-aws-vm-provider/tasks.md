## Phase 3: User Story 1 - Create an AWS EC2 Linux server (Priority: P1) 🎯 MVP

**Goal**: `ansible-playbook playbooks/create-vm.yml -e provider=aws` ensures
the shared `ansible-sg` security group exists, creates one new EC2 instance,
registers it in `inventories/aws_autogenerated.yml` with a verified-host-key
SSH connection, and prints the assigned hostname.

**Independent Test** (spec.md US1): With AWS credentials configured, run
`ansible-playbook playbooks/create-vm.yml -e provider=aws`. The command
completes, prints `VM created with hostname: <name>`, and
`ansible -i inventories/aws_autogenerated.yml <name> -m ping` succeeds.

  `giedi-prime`, `kaitain`, `ix`, `chusuk`, `tleilax`, `salusa-secundus`,
  `wallach-ix`, `ginaz`. Follow the header-comment style of
  `playbooks/vars/hostname_pool_hcloud.yml` (explain the pool's

- [X] T004 [P] [US1] In `playbooks/create-vm.yml`, update
  `assert_provider_usage_hint` from
  `Run: ansible-playbook playbooks/create-vm.yml -e provider=<tart|docker|hcloud>`

  `Run: ansible-playbook playbooks/create-vm.yml -e provider=<tart|docker|hcloud|aws>`

- [X] T005 [US1] Create `playbooks/tasks/create/aws.yml` implementing the full
  create algorithm, mirroring the six-step shape of
  `playbooks/tasks/create/hcloud.yml` (research.md "Reuse the
  hcloud/docker/tart task-file pattern, plus one extra step") with a leading
  header comment describing the algorithm:
  1. **Preconditions before any API call** (contracts/create-vm-aws.md):
     - `include_vars` the existing default-source files per FR-013 (mirroring
       the `vars_files`/`include_vars` pattern in `provisioners/aws-linux.yml`):
       `inventories/group_vars/aws_ec2_linux/vars.yml` (for
       `aws_default_ami_id`, `aws_default_instance_type`),
       `inventories/group_vars/aws_ec2/vars.yml` (for `aws_default_region`),
       and `inventories/group_vars/all/vars.yml` (for `my_ssh_key_name`). Do
       not duplicate these values (XI/DRY).
     - Load `playbooks/vars/hostname_pool_aws.yml` (T003).
     - Check for / load `inventories/aws_autogenerated.yml`
       (`ansible.builtin.stat` + conditional `include_vars`, else initialize
       an empty `current_inventory` with `all`/`linux`/`aws` groups — same
       pattern as `tasks/create/hcloud.yml`).
     - Compute `used_hostnames` from `current_inventory.all.hosts.keys()`,
       then `available_hostnames` via
       `hostname_pool | reject('in', used_hostnames) | list`
       (research.md "Hostname selection algorithm").
     - `fail` if `available_hostnames | length == 0`, naming
       `playbooks/vars/hostname_pool_aws.yml` (FR-007).
     - Set `hostname` to `available_hostnames | first` (FR-003).
  2. **Resolve creation parameters** (FR-013, contracts table): set
     `image_id` from extra-var `aws_ami_id` defaulting to `aws_default_ami_id`;
     `instance_type` from extra-var `aws_instance_type` defaulting to
     `aws_default_instance_type`; `region` from extra-var `aws_region`
     defaulting to `aws_default_region`.
  3. **block**: ensure the security group, create the instance, and register
     it —
     - `amazon.aws.ec2_security_group`: `name: ansible-sg`, `state: present`,
       `region: "{{ region }}"`, an inbound rule allowing TCP/22 from
       `0.0.0.0/0` (FR-013, research.md "Shared `ansible-sg` security group").
     - `amazon.aws.ec2_instance`: `state: present`, `name: "{{ hostname }}"`,
       `image_id`, `instance_type`, `region`, `key_name: "{{
       my_ssh_key_name }}"`, `security_groups: [ansible-sg]`, `wait: true`.
       Register the result (e.g. `aws_instance_result`) to read
       `aws_instance_result.instances[0].public_ip_address` and
       `aws_instance_result.instances[0].instance_id` (FR-002, FR-013).
     - Idempotent stale-key guard on the new instance's public IP against
       `inventories/aws_known_hosts` per research.md "Idempotent `ssh-keygen
       -R` guard": `ssh-keygen -f {{ playbook_dir }}/../inventories/aws_known_hosts
       -F <ip>` (`failed_when: false`, `changed_when: false`) to detect a
       stale entry, then `ssh-keygen -f {{ playbook_dir
       }}/../inventories/aws_known_hosts -R <ip>` only `when:` a key was
       found, `failed_when: false`, `changed_when: true` on that conditional
       step (FR-005).
     - `wait_for_connection` against the new instance's public IPv4, port 22,
       as `ubuntu`, with `ansible_ssh_common_args: "-o
       StrictHostKeyChecking=accept-new -o UserKnownHostsFile={{ playbook_dir
       }}/../inventories/aws_known_hosts"` (FR-005, contracts/create-vm-aws.md
       effect 4). Use the same `UserKnownHostsFile` path form for this
       connection and for the persisted entry in
       `inventories/aws_autogenerated.yml` — see data-model.md "AWS Inventory
       Entry".
     - Build `updated_inventory` adding the new host under `all`/`linux`/`aws`
       groups with `ansible_host` (the public IPv4 from above),
       `ansible_port: 22`, `ansible_user: ubuntu`, `aws_instance_id` (the
       `instance_id` from above), and the same `ansible_ssh_common_args` as
       above (FR-004, FR-005, data-model.md "AWS Inventory Entry" — adds
       `aws_instance_id` beyond the hcloud shape, no `ansible_ssh_pass`
       field).
     - Write `inventories/aws_autogenerated.yml` via `ansible.builtin.copy`
       with the `"# AUTO-GENERATED — do not edit manually. Overwritten by
       create-vm.yml and destroy-vm.yml.\n"` header and `to_nice_yaml(indent=2)`,
       `mode: "0644"` (same as `tasks/create/hcloud.yml`).
     - Print `VM created with hostname: {{ hostname }}` via
       `ansible.builtin.debug` (FR-006).
  4. **rescue**: `amazon.aws.ec2_instance: state: absent, instance_ids: ["{{
     aws_instance_result.instances[0].instance_id }}"], region: "{{ region
     }}"` with `failed_when: false` (only when an instance was actually
     created — guard on the registered result being defined), then `fail`
     with a message naming `{{ hostname }}` and instructing re-run of
     `create-vm.yml -e provider=aws` (FR-014, contracts/create-vm-aws.md
     "Effects on failure"). Both `inventories/aws_autogenerated.yml` and
     `inventories/aws_known_hosts` remain unchanged on this path since the
     inventory write happens after `wait_for_connection` succeeds. The
     `ansible-sg` security group, if ensured, is left in place (shared
     resource, not part of rollback).

## Phase 4: User Story 2 - Destroy an aws-provider VM by hostname (Priority: P2)

**Goal**: `ansible-playbook playbooks/destroy-vm.yml -e provider=aws -e
hostname=<name>` terminates the EC2 instance (by its stored
`aws_instance_id`), removes its SSH host key from
`inventories/aws_known_hosts`, and removes the entry from
`inventories/aws_autogenerated.yml`, with no dangling state either way. The
shared `ansible-sg` security group is left untouched.

**Independent Test** (spec.md US2): Given a running aws-provider server
`arrakis` (created in Phase 3), run `ansible-playbook
playbooks/destroy-vm.yml -e provider=aws -e hostname=arrakis`. Verify the
instance no longer exists in AWS and `arrakis` is absent from
`inventories/aws_autogenerated.yml` and from `inventories/aws_known_hosts`.

### Implementation for User Story 2

  `assert_provider_usage_hint` from:

  ```text
  Run: ansible-playbook playbooks/destroy-vm.yml -e provider=<tart|docker|hcloud> -e hostname=<name>
  ```

  to:

  ```text
  Run: ansible-playbook playbooks/destroy-vm.yml -e provider=<tart|docker|hcloud|aws> -e hostname=<name>
  ```

  (plan.md Project Structure; FR-001).

- [X] T008 [US2] Create `playbooks/tasks/destroy/aws.yml` implementing the
  full destroy algorithm, mirroring the shape of
  `playbooks/tasks/destroy/hcloud.yml` with a leading header comment
  describing the algorithm:
  1. `hostname` is already asserted defined/non-empty by `destroy-vm.yml`'s
     shared pre_task — no re-check needed (contracts/destroy-vm-aws.md
     precondition 2).
  2. Check for / load `inventories/aws_autogenerated.yml` (`stat` +
     conditional `include_vars`, else empty `current_inventory` with
     `all`/`linux`/`aws` groups — same pattern as
     `tasks/destroy/hcloud.yml`).
  3. `fail` if `hostname not in (current_inventory.all.hosts | default({}))`,
     naming the unknown hostname, before any AWS action (FR-008, FR-010,
     contracts/destroy-vm-aws.md precondition 3).
  4. Capture `ansible_host` (the instance's public IPv4) and
     `aws_instance_id` for `hostname` from
     `current_inventory.all.hosts[hostname]` before the entry is removed —
     needed for the known_hosts cleanup (step 6) and the termination call
     (step 5) (data-model.md "AWS Known Hosts": "keyed by `ansible_host`...
     from the corresponding AWS Inventory Entry"; research.md "Destroy
     terminates by stored `instance_id`").
  5. `include_vars` `inventories/group_vars/aws_ec2/vars.yml` for
     `aws_default_region` (used as the `region` for the AWS API calls below,
     overridable via extra-var `aws_region` as in T005). Instance termination
     (FR-009, contracts/destroy-vm-aws.md):
     - Check whether the instance with id `aws_instance_id` still exists and
       is not already terminated via `amazon.aws.ec2_instance_info:
       instance_ids: ["{{ aws_instance_id }}"]`, `region: "{{ region }}"`.
     - If it exists and is not `terminated`/`terminating`, terminate it via
       `amazon.aws.ec2_instance: state: absent, instance_ids: ["{{
       aws_instance_id }}"], region: "{{ region }}"`.
     - If it does not exist or is already terminated, print a warning
       identifying `hostname` as a stale entry (mirrors
       `tasks/destroy/hcloud.yml`'s "WARNING: ... Proceeding with inventory
       cleanup only.") and skip the termination call — both paths continue to
       steps 6-7 (FR-009 stale-entry behavior, contracts "Effects on success —
       instance already absent/terminated").
  6. Idempotent known_hosts cleanup against the captured `ansible_host` IP
     using the same `-F` / conditional `-R` guard as T005 step 3, on
     `inventories/aws_known_hosts` (FR-009, research.md "Idempotent
     `ssh-keygen -R` guard"). Runs on both the normal and stale-entry paths.
  7. Build `cleaned_inventory` removing `hostname` from `all`/`linux`/`aws`
     hosts (same `dict2items | selectattr('key', '!=', hostname) |
     items2dict` pattern as `tasks/destroy/hcloud.yml`), write it back to
     `inventories/aws_autogenerated.yml` with the same AUTO-GENERATED header
     and `to_nice_yaml(indent=2)`, `mode: "0644"`.
  8. `assert` `hostname not in (cleaned_inventory.all.hosts | default({}))`,
     `fail_msg: "Failed to remove '{{ hostname }}' from
     inventories/aws_autogenerated.yml"` (mirrors
     `tasks/destroy/hcloud.yml`).
  9. The shared `ansible-sg` security group is not referenced or modified by
     this file (FR-013, contracts "Postconditions" — left unaffected).

  BLOCKS both User Story 1 and User Story 2 (the shared `provider` allow-list

## Implementation Strategy

### MVP First (User Story 1 Only)

1. Phase 1 (T001) + Phase 2 (T002).
2. Phase 3 (T003-T006) — `create-vm.yml -e provider=aws` works end-to-end
   against the real AWS EC2 API.
3. **STOP and VALIDATE**: T006's Independent Test passes.

### Incremental Delivery

1. Setup + Foundational (T001-T002).
2. User Story 1 (T003-T006) → validate → MVP.
3. User Story 2 (T007-T009) → validate → full create/destroy lifecycle.
4. User Story 3 (T010) → regression check, no new code.
5. Polish (T011).

### Cleanup note

T006 and T009 each create a real, billable EC2 instance (and, on first run,
the shared `ansible-sg` security group). T009 destroys the instance T006
created; if T009 is skipped or fails, manually run `destroy-vm.yml -e
provider=aws -e hostname=<name>` to avoid leaving a billable resource running.
The `ansible-sg` security group itself is shared/long-lived and is not cleaned
up by either task — this matches the existing `provisioners/aws-linux.yml`
convention.
