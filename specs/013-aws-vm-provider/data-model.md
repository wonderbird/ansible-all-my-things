# Phase 1 Data Model: AWS EC2 Provider

All entities are flat YAML structures read/written by Ansible task files —
there is no database or ORM. This document describes the shape of each file
and the fields within it.

## AWS Hostname Pool

**File**: `playbooks/vars/hostname_pool_aws.yml`
**Variable**: `hostname_pool` (list of strings)

```yaml
hostname_pool:
  - arrakis
  - caladan
  - giedi-prime
  - kaitain
  - ix
  - chusuk
  - tleilax
  - salusa-secundus
  - wallach-ix
  - ginaz
```

- **Fields**: ordered list of hostname strings (Dune place names). All 10 ship
  "available" — no entry is pre-reserved (FR-011, spec Assumptions).
- **Relationships**: an entry is "in use" iff it appears as a key under
  `all.hosts` in `inventories/aws_autogenerated.yml`. The set is disjoint from
  `hostname_pool_tart.yml`, `hostname_pool_docker.yml`, and
  `hostname_pool_hcloud.yml` (different naming themes; FR-003).
- **State transitions**: available -> in-use (on create, FR-003) -> available
  again (on destroy, FR-009, by removal from the autogenerated inventory).
  Pool entries themselves never change; only inventory membership changes.
- **Validation**: none beyond non-empty strings; the pool is "extensible" —
  operators may append entries (FR-011). Exhaustion (no available entry) is a
  hard failure before any API call (FR-007).

## AWS Inventory Entry

**File**: `inventories/aws_autogenerated.yml` (generated, gitignored)
**Format**: static YAML inventory, groups `all` / `linux` / `aws` (FR-004),
written via `to_nice_yaml`, identical top-level shape to
`tart_autogenerated.yml` / `docker_autogenerated.yml` /
`hcloud_autogenerated.yml`, plus one extra field (`aws_instance_id`).

```yaml
# AUTO-GENERATED — do not edit manually. Overwritten by create-vm.yml and destroy-vm.yml.
all:
  hosts:
    arrakis:
      ansible_host: <public IPv4 from amazon.aws.ec2_instance>
      ansible_port: 22
      ansible_user: ubuntu
      ansible_ssh_common_args: >-
        -o StrictHostKeyChecking=accept-new
        -o UserKnownHostsFile=<repo>/inventories/aws_known_hosts
      aws_instance_id: <instance-id from amazon.aws.ec2_instance>
linux:
  hosts:
    arrakis: {}
aws:
  hosts:
    arrakis: {}
```

- **Fields per host** (FR-005):
  - `ansible_host` — the instance's public IPv4 address, returned by
    `amazon.aws.ec2_instance` (`instances[0].public_ip_address`).
  - `ansible_port` — always `22`.
  - `ansible_user` — always `ubuntu` (matches `admin_user_on_fresh_system` for
    `aws_ec2_linux`).
  - `ansible_ssh_common_args` — fixed string applying ADR-003's `accept-new` +
    project-scoped `UserKnownHostsFile=inventories/aws_known_hosts`.
  - `aws_instance_id` — the EC2 `instance-id` (`instances[0].instance_id`),
    persisted so `destroy-vm.yml` can terminate by id without a Name-tag
    lookup (FR-005, FR-009).
  - No `ansible_ssh_pass` field — auth is key-based via the registered AWS key
    pair (FR-005), unlike `tart`/`docker`.
- **Relationships**: one entry per live EC2 instance; hostname is the dict key
  in all three groups (`all`, `linux`, `aws`), matching the
  `tart`/`docker`/`hcloud` shape exactly (just a different leaf-group name and
  the added `aws_instance_id` field).
- **State transitions**: absent -> present (create, FR-004, written before
  `create-vm.yml` completes) -> absent (destroy, FR-009, removed on completion
  even for stale entries).
- **Validation**: `destroy-vm.yml -e provider=aws` requires the given
  `hostname` to be a key under `all.hosts` before taking any AWS action
  (FR-008, FR-010).
- **Distinctness**: this file is separate from, and never read/written/replaced
  by, the pre-existing dynamic-inventory plugin file `inventories/aws_ec2.yml`
  (FR-004, FR-015).

## AWS Known Hosts

**File**: `inventories/aws_known_hosts` (generated, gitignored, plus `.old`
backup from `ssh-keygen -R`)
**Format**: standard OpenSSH `known_hosts` file, one line per
`(ip, host-key)` pair.

- **Relationships**: keyed by `ansible_host` (the instance's public IPv4) from
  the corresponding `AWS Inventory Entry`. Created implicitly by `accept-new`
  on first connection (FR-005); entries removed via
  `ssh-keygen -f inventories/aws_known_hosts -R <ip>` on create (stale-IP
  guard) and on destroy (FR-005, FR-009).
- **State transitions**: absent -> contains key for IP (first successful
  connection, TOFU) -> absent for that IP (destroy, or pre-create guard for a
  reused IP). A *changed* key for a known IP is a hard SSH failure
  (`REMOTE HOST IDENTIFICATION HAS CHANGED`), by design (ADR-003, FR-005 edge
  case).

## EC2 Instance

**Resource**: managed via `amazon.aws.ec2_instance` (collection `amazon.aws`),
not stored as a repo file — its state lives in AWS.

- **Fields used** (FR-013): `name` (= hostname from the pool, written as the
  `Name` tag), `image_id` (default `aws_default_ami_id`, override via
  extra-var), `instance_type` (default `aws_default_instance_type`, override
  via extra-var), `region` (default `aws_default_region`, override via
  extra-var), `key_name` (= `my_ssh_key_name`), `security_groups: [ansible-sg]`,
  `state` (`present` on create, `absent` on destroy / rescue).
- **Relationships**: 1:1 with an `AWS Inventory Entry` while it exists;
  identified by `instance_id` (persisted in the inventory entry) for destroy
  and rescue (FR-002, FR-009).
- **State transitions**: absent -> present (create, FR-002 — always a *new*
  instance, never reuse/restart) -> absent (destroy, FR-009; or rescue cleanup
  on create failure, FR-014). If already absent/terminated at destroy time, the
  termination step is skipped with a warning (FR-009, edge case).
- **Validation**: creation requires valid AWS credentials, surfaced via the
  AWS SDK's own authentication error on the first `amazon.aws` module call
  (FR-012) — no project-level pre-check, unlike the Hetzner Cloud provider's
  `HCLOUD_TOKEN` assertion.

## Shared Security Group (`ansible-sg`)

**Resource**: managed via `amazon.aws.ec2_security_group` (collection
`amazon.aws`), not stored as a repo file — its state lives in AWS. Long-lived,
shared across all aws-provider VMs and the pre-existing
`provisioners/aws-linux.yml` workflow.

- **Fields used** (FR-013): `name: ansible-sg`, `state: present`, an inbound
  rule allowing TCP/22 from `0.0.0.0/0`, in the resolved region.
- **Relationships**: many-to-one — every aws-provider EC2 instance references
  this security group by name.
- **State transitions**: ensured `present` (idempotent) before every
  `create-vm.yml -e provider=aws` launch. Never transitioned to `absent` by
  this feature's `destroy-vm.yml` (FR-013, Assumptions) — it persists across
  individual VM lifecycles.
- **Validation**: none beyond the idempotent `ec2_security_group` call
  succeeding.
