# Feature Specification: AWS EC2 Provider for Create and Destroy VM Playbooks

**Feature Branch**: `013-aws-vm-provider`
**Created**: 2026-06-14
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Create an AWS EC2 Linux server as a "VM" (Priority: P1)

An engineer wants a disposable cloud Linux server for testing or development,
without requiring local virtualization tooling (Tart/macOS), a local container
runtime (Docker), or a Hetzner Cloud account. They run a single command
requesting the AWS (`aws`) provider. The system picks the next unused hostname
from an AWS-specific pool, creates an EC2 instance, registers it in inventory,
and reports the hostname on completion. If the AWS hostname pool is exhausted,
the command refuses immediately — before any AWS API call — with an error
naming the exhausted pool.

**Why this priority**: This is the core value of the feature: it gives
engineers a disposable AWS test target with the same operational shape
(`create-vm.yml` / `destroy-vm.yml`) as the existing Tart, Docker, and Hetzner
Cloud providers, usable from any host with AWS credentials configured.

**Independent Test**: Run `create-vm.yml -e provider=aws` with AWS credentials
configured. The command completes, prints a hostname, and the server is
reachable via SSH using the registered inventory entry.

**Acceptance Scenarios**:

1. **Given** the AWS hostname pool has unused entries, **When**
   `create-vm.yml -e provider=aws` runs, **Then** a new EC2 instance is
   created, the next unused hostname from the AWS pool is assigned, the
   instance is registered in `inventories/aws_autogenerated.yml`, and the
   hostname is printed on completion.
2. **Given** one aws-provider server already exists, **When**
   `create-vm.yml -e provider=aws` runs again, **Then** a second instance is
   created with the next unused AWS hostname, and both servers coexist in
   `inventories/aws_autogenerated.yml`.
3. **Given** all hostnames in the AWS pool are in use, **When**
   `create-vm.yml -e provider=aws` runs, **Then** the playbook fails
   immediately — before any AWS API call — with an error message naming the
   exhausted pool file (`playbooks/vars/hostname_pool_aws.yml`).
4. **Given** AWS credentials are not configured or invalid, **When**
   `create-vm.yml -e provider=aws` runs, **Then** the playbook fails with an
   error from the AWS API call, after the hostname has already been selected
   (credential validation happens at the first AWS API call, matching the
   existing AWS provisioning workflow).

### User Story 2 - Destroy an aws-provider VM by hostname (Priority: P2)

An engineer is done with an aws-provider server and wants to remove it
cleanly. They run a single command specifying the provider and hostname. The
EC2 instance is terminated, its SSH host key is removed from the
project-scoped `inventories/aws_known_hosts` file, and all references to the
hostname are removed from `inventories/aws_autogenerated.yml`, leaving no
dangling entries.

**Why this priority**: Symmetrical to creation. Lifecycle management is
incomplete without clean teardown; dangling EC2 instances accumulate cost and
stale inventory entries cause confusion.

**Independent Test**: Given a running aws-provider server `arrakis`, run
`destroy-vm.yml -e provider=aws -e hostname=arrakis`. Verify the instance no
longer exists in AWS (terminated) and `arrakis` is absent from
`inventories/aws_autogenerated.yml`.

**Acceptance Scenarios**:

1. **Given** an aws-provider server exists in
   `inventories/aws_autogenerated.yml`, **When**
   `destroy-vm.yml -e provider=aws -e hostname=<name>` runs, **Then** the EC2
   instance is terminated, its host key is removed from
   `inventories/aws_known_hosts`, and the hostname no longer appears in
   `inventories/aws_autogenerated.yml`.
2. **Given** a hostname not present in `inventories/aws_autogenerated.yml`,
   **When** `destroy-vm.yml -e provider=aws -e hostname=<unknown>` runs,
   **Then** the playbook fails immediately — before any AWS action — with a
   clear error message identifying the unknown hostname.
3. **Given** a hostname present in `inventories/aws_autogenerated.yml` but
   whose instance no longer exists in AWS, **When**
   `destroy-vm.yml -e provider=aws -e hostname=<name>` runs, **Then** the
   playbook warns about the stale entry, skips the termination action
   gracefully, and still removes the hostname from
   `inventories/aws_autogenerated.yml`.

### User Story 3 - Backward compatibility (Priority: P3)

An engineer who has been using `create-vm.yml` and `destroy-vm.yml` with no
`provider` extra-var, or with `-e provider=docker` / `-e provider=hcloud`,
continues to get exactly the existing Tart, Docker, or Hetzner Cloud behavior,
with no change in output, files touched, or required tooling.

**Why this priority**: Backward compatibility is essential — existing
workflows built around the Tart provider (default), the Docker provider, and
the Hetzner Cloud provider must keep working unchanged after this feature
ships.

**Independent Test**: Run `create-vm.yml` (no `provider` extra-var),
`create-vm.yml -e provider=docker`, and `create-vm.yml -e provider=hcloud` on
a host where all providers are available. Verify all three commands behave
exactly as they did before this feature, touching only their respective
hostname pools and autogenerated inventory files.

**Acceptance Scenarios**:

1. **Given** `provider` is omitted, **When** `create-vm.yml` runs, **Then** it
   uses the Tart hostname pool and `inventories/tart_autogenerated.yml`,
   exactly as before this feature.
2. **Given** `provider=docker` or `provider=hcloud`, **When** `create-vm.yml`
   runs, **Then** it uses the respective hostname pool and autogenerated
   inventory file, exactly as before this feature.

### Edge Cases

- What happens when AWS credentials are unset, empty, or invalid? The
  `amazon.aws.ec2_security_group` or `amazon.aws.ec2_instance` call fails with
  the AWS SDK's own authentication error. No hostname-pool-specific
  pre-validation is performed (unlike `HCLOUD_TOKEN`, AWS credentials follow
  the standard AWS SDK credential-chain convention already used by
  `provisioners/aws-linux.yml`, and re-validating that chain would duplicate
  the SDK's own checks).
- What happens when `destroy-vm.yml -e provider=aws` is given a hostname that
  exists in `inventories/aws_autogenerated.yml` but the instance is already
  gone (e.g., manually terminated in the AWS console)? The playbook attempts
  cleanup, warns about the stale entry, and removes it from inventory (mirrors
  the Tart, Docker, and Hetzner Cloud providers' stale-entry handling).
- What happens if instance creation fails after the EC2 instance has already
  been created (e.g., SSH never becomes reachable)? A rescue step terminates
  the partially-created instance before the playbook fails, so no orphaned
  instances are left running and no cost accrues unnecessarily.
- What happens when AWS reassigns a previously used public IPv4 to a new
  instance with a different host key? Because `destroy-vm.yml` removes the old
  host key from `inventories/aws_known_hosts` (FR-009), the next create against
  that IP is treated as first contact under `accept-new` and trusted cleanly,
  with no spurious `REMOTE HOST IDENTIFICATION HAS CHANGED` failure. If a stale
  entry does remain (e.g. a create that was never destroyed), `accept-new` will
  correctly refuse the changed key and fail loudly — the intended
  man-in-the-middle protection.
- What if two operators run `create-vm.yml -e provider=aws` concurrently? No
  locking is implemented; duplicate hostname assignment is possible but
  accepted as out-of-scope, matching the existing Tart, Docker, and Hetzner
  Cloud providers' accepted risk for a single-operator project.
- What happens to the shared `ansible-sg` security group on destroy? It is
  intentionally left in place — it is a shared resource that other aws-provider
  VMs (and the pre-existing `provisioners/aws-linux.yml` workflow) may still
  depend on. `destroy-vm.yml -e provider=aws` only terminates the targeted
  instance and cleans up its own inventory/known_hosts entries.
- What happens to the pre-existing developer-server workflow (`provision.yml`,
  `provisioners/aws-linux.yml`, `provisioners/aws-windows.yml`, `destroy.yml`,
  `destroy-aws-tasks.yml`, `inventories/aws_ec2.yml`, the `rivendell` server)?
  They are unrelated to this feature, are not read, modified, or removed by
  either playbook, and remain unchanged except for read-only reuse of default
  values as described in FR-013.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `create-vm.yml` and `destroy-vm.yml` MUST accept a `provider`
  extra-var with allowed values `tart`, `docker`, `hcloud`, and `aws`,
  defaulting to `tart` when omitted. The default value MUST preserve the exact
  pre-existing behavior of both playbooks, and the existing `docker` and
  `hcloud` behavior MUST remain unchanged.
- **FR-002**: When `provider=aws`, `create-vm.yml` MUST create one new EC2
  instance per invocation; it MUST NOT reuse or restart an existing instance.
- **FR-003**: When `provider=aws`, the system MUST draw the server hostname
  from a single shared ordered list of Dune place names defined in
  `playbooks/vars/hostname_pool_aws.yml` (variable name `hostname_pool`,
  loaded conditionally so it does not collide with the Tart, Docker, or
  Hetzner Cloud pools of the same variable name). Hostnames are assigned
  sequentially; the first name not already present in
  `inventories/aws_autogenerated.yml` is selected on each invocation.
- **FR-004**: When `provider=aws`, the system MUST register each newly created
  instance in `inventories/aws_autogenerated.yml` (YAML format, groups `all` /
  `linux` / `aws`) before `create-vm.yml` completes, so configuration
  automation can target it. This file is distinct from, and not read, written,
  or replaced by, the existing dynamic-inventory plugin file
  `inventories/aws_ec2.yml`.
- **FR-005**: On create with `provider=aws`, the new inventory entry MUST
  include `ansible_host` (the instance's public IPv4 address returned by the
  AWS API), `ansible_port: 22`, `ansible_user: ubuntu` (matching the existing
  `admin_user_on_fresh_system` for `aws_ec2_linux`), and the instance's
  `instance-id` (so `destroy-vm.yml` can terminate by id without an extra
  Name-tag lookup). No password is stored; SSH access relies on the key pair
  registered with AWS (see FR-013). Because an EC2 instance is reachable over
  SSH from the public internet, the Ansible connection MUST **verify** the
  instance's SSH host key rather than disable verification:
  `ansible_ssh_common_args` MUST set `StrictHostKeyChecking=accept-new`
  against a project-scoped `UserKnownHostsFile`
  (`inventories/aws_known_hosts`), and MUST NOT use
  `StrictHostKeyChecking=no` with `UserKnownHostsFile=/dev/null`. This
  exposure-based policy — verify on internet-exposed cloud targets, disable on
  laptop-local ones (Tart, Docker) — is mandated by Constitution Principle XIV;
  rationale, threat model, and rejected alternatives are in ADR-003. The same
  host-key policy and the same `inventories/aws_known_hosts` file MUST also be
  used for the create-time SSH connection (e.g. `wait_for_connection`). Before
  that connection, the system MUST remove any stale host key for the new
  instance's IP from `inventories/aws_known_hosts` (idempotent
  `ssh-keygen -f inventories/aws_known_hosts -R <ip>`), so a public IP reused
  after a failed or skipped teardown is treated as clean first contact instead
  of failing with a changed-key error. This known_hosts file — and the `.old`
  backup `ssh-keygen -R` leaves beside it — is local machine state, like
  `inventories/aws_autogenerated.yml`, and MUST NOT be committed.
- **FR-006**: `create-vm.yml -e provider=aws` MUST print the assigned hostname
  on successful completion.
- **FR-007**: When the AWS hostname pool is exhausted, `create-vm.yml -e
  provider=aws` MUST fail immediately with an error message naming the
  exhausted pool file (`playbooks/vars/hostname_pool_aws.yml`), without making
  any AWS API call.
- **FR-008**: `destroy-vm.yml -e provider=aws` MUST accept `hostname`
  (required) identifying a server present in
  `inventories/aws_autogenerated.yml`.
- **FR-009**: `destroy-vm.yml -e provider=aws` MUST terminate the EC2 instance
  by the `instance-id` stored in the inventory entry (FR-005), remove the
  instance's SSH host key from the project-scoped known_hosts file written on
  create (`inventories/aws_known_hosts`, FR-005) via
  `ssh-keygen -f inventories/aws_known_hosts -R <ansible_host>` — mirroring
  `destroy-aws-tasks.yml`'s `known_hosts` cleanup step, but targeting the
  scoped file rather than the operator's `~/.ssh/known_hosts` — and remove the
  hostname from `inventories/aws_autogenerated.yml` on completion. The
  known_hosts removal MUST be idempotent (guarded so a re-run does not report a
  spurious change). If the instance is already absent or already terminated in
  AWS, the termination step MUST be skipped gracefully with a warning, and both
  the known_hosts cleanup and the inventory cleanup MUST still occur (mirroring
  the Tart, Docker, and Hetzner Cloud providers' stale-entry behavior). Removing
  the host key on destroy prevents a stale entry from raising a false
  `REMOTE HOST IDENTIFICATION HAS CHANGED` alarm when AWS later reassigns the
  public IP to a different instance.
- **FR-010**: `destroy-vm.yml -e provider=aws` MUST fail with a clear,
  actionable error if the specified hostname is not found in
  `inventories/aws_autogenerated.yml`, before any AWS action is taken.
- **FR-011**: `playbooks/vars/hostname_pool_aws.yml` ships with ten ordered
  Dune place names: `arrakis`, `caladan`, `giedi-prime`, `kaitain`, `ix`,
  `chusuk`, `tleilax`, `salusa-secundus`, `wallach-ix`, `ginaz`. All entries are
  available for assignment; none are pre-reserved. The pool is extensible:
  operators may append names to this file at any time.
- **FR-012**: The system MUST authenticate to the AWS API using the existing
  AWS SDK credential chain (environment variables, shared credentials file, or
  instance profile — the same convention used by `provisioners/aws-linux.yml`
  and `destroy-aws-tasks.yml`; the `amazon.aws` collection is already declared
  in `requirements.yml`). No new credential mechanism is introduced. Unlike the
  Hetzner Cloud provider's explicit `HCLOUD_TOKEN` pre-check, AWS credential
  validity is determined by the first `amazon.aws` module call (the AWS SDK's
  own error), since the SDK credential chain has multiple valid sources and
  re-implementing its resolution/validation would duplicate SDK behavior.
- **FR-013**: When `provider=aws`, the instance MUST be created via the
  `amazon.aws.ec2_instance` module with: `image_id` defaulting to the existing
  `aws_default_ami_id` (`inventories/group_vars/aws_ec2_linux/vars.yml`);
  `instance_type` defaulting to the existing `aws_default_instance_type`
  (`inventories/group_vars/aws_ec2_linux/vars.yml`); `region` defaulting to the
  existing `aws_default_region` (`inventories/group_vars/aws_ec2/vars.yml`);
  and `key_name` set to the existing vault-configured `my_ssh_key_name`
  (`inventories/group_vars/all/vars.yml`). These defaults MUST be reused via
  `include_vars` rather than duplicated. Operators may override
  `instance_type`, `image_id`, and `region` via extra-vars. Before launching,
  the system MUST ensure a shared security group named `ansible-sg` exists
  (idempotent `amazon.aws.ec2_security_group`, `state: present`, allowing
  inbound TCP/22 from `0.0.0.0/0`) and attach it to the instance. This security
  group is shared across aws-provider VMs (and the pre-existing
  `provisioners/aws-linux.yml` workflow) and is NOT deleted by
  `destroy-vm.yml`.
- **FR-014**: If instance creation fails after the EC2 instance has been
  created, the system MUST run a rescue step that terminates the
  partially-created instance (`amazon.aws.ec2_instance: state: absent`) before
  failing the playbook, mirroring the Tart provider's `tart stop`/`tart
  delete`, the Docker provider's `docker stop`/`docker rm`, and the Hetzner
  Cloud provider's rescue behavior.
- **FR-015**: This feature MUST NOT modify, read, or remove `provision.yml`,
  `provisioners/aws-linux.yml`, `provisioners/aws-windows.yml`, `destroy.yml`,
  `destroy-aws-tasks.yml`, or `inventories/aws_ec2.yml`. These belong to the
  pre-existing developer-server provisioning workflow and are out of scope,
  except for read-only reuse of default values from
  `inventories/group_vars/aws_ec2/vars.yml`,
  `inventories/group_vars/aws_ec2_linux/vars.yml`, and
  `inventories/group_vars/all/vars.yml` as described in FR-013.
- **FR-016**: This feature MUST NOT change the Tart, Docker, or Hetzner Cloud
  providers' behavior, files, or output when `provider` is omitted, `tart`,
  `docker`, or `hcloud`.
- **FR-017**: This feature is AWS EC2 **Linux** only. Windows Server EC2
  instances (`inventories/group_vars/aws_ec2_windows/vars.yml`,
  `provisioners/aws-windows.yml`) are out of scope and are neither read nor
  modified.

### Key Entities

- **AWS Hostname Pool**: An ordered list of Dune place names defined in
  `playbooks/vars/hostname_pool_aws.yml` (variable `hostname_pool`). Ships with
  ten names; all entries are available for assignment. Entries are either
  available or in use and are allocated sequentially — the first name not
  already in `inventories/aws_autogenerated.yml` is chosen next. This pool is
  independent from, and disjoint with, the Tart, Docker, and Hetzner Cloud
  hostname pools.
- **AWS Inventory Entry**: A record of an aws-provider server's hostname,
  connection details (host, port, user, SSH common-args), EC2 instance-id, and
  group membership, persisted in `inventories/aws_autogenerated.yml` so
  automation can target the server.
- **EC2 Instance**: The disposable cloud server resource created via the
  `amazon.aws.ec2_instance` module, using the existing default AMI, instance
  type, region, registered SSH key, and the shared `ansible-sg` security group.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An engineer with AWS credentials configured creates a disposable
  EC2 Linux server with a single command (`create-vm.yml -e provider=aws`); no
  other extra-vars are required for the common case.
- **SC-002**: An engineer destroys any aws-provider server with a single
  command specifying only the provider and hostname.
- **SC-003**: AWS hostname pool exhaustion is detected and reported before any
  AWS API call is made.
- **SC-004**: Concurrent server creation is not a supported use case; no
  locking mechanism is implemented. Simultaneous invocations may assign the
  same hostname. This risk is accepted for a single-operator project (YAGNI),
  matching the existing Tart, Docker, and Hetzner Cloud providers' accepted
  risk.
- **SC-005**: An engineer who omits `provider`, or passes `provider=docker` or
  `provider=hcloud`, continues to get identical behavior, output, and file
  changes as before this feature shipped.
- **SC-006**: The AWS provider verifies the instance's SSH host key
  (`StrictHostKeyChecking=accept-new` against a project-scoped known_hosts
  file); a *changed* host key on a known IP causes the connection to fail
  loudly rather than silently re-trusting. Destroy removes the host key so a
  reassigned public IP does not raise a false-positive identification-changed
  alarm.

## Assumptions

- The AWS hostname pool ships with ten Dune place names (`arrakis`, `caladan`,
  `giedi-prime`, `kaitain`, `ix`, `chusuk`, `tleilax`, `salusa-secundus`,
  `wallach-ix`, `ginaz`), all available for assignment. There are no
  pre-existing aws-provider servers to reserve; the pool is a clean slate.
- The operator already has AWS credentials configured via the standard AWS SDK
  credential chain, and the SSH private key matching `my_ssh_key_name`
  (registered as an AWS key pair) available to their local SSH agent, as
  required by the existing `provisioners/aws-linux.yml` workflow. No new
  credential mechanism is introduced.
- The existing developer-server resources (`rivendell`, managed by
  `provision.yml` / `destroy.yml`) are unrelated to this feature, are not read,
  generated, or modified by it, and remain unchanged.
- If an instance exists in AWS but is absent from
  `inventories/aws_autogenerated.yml`, `destroy-vm.yml -e provider=aws` treats
  it as unknown and fails; manual cleanup of orphaned instances is out of
  scope, matching the Tart, Docker, and Hetzner Cloud providers' equivalent
  assumption.
- Default instance type (`aws_default_instance_type`), AMI
  (`aws_default_ami_id`), and region (`aws_default_region`) are reused from
  existing configuration rather than introducing aws-specific resource
  defaults.
- The shared security group `ansible-sg` is created on first use (idempotent)
  and is treated as a long-lived shared resource, not owned or deleted by any
  single `create-vm.yml`/`destroy-vm.yml` invocation.
