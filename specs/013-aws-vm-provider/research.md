# Phase 0 Research: AWS EC2 Provider for Create/Destroy VM Playbooks

No `NEEDS CLARIFICATION` markers remain in the Technical Context — the spec
(`spec.md`) and ADR-003 already resolve every open question. This document
records the concrete technical decisions derived from them, for traceability
into Phase 1 design and `tasks.md`.

## Decision: Reuse the hcloud/docker/tart task-file pattern, plus one extra step

- **Decision**: `playbooks/tasks/create/aws.yml` and
  `playbooks/tasks/destroy/aws.yml` follow the same shape as
  `tasks/create/hcloud.yml` / `tasks/destroy/hcloud.yml`: load pool → load/init
  autogenerated inventory → pick/validate hostname → create or destroy the
  resource → wait for SSH (create only) → write inventory; create wraps the
  resource-creating steps in a `block`/`rescue`. Create additionally ensures
  the shared `ansible-sg` security group exists before launching the instance
  (FR-013).
- **Rationale**: Constitution IV (YAGNI) and XI (DRY) — a fourth provider that
  introduces a different shape than the three existing ones would be a new
  pattern to learn and maintain for no benefit. The inventory schema
  (`all`/`linux`/`<provider>` groups) is already established. The
  security-group step is the only AWS-specific addition, required because EC2
  instances (unlike Hetzner Cloud servers) need an explicit security group to
  permit inbound SSH.
- **Alternatives considered**: A dedicated `roles/aws_vm/` role with a
  Molecule scenario — rejected because Molecule cannot provision real EC2
  instances (the resource under test is the cloud API itself, not a
  container), and because `tart`/`docker`/`hcloud` already established that
  VM-lifecycle providers live as task files, not roles.

## Decision: Hostname selection algorithm

- **Decision**: Identical to `tart`/`docker`/`hcloud` —
  `available_hostnames = hostname_pool | reject('in', used_hostnames) | list`,
  `hostname = available_hostnames | first`, fail before any API call if empty
  (FR-003, FR-007, FR-011).
- **Rationale**: FR-003 explicitly specifies "first name not already present",
  i.e. the existing algorithm. `used_hostnames` is read from
  `current_inventory.all.hosts.keys()`, computed the same way as the other
  three providers.
- **Alternatives considered**: Random selection — rejected, FR-003 requires
  sequential/deterministic assignment.

## Decision: Instance creation via `amazon.aws.ec2_instance`, defaults via `include_vars`

- **Decision**: `amazon.aws.ec2_instance` with `state: present`, `name:
  hostname`, `image_id` / `instance_type` / `region` defaulted from
  `aws_default_ami_id` and `aws_default_instance_type`
  (`inventories/group_vars/aws_ec2_linux/vars.yml`) and `aws_default_region`
  (`inventories/group_vars/aws_ec2/vars.yml`), each overridable via
  extra-vars; `key_name: "{{ my_ssh_key_name }}"` from
  `inventories/group_vars/all/vars.yml`; `security_groups: [ansible-sg]`;
  tags `Name: hostname`.
- **Rationale**: FR-013 mandates reuse via `include_vars` rather than
  duplicating these values; `provisioners/aws-linux.yml` already demonstrates
  the same `include_vars` + `amazon.aws.ec2_instance` pattern, so this is a
  proven approach in this repo.
- **Alternatives considered**: Hardcoding the defaults in
  `hostname_pool_aws.yml` or a new `aws_vm_defaults.yml` — rejected, violates
  XI (DRY) and FR-013 explicitly forbids duplication.

## Decision: Shared, idempotent `ansible-sg` security group, never deleted

- **Decision**: Before launching, run `amazon.aws.ec2_security_group` with
  `name: ansible-sg`, `state: present`, an inbound rule allowing TCP/22 from
  `0.0.0.0/0`, in the resolved region. `destroy-vm.yml -e provider=aws` does
  not touch this security group.
- **Rationale**: FR-013 / Assumptions — the security group is a shared,
  long-lived resource (also used by `provisioners/aws-linux.yml`). Per-VM
  security groups would multiply AWS resources with no benefit (YAGNI), and
  deleting a shared resource on a single VM's destroy would break other
  aws-provider VMs and the pre-existing provisioning workflow.
  `ec2_security_group` with `state: present` is natively idempotent — running
  it repeatedly with the same rule set reports no change after the first run.
- **Alternatives considered**: Per-VM security group named after the hostname,
  deleted on destroy — rejected as unnecessary AWS resource churn (YAGNI) and
  inconsistent with `destroy-aws-tasks.yml`'s existing "shared `ansible-sg`"
  convention, which this feature reuses rather than replaces.

## Decision: SSH host-key policy — `accept-new` + scoped known_hosts (ADR-003)

- **Decision**: Both the create-time `wait_for_connection` and the persisted
  `ansible_ssh_common_args` in `inventories/aws_autogenerated.yml` use
  `-o StrictHostKeyChecking=accept-new -o UserKnownHostsFile=<repo>/inventories/aws_known_hosts`.
  Before `wait_for_connection`, create runs
  `ssh-keygen -f inventories/aws_known_hosts -R <ip>` guarded so it reports
  `changed` only when a key was actually removed (idempotent per Principle I).
  Destroy runs the identical removal — on both the normal and stale-entry
  paths — before removing the inventory entry (FR-009).
- **Rationale**: This is the literal decision of ADR-003 (already accepted),
  driven by Constitution Principle XIV, and is the same decision already
  applied to the hcloud provider. `tart`/`docker` keep
  `StrictHostKeyChecking=no` + `/dev/null` (laptop-local, unchanged — FR-016).
- **Alternatives considered**: All rejected alternatives are documented in
  ADR-003; not re-litigated here.

## Decision: Idempotent `ssh-keygen -R` guard

- **Decision**: Run `ssh-keygen -f inventories/aws_known_hosts -F <ip>` first
  (exit 0 = key present, exit 1 = absent, `failed_when: false`, `changed_when:
  false`); run `ssh-keygen -f inventories/aws_known_hosts -R <ip>` only `when:`
  the `-F` check found a key, with `changed_when: true` on that conditional
  step (and `failed_when: false` since the file may not yet exist on first
  create).
- **Rationale**: Identical reasoning to `tasks/{create,destroy}/hcloud.yml` —
  required for Principle I's idempotency bar (no spurious `changed` reports).
- **Alternatives considered**: Always running `-R` with `changed_when: false`
  — rejected, would mask a real removal as a no-op and diverge from the
  established convention.

## Decision: Rescue-block cleanup on create failure

- **Decision**: Mirror `tasks/create/hcloud.yml`'s `block`/`rescue`: on any
  failure inside the block (after the instance is created), `rescue` runs
  `amazon.aws.ec2_instance: state: absent, instance_ids: [<id>]` with
  `failed_when: false`, then `fail`s with a message naming the hostname and
  instructing re-run (FR-014).
- **Rationale**: FR-014 explicitly mirrors the tart/docker/hcloud rescue
  pattern; `amazon.aws.ec2_instance` with `state: absent` and `instance_ids` is
  the documented teardown call already used by `destroy-aws-tasks.yml`.
- **Alternatives considered**: None — FR-014 specifies the mechanism.

## Decision: Destroy terminates by stored `instance_id`, not by Name-tag lookup

- **Decision**: The inventory entry written on create stores the instance's
  `instance_id` (FR-005). `destroy-vm.yml -e provider=aws` reads it directly
  and calls `amazon.aws.ec2_instance_info: instance_ids: [<id>]` to check
  current state (stale-entry detection), then
  `amazon.aws.ec2_instance: state: absent, instance_ids: [<id>]` to terminate.
- **Rationale**: Avoids a Name-tag-based `ec2_instance_info` filter query
  (used by `destroy-aws-tasks.yml` for its bulk "terminate everything"
  use case, which doesn't apply here — this feature targets exactly one
  instance by hostname). Storing the id is one extra scalar field in the
  inventory entry and removes ambiguity if two instances ever shared a Name
  tag.
- **Alternatives considered**: Filter by `tag:Name == hostname` at destroy
  time (mirrors `destroy-aws-tasks.yml`) — rejected as an unnecessary extra
  API round-trip and a possible ambiguity source; the instance-id is already
  known at create time and free to persist.

## Decision: `.gitignore` addition

- **Decision**: Add `inventories/aws_autogenerated.yml` next to the existing
  `tart_autogenerated.yml`/`docker_autogenerated.yml`/`hcloud_autogenerated.yml`
  lines. `inventories/aws_known_hosts` (and its `.old` backup) is already
  covered by the existing `inventories/*_known_hosts*` pattern — no change
  needed there.
- **Rationale**: FR-004 / FR-005 state these are local generated state, not
  committed, matching the other three providers' convention.
