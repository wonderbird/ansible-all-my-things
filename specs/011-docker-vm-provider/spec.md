# Feature Specification: Docker Provider for Create and Destroy VM Playbooks

**Feature Branch**: `011-docker-vm-provider`
**Created**: 2026-06-11
**Status**: Draft

## Clarifications

### Session 2026-06-11

- Q: FR-013 says the `ansible-vm-docker:latest` image is "built once... then reused." What should determine whether a build is triggered on a given `create-vm.yml -e provider=docker` run? → A: Build only if image tag `ansible-vm-docker:latest` doesn't exist; never auto-rebuild (idempotent existence check, mirroring the `creates:`-guard pattern used by the Tart provider's `tart clone` step).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Create a local Docker container as a "VM" (Priority: P1)

An engineer on a host with Docker installed (any Linux host, or any host that
is not macOS ARM64) wants a local disposable Linux environment for testing or
development, without requiring the Tart/macOS toolchain. They run a single
command requesting the Docker provider. The system picks the next unused
hostname from a Docker-specific pool, builds (if needed) and starts a
container, registers it in inventory, and reports the hostname on completion.
If the Docker hostname pool is exhausted, the command refuses immediately —
before any container action — with an error naming the exhausted pool.

**Why this priority**: This is the core value of the feature: it gives
engineers without access to macOS/Tart a local, disposable test target with
the same operational shape as the existing Tart provider.

**Independent Test**: Run `create-vm.yml -e provider=docker` with no other
extra-vars. The command completes, prints a hostname, and the container is
reachable via SSH using the registered inventory entry.

**Acceptance Scenarios**:

1. **Given** the Docker hostname pool has unused entries, **When**
   `create-vm.yml -e provider=docker` runs, **Then** a new container is
   created, the next unused hostname from the Docker pool is assigned, the
   container is registered in `inventories/docker_autogenerated.yml`, and the
   hostname is printed on completion.
2. **Given** one Docker container already exists, **When**
   `create-vm.yml -e provider=docker` runs again, **Then** a second container
   is created with the next unused Docker hostname, and both containers
   coexist in `inventories/docker_autogenerated.yml`.
3. **Given** all hostnames in the Docker pool are in use, **When**
   `create-vm.yml -e provider=docker` runs, **Then** the playbook fails
   immediately — before any container build or run action — with an error
   message naming the exhausted pool file
   (`playbooks/vars/hostname_pool_docker.yml`).
4. **Given** the container image has not been built yet, **When**
   `create-vm.yml -e provider=docker` runs, **Then** the image is built once
   (tagged `ansible-vm-docker:latest`) before the container is started.

---

### User Story 2 - Destroy a Docker-provider VM by hostname (Priority: P2)

An engineer is done with a Docker-provider container and wants to remove it
cleanly. They run a single command specifying the provider and hostname. The
container is stopped and removed, and all references to the hostname are
removed from `inventories/docker_autogenerated.yml`, leaving no dangling
entries.

**Why this priority**: Symmetrical to creation. Lifecycle management is
incomplete without clean teardown, and dangling containers and stale
inventory entries cause confusion and resource waste.

**Independent Test**: Given a running Docker-provider container `tatooine`,
run `destroy-vm.yml -e provider=docker -e hostname=tatooine`. Verify the
container no longer exists and `tatooine` is absent from
`inventories/docker_autogenerated.yml`.

**Acceptance Scenarios**:

1. **Given** a Docker-provider container exists in
   `inventories/docker_autogenerated.yml`, **When**
   `destroy-vm.yml -e provider=docker -e hostname=<name>` runs, **Then** the
   container is stopped and removed, and the hostname no longer appears in
   `inventories/docker_autogenerated.yml`.
2. **Given** a hostname not present in `inventories/docker_autogenerated.yml`,
   **When** `destroy-vm.yml -e provider=docker -e hostname=<unknown>` runs,
   **Then** the playbook fails immediately — before any Docker action — with a
   clear error message identifying the unknown hostname.
3. **Given** a hostname present in `inventories/docker_autogenerated.yml` but
   whose container no longer exists in Docker, **When**
   `destroy-vm.yml -e provider=docker -e hostname=<name>` runs, **Then** the
   playbook warns about the stale entry, skips the `docker stop`/`docker rm`
   actions gracefully, and still removes the hostname from
   `inventories/docker_autogenerated.yml`.

---

### User Story 3 - Default behavior remains the Tart provider (Priority: P3)

An engineer who has been using `create-vm.yml` and `destroy-vm.yml` without
any `provider` extra-var continues to get exactly the existing Tart-based
behavior, with no change in output, files touched, or required tooling.

**Why this priority**: Backward compatibility is essential — existing
workflows, documentation, and any in-progress automation built around the
Tart provider must keep working unchanged after this feature ships.

**Independent Test**: Run `create-vm.yml` (no `provider` extra-var) and
`destroy-vm.yml -e hostname=<name>` (no `provider` extra-var) on a host where
both providers are available. Verify both commands behave exactly as they did
before this feature, touching only `playbooks/vars/hostname_pool_tart.yml` and
`inventories/tart_autogenerated.yml`.

**Acceptance Scenarios**:

1. **Given** `provider` is omitted, **When** `create-vm.yml` runs, **Then** it
   uses the Tart hostname pool (`playbooks/vars/hostname_pool_tart.yml`) and
   `inventories/tart_autogenerated.yml`, exactly as before this feature.
2. **Given** `provider` is omitted, **When** `destroy-vm.yml -e hostname=<name>`
   runs, **Then** it uses `inventories/tart_autogenerated.yml`, exactly as
   before this feature.

### Edge Cases

- What happens when Docker is unavailable (daemon not running, `docker`
  command missing) at container creation time? The playbook fails with an
  explicit error before any hostname is consumed from the Docker pool.
- What happens when `destroy-vm.yml -e provider=docker` is given a hostname
  that exists in `inventories/docker_autogenerated.yml` but the container is
  already gone (e.g., manually removed)? The playbook attempts cleanup, warns
  about the stale entry, and removes it from inventory (mirrors the Tart
  provider's stale-entry handling).
- What happens if container creation fails after the container has already
  been started (e.g., SSH never becomes reachable)? A rescue step stops and
  removes the partially-created container before the playbook fails, so no
  orphaned containers are left behind.
- What if two operators run `create-vm.yml -e provider=docker` concurrently?
  No locking is implemented; duplicate hostname assignment is possible but
  accepted as out-of-scope, matching the existing Tart provider's accepted
  risk for a single-operator project.
- What happens to the pre-existing `inventories/vagrant_docker.yml` and
  `test/docker/*` (the old Vagrant+Docker test setup)? They are unrelated to
  this feature, are not read, modified, or removed by either playbook, and
  remain unchanged.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `create-vm.yml` and `destroy-vm.yml` MUST accept a `provider`
  extra-var with allowed values `tart` and `docker`, defaulting to `tart` when
  omitted. The default value MUST preserve the exact pre-existing behavior of
  both playbooks.
- **FR-002**: When `provider=docker`, `create-vm.yml` MUST create one new
  Docker container per invocation; it MUST NOT reuse or restart an existing
  container.
- **FR-003**: When `provider=docker`, the system MUST draw the container
  hostname from a single shared ordered list of Star Wars planet names defined
  in `playbooks/vars/hostname_pool_docker.yml` (variable name `hostname_pool`,
  loaded conditionally so it does not collide with the Tart pool of the same
  variable name). Hostnames are assigned sequentially; the first name not
  already present in `inventories/docker_autogenerated.yml` is selected on
  each invocation.
- **FR-004**: When `provider=docker`, the system MUST register each newly
  created container in `inventories/docker_autogenerated.yml` (YAML format,
  groups `all` / `linux` / `docker`) before `create-vm.yml` completes, so
  configuration automation can target it.
- **FR-005**: On create with `provider=docker`, the new inventory entry MUST
  include `ansible_host: 127.0.0.1`, `ansible_port` (the host port dynamically
  published for container port 22, discovered via `docker port`),
  `ansible_user: root`, and `ansible_ssh_common_args` disabling strict host key
  checking (each fresh container has a new SSH host key, mirroring the Tart
  provider's inventory entries). The entry MUST NOT include `ansible_ssh_pass`:
  authentication is key-based (ADR-004). The operator's `my_ssh_public_key` is
  installed into the container's `/root/.ssh/authorized_keys` at create time
  over a one-time password connection (`docker_root_password`, via `sshpass`),
  and the inventory then connects by key, matching the cloud providers.
- **FR-006**: `create-vm.yml -e provider=docker` MUST print the assigned
  hostname on successful completion.
- **FR-007**: When the Docker hostname pool is exhausted, `create-vm.yml -e
  provider=docker` MUST fail immediately with an error message naming the
  exhausted pool file (`playbooks/vars/hostname_pool_docker.yml`), without
  performing any container build or run action.
- **FR-008**: `destroy-vm.yml -e provider=docker` MUST accept `hostname`
  (required) identifying a container present in
  `inventories/docker_autogenerated.yml`.
- **FR-009**: `destroy-vm.yml -e provider=docker` MUST stop and remove the
  Docker container (`docker stop` + `docker rm`) and remove the hostname from
  `inventories/docker_autogenerated.yml` on completion. If the container is
  already absent from Docker, the stop/remove steps MUST be skipped gracefully
  with a warning, and the inventory cleanup MUST still occur (mirroring the
  Tart provider's stale-entry behavior).
- **FR-010**: `destroy-vm.yml -e provider=docker` MUST fail with a clear,
  actionable error if the specified hostname is not found in
  `inventories/docker_autogenerated.yml`, before any Docker action is taken.
- **FR-011**: `playbooks/vars/hostname_pool_docker.yml` ships with ten ordered
  Star Wars planet names: `tatooine`, `coruscant`, `naboo`, `alderaan`, `hoth`,
  `dagobah`, `endor`, `kamino`, `mustafar`, `yavin`. All entries are available
  for assignment; none are pre-reserved. The pool is extensible: operators may
  append names to this file at any time.
- **FR-012**: When `provider=docker`, the container MUST be created with
  Ansible-variable-defined resource defaults: 2 vCPUs and 4 GB RAM (via
  `docker run --cpus 2 --memory 4g`). Operators may override these via
  extra-vars. Disk size limits are not applicable to this provider and are not
  configured.
- **FR-013**: The system MUST provide a container image definition at
  `playbooks/files/docker/Dockerfile`, based on `ubuntu:24.04` with systemd as
  the container entrypoint and `sshd` enabled, configured for root login with
  password authentication (`PermitRootLogin yes`,
  `PasswordAuthentication yes`, root password set at build time via
  `chpasswd` from `docker_root_password`). On each `create-vm.yml -e
  provider=docker` invocation, the system MUST check whether an image tagged
  `ansible-vm-docker:latest` already exists; if it does not exist, the system
  builds it from the Dockerfile, then reuses the existing or newly-built image
  for the container instance. The system MUST NOT rebuild an existing
  `ansible-vm-docker:latest` image (no automatic rebuild-on-change detection),
  mirroring the idempotent `creates:`-guard pattern used by the Tart
  provider's image-clone step.
- **FR-014**: When `provider=docker`, each container MUST be started with a
  dynamically published SSH port (`-p 127.0.0.1::22`), running in privileged
  mode (`--privileged`), with `--name` and `--hostname` both set to the
  assigned pool hostname.
- **FR-015**: If container creation fails after the container has been
  started, the system MUST run a rescue step that stops and removes the
  partially-created container (`docker stop` + `docker rm`) before failing the
  playbook, mirroring the Tart provider's `tart stop`/`tart delete` rescue
  behavior.
- **FR-016**: This feature MUST NOT modify, read, or remove
  `inventories/vagrant_docker.yml` or any file under `test/docker/`. These
  belong to the pre-existing Vagrant+Docker test setup and are out of scope.
- **FR-017**: This feature MUST NOT change the Tart provider's behavior,
  files, or output when `provider` is omitted or set to `tart`.

### Key Entities

- **Docker Hostname Pool**: An ordered list of Star Wars planet names defined
  in `playbooks/vars/hostname_pool_docker.yml` (variable `hostname_pool`).
  Ships with ten names; all entries are available for assignment. Entries are
  either available or in use and are allocated sequentially — the first name
  not already in `inventories/docker_autogenerated.yml` is chosen next. This
  pool is independent from, and disjoint with, the Tart hostname pool.
- **Docker Inventory Entry**: A record of a Docker-provider container's
  hostname, connection details (host, port, user, password,
  SSH-common-args), and group membership, persisted in
  `inventories/docker_autogenerated.yml` so automation can target the
  container.
- **Container Image**: The reusable `ansible-vm-docker:latest` image, built
  once from `playbooks/files/docker/Dockerfile`, providing a systemd-enabled,
  SSH-reachable Ubuntu 24.04 environment with root/password authentication.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An engineer on any host with Docker installed creates a local
  test container with a single command (`create-vm.yml -e provider=docker`);
  no other extra-vars are required for the common case.
- **SC-002**: An engineer destroys any Docker-provider container with a single
  command specifying only the provider and hostname.
- **SC-003**: Docker hostname pool exhaustion is detected and reported before
  any container build or run action is attempted, with an error message naming
  the depleted pool file.
- **SC-004**: Concurrent container creation is not a supported use case; no
  locking mechanism is implemented. Simultaneous invocations may assign the
  same hostname. This risk is accepted for a single-operator project (YAGNI),
  matching the existing Tart provider's accepted risk.
- **SC-005**: An engineer who never passes `provider` continues to get
  identical Tart-provider behavior, output, and file changes as before this
  feature shipped.

## Assumptions

- The Docker hostname pool ships with ten Star Wars planet names (`tatooine`,
  `coruscant`, `naboo`, `alderaan`, `hoth`, `dagobah`, `endor`, `kamino`,
  `mustafar`, `yavin`), all available for assignment. There are no
  pre-existing Docker-provider containers to reserve; the pool is a clean
  slate.
- `sshpass` is already a required dependency for the Tart provider and is
  assumed to be present for the Docker provider's password-based SSH
  authentication; no new dependency is introduced.
- The Docker provider is usable on any host with a working Docker daemon,
  including Linux hosts where the Tart provider is unavailable. Architecture
  (ARM64 vs AMD64) is determined by the host's Docker daemon and the
  `ubuntu:24.04` base image's multi-arch support; no architecture-specific
  branching is introduced.
- If a container exists in Docker but is absent from
  `inventories/docker_autogenerated.yml`, `destroy-vm.yml -e provider=docker`
  treats it as unknown and fails; manual cleanup of orphaned containers is out
  of scope, matching the Tart provider's equivalent assumption.
- `inventories/vagrant_docker.yml` and `test/docker/*` are pre-existing
  artifacts of an unrelated, older Vagrant+Docker test setup. They are not
  read, generated, or modified by this feature.
