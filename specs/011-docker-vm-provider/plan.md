# Implementation Plan: Docker Provider for Create and Destroy VM Playbooks

**Branch**: `011-docker-vm-provider` | **Date**: 2026-06-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/011-docker-vm-provider/spec.md`

## Summary

Add a `docker` provider to the existing `create-vm.yml` / `destroy-vm.yml`
playbooks, selected via `-e provider=docker` (default remains `tart`,
preserving exact existing behavior). The Docker provider draws hostnames from
a dedicated Star Wars planet name pool (`playbooks/vars/hostname_pool_docker.yml`),
builds a reusable `ansible-vm-docker:latest` image (once, idempotently) from a
new `playbooks/files/docker/Dockerfile`, runs a privileged systemd container
with a dynamically published SSH port on `127.0.0.1`, and registers/removes
entries in `inventories/docker_autogenerated.yml`. The implementation mirrors
the structure of the Tart provider (`playbooks/tasks/create/tart.yml` and
`playbooks/tasks/destroy/tart.yml`) with parallel `docker.yml` task files and
a provider dispatch in the two top-level playbooks.

## Technical Context

**Language/Version**: Ansible (YAML), ansible-core (project default)
**Primary Dependencies**: `docker` CLI/daemon (any Linux host, or any
non-macOS-ARM64 host); `sshpass` for password auth (already required by the
Tart provider)
**Storage**: Autogenerated YAML inventory `inventories/docker_autogenerated.yml`;
Docker-managed container storage (`docker run`/`docker build` state)
**Testing**: Manual validation on a host with a working Docker daemon, per
`CONTRIBUTING.md` (control-node localhost orchestration — same testing
approach as the Tart provider; Molecule does not apply, see Constitution
Check)
**Target Platform**: Any host with Docker installed (Linux primary; macOS/
other hosts where Docker Desktop is available) — control node only, no
managed remote hosts
**Project Type**: Ansible playbooks (localhost CLI tools)
**Performance Goals**: N/A — single-operator, interactive use
**Constraints**: No locking; single-operator project (YAGNI, SC-004); MUST
NOT alter Tart provider behavior, files, or output (FR-017); MUST NOT touch
`inventories/vagrant_docker.yml` or `test/docker/*` (FR-016)
**Scale/Scope**: Up to 10 containers (Docker hostname pool size); one
container per invocation

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Idempotency | PASS (constrained) | Image build guarded by an existence check on `ansible-vm-docker:latest` (idempotent `creates:`-equivalent, FR-013). Container run is NOT idempotent by design — each invocation creates a new container with a new pool hostname (FR-002), mirroring `tart clone`'s per-invocation semantics. Inventory update via load→modify→write is idempotent. Pool check via inventory membership is idempotent. Shell tasks parsing `docker port` / `docker inspect` output use `changed_when: false` and explicit validation (Principle XII). |
| II. Role-First | JUSTIFIED EXCEPTION (same as 009) | Runs entirely on `localhost`, orchestrating the local Docker daemon as a VM-provider — not against managed hosts. Identical exception already accepted for the Tart provider in `specs/009-create-destroy-vm/plan.md`; the new task files (`playbooks/tasks/create/docker.yml`, `playbooks/tasks/destroy/docker.yml`) extend that same accepted pattern rather than introducing a new one. No Molecule scenario (control-node orchestration of `docker` CLI itself, not a target-host role). No versioned tool pins introduced → version-update registration N/A (the `ubuntu:24.10` base image tag is fixed per FR-013, not a pinned tool version). |
| III. Test Locally Before Cloud | PASS | Feature is local-container management only; cloud targets not applicable. Validate by running both playbooks with `-e provider=docker` on a host with Docker before any cloud work, per `CONTRIBUTING.md`. |
| IV. Simplicity (YAGNI) | PASS | Two providers only (`tart`, `docker`), dispatched via a single `provider` extra-var with a documented default. No locking. No image rebuild-on-change detection (FR-013). Resource limits (`--cpus`, `--memory`) are simple Ansible variables with defaults, mirroring `vm_cpus`/`vm_memory_mb` in `create-vm.yml`. |
| XII. Fail Loud | PASS (constrained) | Docker hostname pool exhaustion MUST assert before any container build/run action (FR-007). Unknown hostname on destroy MUST fail before any Docker action (FR-010). `docker port` output parsing MUST be validated (assert non-empty port) before writing inventory. Docker daemon unavailability surfaces as an explicit task failure (no `ignore_errors`). |
| X. No External-System References | PASS | No beads IDs or tracker references in any durable artefact. |
| XI. Avoid Duplication | PASS | Provider dispatch logic lives once in each top-level playbook (`create-vm.yml`, `destroy-vm.yml`) via `include_tasks` keyed on `provider`. Docker-specific pool/inventory/image logic lives in `playbooks/tasks/create/docker.yml` and `playbooks/tasks/destroy/docker.yml`, parallel to but not copy-pasted from the Tart task files (shared *shape*, provider-specific *content*, consistent with how `tart.yml` create/destroy already diverge from each other). |
| XIII. No Empty Artefacts | PASS | No placeholder files. Every new file (`hostname_pool_docker.yml`, `docker_credentials.yml`, `docker.yml` task files, `Dockerfile`, `docker_autogenerated.yml`) carries real content from first commit. `docker_autogenerated.yml` is generated at first `create-vm.yml -e provider=docker` run, not committed empty. |

**Structure Decision**: Extend the existing playbook-centric layout from
`specs/009-create-destroy-vm/plan.md` with a parallel set of provider-specific
files. `create-vm.yml` and `destroy-vm.yml` gain a top-level `provider`
extra-var (default `tart`) that selects which `tasks/create/<provider>.yml` /
`tasks/destroy/<provider>.yml` file is included — the Tart branch is an
unmodified `include_tasks: tasks/create/tart.yml` / `tasks/destroy/tart.yml`,
guaranteeing FR-017 (no behavior change for the default/`tart` path). The
Docker branch follows the same load-pool → check-exhaustion →
provider-action → update-inventory shape as the Tart task files, substituting
`docker build`/`docker run`/`docker port`/`docker stop`/`docker rm` for the
`tart` CLI equivalents, and `playbooks/files/docker/Dockerfile` /
`playbooks/vars/hostname_pool_docker.yml` /
`playbooks/vars/docker_credentials.yml` /
`inventories/docker_autogenerated.yml` for the corresponding Tart-side
artefacts.

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|--------------------------------------|
| Principle II: implementation logic in `playbooks/tasks/create/docker.yml` and `playbooks/tasks/destroy/docker.yml` instead of roles | Same justification already accepted for the Tart provider (`specs/009-create-destroy-vm/plan.md`): control-node orchestration of a local VM-provider CLI (`docker`), not configuration of a managed host. | A role wrapping `docker build`/`docker run` on localhost would have no `molecule/default/` scenario relevant to its actual function (it provisions Molecule's own kind of target, it isn't one), and would duplicate the existing accepted `playbooks/tasks/` pattern under a different name for no added reuse. |
