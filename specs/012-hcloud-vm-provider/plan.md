# Implementation Plan: Hetzner Cloud Provider for Create and Destroy VM Playbooks

**Branch**: `012-hcloud-vm-provider` | **Date**: 2026-06-13 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/012-hcloud-vm-provider/spec.md`

## Summary

Add `hcloud` as a third provider for `playbooks/create-vm.yml` and
`playbooks/destroy-vm.yml`, alongside the existing `tart` (default) and
`docker` providers. The new provider creates/destroys a Hetzner Cloud server
via the `hetzner.hcloud.server` module, drawing hostnames from a new ordered
pool (`playbooks/vars/hostname_pool_hcloud.yml`, Lord of the Rings names),
registering/removing it in `inventories/hcloud_autogenerated.yml`, and
applying the exposure-based SSH host-key policy from ADR-003
(`StrictHostKeyChecking=accept-new` against a project-scoped
`inventories/hcloud_known_hosts`). The implementation mirrors the existing
`tart`/`docker` task-file pattern (`playbooks/tasks/create/<provider>.yml`,
`playbooks/tasks/destroy/<provider>.yml`) with no new abstractions, and reuses
existing hcloud defaults (`hcloud_default_type`, `hcloud_default_location`,
`my_ssh_key_name`) via `include_vars` per FR-013.

## Technical Context

**Language/Version**: Ansible YAML (playbooks/tasks), consistent with the
existing `tart`/`docker` provider implementations in this repo.

**Primary Dependencies**: `hetzner.hcloud` collection (`>=1.0.0`), already
declared in `requirements.yml` and already used by
`provisioners/hcloud-linux.yml` / `destroy-hcloud-tasks.yml`.

**Storage**: Flat YAML files only —
`playbooks/vars/hostname_pool_hcloud.yml` (static, committed pool of 10 LOTR
names); `inventories/hcloud_autogenerated.yml` (generated, gitignored,
mirrors `tart_autogenerated.yml`/`docker_autogenerated.yml` structure);
`inventories/hcloud_known_hosts` (generated, gitignored, project-scoped SSH
known_hosts per ADR-003).

**Testing**: No Molecule scenario. The new task files are imperative
localhost orchestration against the Hetzner Cloud API (create/destroy a real
cloud resource), not a reusable configuration role — the same is true of the
existing `tart`/`docker` create/destroy task files, neither of which has a
Molecule scenario. Validation follows the spec's "Independent Test" sections:
run `create-vm.yml -e provider=hcloud` with `HCLOUD_TOKEN` set against the
real Hetzner Cloud API, verify SSH reachability and the
`inventories/hcloud_autogenerated.yml` entry, then run
`destroy-vm.yml -e provider=hcloud -e hostname=<name>` and verify cleanup.
Constitution Principle III ("Test Locally Before Cloud") does not add a local
substitute step here — there is no local equivalent of "create a Hetzner
Cloud server"; the cloud run *is* the test, exactly as scoped by the spec's
Independent Test.

**Target Platform**: Controller = `localhost` (existing `create-vm.yml`
/ `destroy-vm.yml` play target, `gather_facts: true`). Managed node = a
Hetzner Cloud server running Ubuntu 24.04, reachable over SSH from the public
internet on its assigned IPv4.

**Project Type**: Single Ansible project — task-file additions to the
existing `playbooks/` tree. No new roles, no new playbooks.

**Performance Goals**: N/A (one-shot provisioning command; existing
`wait_for_connection` timeout conventions from `tart`/`docker` apply).

**Constraints**: Constitution Principles I (idempotency — `ssh-keygen -R`
guards, rescue cleanup), IV (YAGNI — reuse the docker/tart pattern verbatim),
XI (DRY — `include_vars` for `hcloud_default_type`/`hcloud_default_location`
/`my_ssh_key_name` per FR-013), XII (fail loud — `HCLOUD_TOKEN`, pool
exhaustion, unknown hostname, all checked before any API call), XIV
(SSH host-key verification by exposure, per ADR-003). FR-001 through FR-016
in [spec.md](./spec.md) are the binding requirements.

**Scale/Scope**: 2 new task files (`playbooks/tasks/create/hcloud.yml`,
`playbooks/tasks/destroy/hcloud.yml`), 1 new vars file
(`playbooks/vars/hostname_pool_hcloud.yml`), small edits to
`playbooks/tasks/assert-provider-profile.yml` and the `assert_provider_usage_hint`
strings in `create-vm.yml`/`destroy-vm.yml`, 1 new line in `.gitignore`.
