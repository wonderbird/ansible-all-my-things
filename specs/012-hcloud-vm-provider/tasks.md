# Tasks: Hetzner Cloud Provider for Create and Destroy VM Playbooks

**Input**: Design documents from `/specs/012-hcloud-vm-provider/`

**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/,
quickstart.md (all present)

"Independent Test" per story, run against the real Hetzner Cloud API per

to enable independent implementation and testing.

- **[P]**: Can run in parallel (different files, no dependencies)
- **[Story]**: Maps task to US1 / US2 / US3 from spec.md
- File paths are exact and relative to repo root unless stated otherwise

## Phase 3: User Story 1 - Create an hcloud VM (Priority: P1) 🎯 MVP

**Goal**: `ansible-playbook playbooks/create-vm.yml -e provider=hcloud`
creates one new Hetzner Cloud server, registers it in
`inventories/hcloud_autogenerated.yml` with a verified-host-key SSH
connection, and prints the assigned hostname.

**Independent Test** (spec.md US1): With `HCLOUD_TOKEN` exported, run
`ansible-playbook playbooks/create-vm.yml -e provider=hcloud`. The command
completes, prints `VM created with hostname: <name>`, and
`ansible -i inventories/hcloud_autogenerated.yml <name> -m ping` succeeds.

  `fangorn`, `bree`, `edoras`, `minas-tirith`, `mordor`, `isengard`, `rohan`,
  `lothlorien`, `helms-deep`. Follow the header-comment style of
  `playbooks/vars/hostname_pool_docker.yml` (explain the pool's
  independence/disjointness from the tart/docker pools per FR-003).

- [X] T004 [P] [US1] In `playbooks/create-vm.yml`, update
  `assert_provider_usage_hint` from
  `Run: ansible-playbook playbooks/create-vm.yml -e provider=<tart|docker>`
  to
  `Run: ansible-playbook playbooks/create-vm.yml -e provider=<tart|docker|hcloud>`
  (plan.md Project Structure; FR-001).

- [X] T005 [US1] Create `playbooks/tasks/create/hcloud.yml` implementing the
  full create algorithm, mirroring the six-step shape of
  `playbooks/tasks/create/docker.yml` (research.md "Reuse the docker/tart
  task-file pattern verbatim") with a leading header comment describing the
  algorithm:
  1. **Preconditions before any API call** (contracts/create-vm-hcloud.md):
     - `include_vars` the three existing default-source files per FR-013
       (mirroring the `vars_files` list in `provisioners/hcloud-linux.yml`):
       `inventories/group_vars/hcloud/vars.yml` (for
       `hcloud_default_location`), `inventories/group_vars/hcloud_linux/vars.yml`
       (for `hcloud_default_type`), and `inventories/group_vars/all/vars.yml`
       (for `my_ssh_key_name`). Do not duplicate these values (XI/DRY).
     - Read the Hetzner API token via `lookup('env', 'HCLOUD_TOKEN')` and
       `assert` it is defined and non-empty, failing with a message naming
       `HCLOUD_TOKEN`, before touching the hostname pool (FR-012).
     - Load `playbooks/vars/hostname_pool_hcloud.yml` (T003).
     - Check for / load `inventories/hcloud_autogenerated.yml`
       (`ansible.builtin.stat` + conditional `include_vars`, else initialize
       an empty `current_inventory` with `all`/`linux`/`hcloud` groups —
       same pattern as `tasks/create/docker.yml` lines 40-60).
     - Compute `used_hostnames` from `current_inventory.all.hosts.keys()`,
       then `available_hostnames` via
       `hostname_pool | reject('in', used_hostnames) | list`
       (research.md "Hostname selection algorithm").
     - `fail` if `available_hostnames | length == 0`, naming
       `playbooks/vars/hostname_pool_hcloud.yml` (FR-007).
     - Set `hostname` to `available_hostnames | first` (FR-003).
  2. **Resolve creation parameters** (FR-013, contracts table): set
     `server_type` from extra-var `hcloud_server_type` defaulting to
     `hcloud_default_type`; `location` from extra-var
     `hcloud_server_location` defaulting to `hcloud_default_location`;
     `image` from extra-var `image` defaulting to `ubuntu-24.04`.
  3. **block**: create the server and register it —
     - `hetzner.hcloud.server`: `state: present`, `name: "{{ hostname }}"`,
       `server_type`, `location`, `image`, `ssh_keys: ["{{ my_ssh_key_name }}"]`,
       `api_token: "{{ <token from step 1> }}"`. Register the result (e.g.
       `hcloud_server_result`) to read
       `hcloud_server_result.hcloud_server.ipv4_address` (FR-002, FR-013).
     - Idempotent stale-key guard on the new server's IP against
       `inventories/hcloud_known_hosts` per research.md "Idempotent
       `ssh-keygen -R` guard": `ssh-keygen -f
       {{ playbook_dir }}/../inventories/hcloud_known_hosts -F <ip>`
       (`failed_when: false`, `changed_when: false`) to detect a stale entry,
       then `ssh-keygen -f {{ playbook_dir }}/../inventories/hcloud_known_hosts
       -R <ip>` only `when:` a key was found, `failed_when: false`,
       `changed_when: true` on that conditional step (FR-005).
     - `wait_for_connection` against the new server's IPv4, port 22, as
       `root`, with `ansible_ssh_common_args:
       "-o StrictHostKeyChecking=accept-new -o UserKnownHostsFile={{
       playbook_dir }}/../inventories/hcloud_known_hosts"` (FR-005,
       contracts/create-vm-hcloud.md effect 3). Resolve the exact
       `UserKnownHostsFile` path form (absolute vs. `{{ playbook_dir }}/..`
       relative) so the *same* string is valid both for this connection and
       once persisted into `inventories/hcloud_autogenerated.yml` — see
       data-model.md "Hcloud Inventory Entry" for the persisted form.
     - Build `updated_inventory` adding the new host under `all`/`linux`/
       `hcloud` groups with `ansible_host` (the IPv4 from step above),
       `ansible_port: 22`, `ansible_user: root`, and the same
       `ansible_ssh_common_args` as above (FR-004, FR-005, data-model.md
       "Hcloud Inventory Entry" — no `ansible_ssh_pass` field, unlike
       docker).
     - Write `inventories/hcloud_autogenerated.yml` via
       `ansible.builtin.copy` with the
       `"# AUTO-GENERATED — do not edit manually. Overwritten by create-vm.yml
       and destroy-vm.yml.\n"` header and `to_nice_yaml(indent=2)`, `mode:
       "0644"` (same as `tasks/create/docker.yml`).
     - Print `VM created with hostname: {{ hostname }}` via
       `ansible.builtin.debug` (FR-006).
  4. **rescue**: `hetzner.hcloud.server: state: absent, name: "{{ hostname
     }}", api_token: ...` with `failed_when: false`, then `fail` with a
     message naming `{{ hostname }}` and instructing re-run of
     `create-vm.yml -e provider=hcloud` (FR-014,
     contracts/create-vm-hcloud.md "Effects on failure"). Both
     `inventories/hcloud_autogenerated.yml` and `inventories/hcloud_known_hosts`
     remain unchanged on this path since the inventory write happens after
     `wait_for_connection` succeeds.

## Phase 4: User Story 2 - Destroy an hcloud VM by hostname (Priority: P2)

**Goal**: `ansible-playbook playbooks/destroy-vm.yml -e provider=hcloud -e
hostname=<name>` deletes the Hetzner Cloud server, removes its SSH host key
from `inventories/hcloud_known_hosts`, and removes the entry from
`inventories/hcloud_autogenerated.yml`, with no dangling state either way.

**Independent Test** (spec.md US2): Given a running hcloud-provider server
`gondor` (created in Phase 3), run `ansible-playbook
playbooks/destroy-vm.yml -e provider=hcloud -e hostname=gondor`. Verify
the server no longer exists in Hetzner Cloud and `gondor` is absent from
`inventories/hcloud_autogenerated.yml` and from
`inventories/hcloud_known_hosts`.

### Implementation for User Story 2

- [X] T007 [P] [US2] In `playbooks/destroy-vm.yml`, update
  `assert_provider_usage_hint` from
  `Run: ansible-playbook playbooks/destroy-vm.yml -e provider=<tart|docker> -e hostname=<name>`


- [X] T008 [US2] Create `playbooks/tasks/destroy/hcloud.yml` implementing
  the full destroy algorithm, mirroring the shape of
  `playbooks/tasks/destroy/docker.yml` with a leading header comment
  describing the algorithm:
  1. `hostname` is already asserted defined/non-empty by `destroy-vm.yml`'s
     shared pre_task — no re-check needed (contracts/destroy-vm-hcloud.md
     precondition 2).
  2. Check for / load `inventories/hcloud_autogenerated.yml` (`stat` +
     conditional `include_vars`, else empty `current_inventory` with
     `all`/`linux`/`hcloud` groups — same pattern as
     `tasks/destroy/docker.yml` lines 16-36).
  3. `fail` if `hostname not in (current_inventory.all.hosts | default({}))`,
     naming the unknown hostname, before any Hetzner Cloud action (FR-008,
     FR-010, contracts/destroy-vm-hcloud.md precondition 3).
  4. Capture `ansible_host` (the server's public IPv4) for `hostname` from
     `current_inventory.all.hosts[hostname]` before it is removed — needed
     for the known_hosts cleanup in step 6 (data-model.md "Hcloud Known
     Hosts": "keyed by `ansible_host`... from the corresponding Hcloud
     Inventory Entry").
  5. `include_vars` `inventories/group_vars/all/vars.yml` for
     `my_ssh_key_name` only if needed for the API call (the delete-by-name
     call below does not need it); read `HCLOUD_TOKEN` via
     `lookup('env', 'HCLOUD_TOKEN')` (no `assert` needed here per FR-012,
     which scopes the check to create — but the token is still required to
     make the delete API call, so fail loudly per Principle XII if it is
     empty, naming `HCLOUD_TOKEN`).
  6. Server deletion (FR-009, contracts/destroy-vm-hcloud.md):
     - Check whether a server named `hostname` exists in Hetzner Cloud
       (e.g. `hetzner.hcloud.server_info` filtered by name, or attempt
       `state: absent` and detect "already absent"). If it exists, delete it
       via `hetzner.hcloud.server: state: absent, name: "{{ hostname }}",
       api_token: ...`.
     - If it does not exist, print a warning identifying `hostname` as a
       stale entry (mirrors `tasks/destroy/docker.yml`'s
       "WARNING: ... Proceeding with inventory cleanup only.") and skip the
       deletion call — both paths continue to steps 7-8 (FR-009 stale-entry
       behavior, contracts "Effects on success — server already absent").
  7. Idempotent known_hosts cleanup against the captured `ansible_host` IP
     using the same `-F` / conditional `-R` guard as T005 step 3, on
     `inventories/hcloud_known_hosts` (FR-009, research.md "Idempotent
     `ssh-keygen -R` guard"). Runs on both the normal and stale-entry paths.
  8. Build `cleaned_inventory` removing `hostname` from `all`/`linux`/
     `hcloud` hosts (same `dict2items | selectattr('key', '!=', hostname) |
     items2dict` pattern as `tasks/destroy/docker.yml` lines 71-89), write
     it back to `inventories/hcloud_autogenerated.yml` with the same
     AUTO-GENERATED header and `to_nice_yaml(indent=2)`, `mode: "0644"`.
  9. `assert` `hostname not in (cleaned_inventory.all.hosts | default({}))`,
     `fail_msg: "Failed to remove '{{ hostname }}' from
     inventories/hcloud_autogenerated.yml"` (mirrors
     `tasks/destroy/docker.yml` lines 97-100).

  BLOCKS both User Story 1 and User Story 2 (the shared `provider` allow-list

  strings it implicitly re-validates) being in place; otherwise has no new

## Implementation Strategy

### MVP First (User Story 1 Only)

1. Phase 1 (T001) + Phase 2 (T002).
2. Phase 3 (T003-T006) — `create-vm.yml -e provider=hcloud` works
   end-to-end against the real Hetzner Cloud API.
3. **STOP and VALIDATE**: T006's Independent Test passes.

### Incremental Delivery

1. Setup + Foundational (T001-T002).
2. User Story 1 (T003-T006) → validate → MVP.
3. User Story 2 (T007-T009) → validate → full create/destroy lifecycle.
4. User Story 3 (T010) → regression check, no new code.
5. Polish (T011).

### Cleanup note

T006 and T009 each create a real, billable Hetzner Cloud server. T009
destroys the server T006 created; if T009 is skipped or fails, manually run
`destroy-vm.yml -e provider=hcloud -e hostname=<name>` to avoid leaving a
billable resource running.
