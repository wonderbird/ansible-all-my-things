# Phase 1 Data Model: Hetzner Cloud Provider

All entities are flat YAML structures read/written by Ansible task files —
there is no database or ORM. This document describes the shape of each file
and the fields within it.

## Hcloud Hostname Pool

**File**: `playbooks/vars/hostname_pool_hcloud.yml`
**Variable**: `hostname_pool` (list of strings)

```yaml
hostname_pool:
  - rivendell
  - hobbiton
  - bree
  - edoras
  - minas-tirith
  - mordor
  - isengard
  - moria
  - lothlorien
  - helms-deep
```

- **Fields**: ordered list of hostname strings (Lord of the Rings place
  names). All 10 ship "available" — no entry is pre-reserved (FR-011, spec
  Assumptions).
- **Relationships**: an entry is "in use" iff it appears as a key under
  `all.hosts` in `inventories/hcloud_autogenerated.yml`. The set is disjoint
  from `hostname_pool_tart.yml` and `hostname_pool_docker.yml` (different
  naming themes; FR-003).
- **State transitions**: available -> in-use (on create, FR-003) -> available
  again (on destroy, FR-009, by removal from the autogenerated inventory).
  Pool entries themselves never change; only inventory membership changes.
- **Validation**: none beyond non-empty strings; the pool is "extensible" —
  operators may append entries (FR-011). Exhaustion (no available entry) is a
  hard failure before any API call (FR-007).

## Hcloud Inventory Entry

**File**: `inventories/hcloud_autogenerated.yml` (generated, gitignored)
**Format**: static YAML inventory, groups `all` / `linux` / `hcloud`
(FR-004), written via `to_nice_yaml`, identical top-level shape to
`tart_autogenerated.yml` / `docker_autogenerated.yml`.

```yaml
# AUTO-GENERATED — do not edit manually. Overwritten by create-vm.yml and destroy-vm.yml.
all:
  hosts:
    rivendell:
      ansible_host: <public IPv4 from hetzner.hcloud.server>
      ansible_port: 22
      ansible_user: root
      ansible_ssh_common_args: >-
        -o StrictHostKeyChecking=accept-new
        -o UserKnownHostsFile=<repo>/inventories/hcloud_known_hosts
linux:
  hosts:
    rivendell: {}
hcloud:
  hosts:
    rivendell: {}
```

- **Fields per host** (FR-005):
  - `ansible_host` — the server's public IPv4 address, returned by
    `hetzner.hcloud.server` (`hcloud_server.ipv4_address`).
  - `ansible_port` — always `22`.
  - `ansible_user` — always `root` (matches
    `admin_user_on_fresh_system` for hcloud Linux servers).
  - `ansible_ssh_common_args` — fixed string applying ADR-003's
    `accept-new` + project-scoped `UserKnownHostsFile=inventories/hcloud_known_hosts`.
  - No `ansible_ssh_pass` field — auth is key-based via the registered
    Hetzner SSH key (FR-005), unlike `tart`/`docker`.
- **Relationships**: one entry per live hcloud server; hostname is the dict
  key in all three groups (`all`, `linux`, `hcloud`), matching the
  `tart`/`docker` shape exactly (just a different leaf-group name).
- **State transitions**: absent -> present (create, FR-004, written before
  `create-vm.yml` completes) -> absent (destroy, FR-009, removed on
  completion even for stale entries).
- **Validation**: `destroy-vm.yml -e provider=hcloud` requires the given
  `hostname` to be a key under `all.hosts` before taking any Hetzner Cloud
  action (FR-008, FR-010).
- **Distinctness**: this file is separate from, and never read/written/replaced
  by, the pre-existing dynamic-inventory plugin file `inventories/hcloud.yml`
  (FR-004, FR-015).

## Hcloud Known Hosts

**File**: `inventories/hcloud_known_hosts` (generated, gitignored, plus
`.old` backup from `ssh-keygen -R`)
**Format**: standard OpenSSH `known_hosts` file, one line per
`(ip, host-key)` pair.

- **Relationships**: keyed by `ansible_host` (the server's public IPv4) from
  the corresponding `Hcloud Inventory Entry`. Created implicitly by
  `accept-new` on first connection (FR-005); entries removed via
  `ssh-keygen -f inventories/hcloud_known_hosts -R <ip>` on create (stale-IP
  guard) and on destroy (FR-005, FR-009).
- **State transitions**: absent -> contains key for IP (first successful
  connection, TOFU) -> absent for that IP (destroy, or pre-create guard for a
  reused IP). A *changed* key for a known IP is a hard SSH failure
  (`REMOTE HOST IDENTIFICATION HAS CHANGED`), by design (ADR-003, FR-005 edge
  case).

## Hetzner Cloud Server

**Resource**: managed via `hetzner.hcloud.server` (collection
`hetzner.hcloud`), not stored as a repo file — its state lives in Hetzner
Cloud.

- **Fields used** (FR-013): `name` (= hostname from the pool), `server_type`
  (default `hcloud_default_type`, override via extra-var), `location`
  (default `hcloud_default_location`, override via extra-var), `image`
  (default `ubuntu-24.04`, override via extra-var), `ssh_keys` (=
  `my_ssh_key_name`), `state` (`present` on create, `absent` on destroy /
  rescue).
- **Relationships**: 1:1 with an `Hcloud Inventory Entry` while it exists;
  identified by `name` (the pool hostname) for both create lookups and destroy
  (FR-002, FR-009).
- **State transitions**: absent -> present (create, FR-002 — always a *new*
  server, never reuse/restart) -> absent (destroy, FR-009; or rescue cleanup
  on create failure, FR-014). If already absent at destroy time, the deletion
  step is skipped with a warning (FR-009, edge case).
- **Validation**: creation requires a non-empty `HCLOUD_TOKEN` (FR-012),
  checked before pool consumption.
