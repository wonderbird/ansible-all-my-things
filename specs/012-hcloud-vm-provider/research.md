# Phase 0 Research: Hetzner Cloud Provider for Create/Destroy VM Playbooks

No `NEEDS CLARIFICATION` markers remain in the Technical Context — the spec
(`spec.md`) and ADR-003 already resolve every open question. This document
records the concrete technical decisions derived from them, for traceability
into Phase 1 design and `tasks.md`.

## Decision: Reuse the docker/tart task-file pattern verbatim

- **Decision**: `playbooks/tasks/create/hcloud.yml` and
  `playbooks/tasks/destroy/hcloud.yml` follow the exact same six-step shape as
  `tasks/create/docker.yml` / `tasks/destroy/docker.yml`: load pool → load/init
  autogenerated inventory → pick/validate hostname → create or destroy the
  resource → wait for SSH (create only) → write inventory; create wraps the
  resource-creating steps in a `block`/`rescue`.
- **Rationale**: Constitution IV (YAGNI) and XI (DRY) — a third provider that
  introduces a different shape than the two existing ones would be a new
  pattern to learn and maintain for no benefit. The inventory schema
  (`all`/`linux`/`<provider>` groups) is already established.
- **Alternatives considered**: A dedicated `roles/hcloud_vm/` role with a
  Molecule scenario — rejected because Molecule cannot provision real Hetzner
  Cloud servers (the resource under test is the cloud API itself, not a
  container), and because `tart`/`docker` already established that VM-lifecycle
  providers live as task files, not roles.

## Decision: Hostname selection algorithm

- **Decision**: Identical to `tart`/`docker` —
  `available_hostnames = hostname_pool | reject('in', used_hostnames) | list`,
  `hostname = available_hostnames | first`, fail before any API call if empty
  (FR-003, FR-007, FR-011).
- **Rationale**: FR-003 explicitly specifies "first name not already present",
  i.e. the existing algorithm. `used_hostnames` is read from
  `current_inventory.all.hosts.keys()`, computed the same way as the other two
  providers.
- **Alternatives considered**: Random selection — rejected, FR-003 requires
  sequential/deterministic assignment.

## Decision: Server creation via `hetzner.hcloud.server`, defaults via `include_vars`

- **Decision**: `hetzner.hcloud.server` with `state: present`, `name:
  hostname`, `server_type` / `location` / `image` defaulted from
  `hcloud_default_type` (`inventories/group_vars/hcloud_linux/vars.yml`),
  `hcloud_default_location` (`inventories/group_vars/hcloud/vars.yml`), and
  literal `ubuntu-24.04` respectively, each overridable via extra-vars;
  `ssh_keys: ["{{ my_ssh_key_name }}"]` from `inventories/group_vars/all/vars.yml`.
  Token comes from `lookup('env', 'HCLOUD_TOKEN')`, asserted non-empty before
  any pool/API interaction (FR-012).
- **Rationale**: FR-013 mandates reuse via `include_vars` rather than
  duplicating these values; `provisioners/hcloud-linux.yml` already
  demonstrates the same three `include_vars` + `hcloud_token` pattern, so this
  is a proven approach in this repo.
- **Alternatives considered**: Hardcoding the defaults in
  `hostname_pool_hcloud.yml` or a new `hcloud_vm_defaults.yml` — rejected,
  violates XI (DRY) and FR-013 explicitly forbids duplication.

## Decision: SSH host-key policy — `accept-new` + scoped known_hosts (ADR-003, Option A)

- **Decision**: Both the create-time `wait_for_connection` and the persisted
  `ansible_ssh_common_args` in `inventories/hcloud_autogenerated.yml` use
  `-o StrictHostKeyChecking=accept-new -o UserKnownHostsFile=<repo>/inventories/hcloud_known_hosts`.
  Before `wait_for_connection`, create runs
  `ssh-keygen -f inventories/hcloud_known_hosts -R <ip>` guarded so it reports
  `changed` only when a key was actually removed (idempotent per Principle I).
  Destroy runs the identical removal — on both the normal and stale-entry
  paths — before removing the inventory entry (FR-009).
- **Rationale**: This is the literal decision of ADR-003 (already accepted,
  2026-06-13), driven by Constitution Principle XIV. `tart`/`docker` keep
  `StrictHostKeyChecking=no` + `/dev/null` (laptop-local, unchanged — FR-016).
- **Alternatives considered**: All five rejected alternatives are documented
  in ADR-003 (B–G); not re-litigated here.

## Decision: Idempotent `ssh-keygen -R` guard

- **Decision**: Run `ssh-keygen -f inventories/hcloud_known_hosts -F <ip>`
  first (exit 0 = key present, exit 1 = absent, `failed_when: false`,
  `changed_when: false`); run `ssh-keygen -f inventories/hcloud_known_hosts -R
  <ip>` only `when:` the `-F` check found a key, with `changed_when: true` on
  that conditional step (and `failed_when: false` since the file may not yet
  exist on first create).
- **Rationale**: `ssh-keygen -R` on a non-existent file/entry still exits 0 in
  recent OpenSSH but historically could warn; gating on `-F` makes the
  `changed` status accurate and the step a no-op (not just harmless) when
  there is nothing to remove — required by Principle I's idempotency bar
  ("running more than once produces the same system state... and reports no
  spurious changes" per the existing convention in `destroy/docker.yml`'s
  `changed_when` usage).
- **Alternatives considered**: Always running `-R` with `changed_when: false`
  — rejected, would mask a real removal as "ok" and diverge from how
  `tasks/destroy/*.yml` report `changed` for the analogous stop/delete steps.

## Decision: Rescue-block cleanup on create failure

- **Decision**: Mirror `tasks/create/docker.yml`'s
  `block`/`rescue`: on any failure inside the block (after the server is
  created), `rescue` runs `hetzner.hcloud.server: state: absent, name:
  hostname` with `failed_when: false`, then `fail`s with a message naming the
  hostname and instructing re-run (FR-014).
- **Rationale**: FR-014 explicitly mirrors the tart/docker rescue pattern;
  `hetzner.hcloud.server` with `state: absent` is the documented teardown call
  already used by `destroy-hcloud-tasks.yml`.
- **Alternatives considered**: None — FR-014 specifies the mechanism.

## Decision: `.gitignore` addition

- **Decision**: Add `inventories/hcloud_autogenerated.yml` next to the
  existing `tart_autogenerated.yml`/`docker_autogenerated.yml` lines.
  `inventories/hcloud_known_hosts` (and its `.old` backup) is already covered
  by the existing `inventories/*_known_hosts*` pattern — no change needed
  there.
- **Rationale**: FR-004 / FR-005 state these are local generated state, not
  committed, matching the other two providers' convention.
