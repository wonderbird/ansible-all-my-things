# Feature Specification: Hetzner Cloud Provider for Create and Destroy VM Playbooks

**Feature Branch**: `012-hcloud-vm-provider`
**Created**: 2026-06-12
**Status**: Draft

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Create a Hetzner Cloud server as a "VM" (Priority: P1)

An engineer wants a disposable cloud Linux server for testing or development,
without requiring local virtualization tooling (Tart/macOS) or a local
container runtime (Docker). They run a single command requesting the Hetzner
Cloud (`hcloud`) provider. The system picks the next unused hostname from an
hcloud-specific pool, creates a Hetzner Cloud server, registers it in
inventory, and reports the hostname on completion. If the hcloud hostname pool
is exhausted, the command refuses immediately — before any Hetzner API call —
with an error naming the exhausted pool.

**Why this priority**: This is the core value of the feature: it gives
engineers a disposable cloud test target with the same operational shape
(`create-vm.yml` / `destroy-vm.yml`) as the existing Tart and Docker
providers, usable from any host with an `HCLOUD_TOKEN`.

**Independent Test**: Run `create-vm.yml -e provider=hcloud` with
`HCLOUD_TOKEN` set. The command completes, prints a hostname, and the server
is reachable via SSH using the registered inventory entry.

**Acceptance Scenarios**:

1. **Given** the hcloud hostname pool has unused entries, **When**
   `create-vm.yml -e provider=hcloud` runs, **Then** a new Hetzner Cloud
   server is created, the next unused hostname from the hcloud pool is
   assigned, the server is registered in `inventories/hcloud_autogenerated.yml`,
   and the hostname is printed on completion.
2. **Given** one hcloud-provider server already exists, **When**
   `create-vm.yml -e provider=hcloud` runs again, **Then** a second server is
   created with the next unused hcloud hostname, and both servers coexist in
   `inventories/hcloud_autogenerated.yml`.
3. **Given** all hostnames in the hcloud pool are in use, **When**
   `create-vm.yml -e provider=hcloud` runs, **Then** the playbook fails
   immediately — before any Hetzner Cloud API call — with an error message
   naming the exhausted pool file (`playbooks/vars/hostname_pool_hcloud.yml`).
4. **Given** `HCLOUD_TOKEN` is not set or empty, **When**
   `create-vm.yml -e provider=hcloud` runs, **Then** the playbook fails
   immediately with an explicit error identifying `HCLOUD_TOKEN`, before any
   hostname is consumed from the pool.

### User Story 2 - Destroy an hcloud-provider VM by hostname (Priority: P2)

An engineer is done with an hcloud-provider server and wants to remove it
cleanly. They run a single command specifying the provider and hostname. The
Hetzner Cloud server is deleted, its SSH host key is removed from the
project-scoped `inventories/hcloud_known_hosts` file, and all references to
the hostname are removed from `inventories/hcloud_autogenerated.yml`, leaving
no dangling entries.

**Why this priority**: Symmetrical to creation. Lifecycle management is
incomplete without clean teardown; dangling cloud servers accumulate cost and
stale inventory entries cause confusion.

**Independent Test**: Given a running hcloud-provider server `rivendell`, run
`destroy-vm.yml -e provider=hcloud -e hostname=rivendell`. Verify the server
no longer exists in Hetzner Cloud and `rivendell` is absent from
`inventories/hcloud_autogenerated.yml`.

**Acceptance Scenarios**:

1. **Given** an hcloud-provider server exists in
   `inventories/hcloud_autogenerated.yml`, **When**
   `destroy-vm.yml -e provider=hcloud -e hostname=<name>` runs, **Then** the
   Hetzner Cloud server is deleted, its host key is removed from
   `inventories/hcloud_known_hosts`, and the hostname no longer appears in
   `inventories/hcloud_autogenerated.yml`.
2. **Given** a hostname not present in `inventories/hcloud_autogenerated.yml`,
   **When** `destroy-vm.yml -e provider=hcloud -e hostname=<unknown>` runs,
   **Then** the playbook fails immediately — before any Hetzner Cloud action —
   with a clear error message identifying the unknown hostname.
3. **Given** a hostname present in `inventories/hcloud_autogenerated.yml` but
   whose server no longer exists in Hetzner Cloud, **When**
   `destroy-vm.yml -e provider=hcloud -e hostname=<name>` runs, **Then** the
   playbook warns about the stale entry, skips the server-deletion action
   gracefully, and still removes the hostname from
   `inventories/hcloud_autogenerated.yml`.

### User Story 3 - Backward compatibility (Priority: P3)

An engineer who has been using `create-vm.yml` and `destroy-vm.yml` with no
`provider` extra-var, or with `-e provider=docker`, continues to get exactly
the existing Tart or Docker behavior, with no change in output, files
touched, or required tooling.

**Why this priority**: Backward compatibility is essential — existing
workflows built around the Tart provider (default) and the Docker provider
must keep working unchanged after this feature ships.

**Independent Test**: Run `create-vm.yml` (no `provider` extra-var) and
`create-vm.yml -e provider=docker` on a host where all three providers are
available. Verify both commands behave exactly as they did before this
feature, touching only their respective hostname pools and autogenerated
inventory files.

**Acceptance Scenarios**:

1. **Given** `provider` is omitted, **When** `create-vm.yml` runs, **Then** it
   uses the Tart hostname pool and `inventories/tart_autogenerated.yml`,
   exactly as before this feature.
2. **Given** `provider=docker`, **When** `create-vm.yml` runs, **Then** it
   uses the Docker hostname pool and `inventories/docker_autogenerated.yml`,
   exactly as before this feature.

---

### Edge Cases

- What happens when `HCLOUD_TOKEN` is unset, empty, or invalid at server
  creation time? The playbook fails with an explicit error before any hostname
  is consumed from the hcloud pool.
- What happens when `destroy-vm.yml -e provider=hcloud` is given a hostname
  that exists in `inventories/hcloud_autogenerated.yml` but the server is
  already gone (e.g., manually deleted in the Hetzner Cloud console)? The
  playbook attempts cleanup, warns about the stale entry, and removes it from
  inventory (mirrors the Tart and Docker providers' stale-entry handling).
- What happens if server creation fails after the Hetzner Cloud server has
  already been created (e.g., SSH never becomes reachable)? A rescue step
  deletes the partially-created server before the playbook fails, so no
  orphaned cloud servers are left behind and no cost accrues unnecessarily.
- What happens when Hetzner reassigns a previously used public IPv4 to a new
  server with a different host key? Because `destroy-vm.yml` removes the old
  host key from `inventories/hcloud_known_hosts` (FR-009), the next create
  against that IP is treated as first contact under `accept-new` and trusted
  cleanly, with no spurious `REMOTE HOST IDENTIFICATION HAS CHANGED` failure.
  If a stale entry does remain (e.g. a create that was never destroyed),
  `accept-new` will correctly refuse the changed key and fail loudly — the
  intended man-in-the-middle protection.
- What if two operators run `create-vm.yml -e provider=hcloud` concurrently?
  No locking is implemented; duplicate hostname assignment is possible but
  accepted as out-of-scope, matching the existing Tart and Docker providers'
  accepted risk for a single-operator project.
- What happens to the pre-existing developer-server workflow
  (`provision.yml`, `provisioners/hcloud-linux.yml`, `destroy.yml`,
  `destroy-hcloud-tasks.yml`, `inventories/hcloud.yml`, the `hobbiton`
  server)? They are unrelated to this feature, are not read, modified, or
  removed by either playbook, and remain unchanged. The hcloud dynamic
  inventory plugin file `inventories/hcloud.yml` and the static
  `inventories/hcloud_autogenerated.yml` introduced by this feature are
  distinct files serving distinct purposes.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `create-vm.yml` and `destroy-vm.yml` MUST accept a `provider`
  extra-var with allowed values `tart`, `docker`, and `hcloud`, defaulting to
  `tart` when omitted. The default value MUST preserve the exact pre-existing
  behavior of both playbooks, and the existing `docker` behavior MUST remain
  unchanged.
- **FR-002**: When `provider=hcloud`, `create-vm.yml` MUST create one new
  Hetzner Cloud server per invocation; it MUST NOT reuse or restart an
  existing server.
- **FR-003**: When `provider=hcloud`, the system MUST draw the server hostname
  from a single shared ordered list of Lord of the Rings place names defined
  in `playbooks/vars/hostname_pool_hcloud.yml` (variable name `hostname_pool`,
  loaded conditionally so it does not collide with the Tart or Docker pools of
  the same variable name). Hostnames are assigned sequentially; the first name
  not already present in `inventories/hcloud_autogenerated.yml` is selected on
  each invocation.
- **FR-004**: When `provider=hcloud`, the system MUST register each newly
  created server in `inventories/hcloud_autogenerated.yml` (YAML format,
  groups `all` / `linux` / `hcloud`) before `create-vm.yml` completes, so
  configuration automation can target it. This file is distinct from, and not
  read, written, or replaced by, the existing dynamic-inventory plugin file
  `inventories/hcloud.yml`.
- **FR-005**: On create with `provider=hcloud`, the new inventory entry MUST
  include `ansible_host` (the server's public IPv4 address returned by the
  Hetzner Cloud API), `ansible_port: 22`, and `ansible_user: root` (matching
  the existing `admin_user_on_fresh_system` convention for Hetzner Cloud Linux
  servers). No password is stored; SSH access relies on the key pair
  registered with Hetzner Cloud (see FR-013). Because an hcloud server is
  reachable over SSH from the public internet, the Ansible connection MUST
  **verify** the server's SSH host key rather than disable verification:
  `ansible_ssh_common_args` MUST set `StrictHostKeyChecking=accept-new`
  against a project-scoped `UserKnownHostsFile` (`inventories/hcloud_known_hosts`),
  and MUST NOT use `StrictHostKeyChecking=no` with
  `UserKnownHostsFile=/dev/null`. This exposure-based policy — verify on
  internet-exposed cloud targets, disable on laptop-local ones (Tart, Docker) —
  is mandated by Constitution Principle XIV; rationale, threat model, and
  rejected alternatives are in ADR-003. The same host-key policy and the same
  `inventories/hcloud_known_hosts` file MUST also be used for the create-time
  SSH connection (e.g. `wait_for_connection`). Before that connection, the
  system MUST remove any stale host key for the new server's IP from
  `inventories/hcloud_known_hosts` (idempotent
  `ssh-keygen -f inventories/hcloud_known_hosts -R <ip>`), so a public IP reused
  after a failed or skipped teardown is treated as clean first contact instead
  of failing with a changed-key error. This known_hosts file — and the `.old`
  backup `ssh-keygen -R` leaves beside it — is local machine state, like
  `inventories/hcloud_autogenerated.yml`, and MUST NOT be committed.
- **FR-006**: `create-vm.yml -e provider=hcloud` MUST print the assigned
  hostname on successful completion.
- **FR-007**: When the hcloud hostname pool is exhausted, `create-vm.yml -e
  provider=hcloud` MUST fail immediately with an error message naming the
  exhausted pool file (`playbooks/vars/hostname_pool_hcloud.yml`), without
  making any Hetzner Cloud API call.
- **FR-008**: `destroy-vm.yml -e provider=hcloud` MUST accept `hostname`
  (required) identifying a server present in
  `inventories/hcloud_autogenerated.yml`.
- **FR-009**: `destroy-vm.yml -e provider=hcloud` MUST delete the Hetzner
  Cloud server by name, remove the server's SSH host key from the
  project-scoped known_hosts file written on create
  (`inventories/hcloud_known_hosts`, FR-005) via
  `ssh-keygen -f inventories/hcloud_known_hosts -R <ansible_host>` — the same
  mechanism as `destroy-hcloud-tasks.yml`'s `ssh-keygen -R` step, but targeting
  the scoped file rather than the operator's `~/.ssh/known_hosts` — and remove
  the hostname from `inventories/hcloud_autogenerated.yml` on completion. The
  known_hosts removal MUST be idempotent (guarded so a re-run does not report a
  spurious change). If the server is already absent from Hetzner Cloud, the
  deletion step MUST be skipped gracefully with a warning, and both the
  known_hosts cleanup and the inventory cleanup MUST still occur (mirroring the
  Tart and Docker providers' stale-entry behavior). Removing the host key on
  destroy prevents a stale entry from raising a false
  `REMOTE HOST IDENTIFICATION HAS CHANGED` alarm when Hetzner later reassigns
  the public IP to a different server.
- **FR-010**: `destroy-vm.yml -e provider=hcloud` MUST fail with a clear,
  actionable error if the specified hostname is not found in
  `inventories/hcloud_autogenerated.yml`, before any Hetzner Cloud action is
  taken.
- **FR-011**: `playbooks/vars/hostname_pool_hcloud.yml` ships with ten ordered
  Lord of the Rings place names: `rivendell`, `hobbiton`, `bree`, `edoras`,
  `minas-tirith`, `mordor`, `isengard`, `moria`, `lothlorien`, `helms-deep`.
  All entries are available for assignment; none are pre-reserved. The pool is
  extensible: operators may append names to this file at any time.
- **FR-012**: The system MUST authenticate to the Hetzner Cloud API using the
  `HCLOUD_TOKEN` environment variable (the existing convention used by
  `provisioners/hcloud-linux.yml`; the `hetzner.hcloud` collection is already
  declared in `requirements.yml`). The system MUST fail with an explicit error
  identifying `HCLOUD_TOKEN` if it is unset or empty, before consuming a
  hostname from the pool.
- **FR-013**: When `provider=hcloud`, the server MUST be created via the
  `hetzner.hcloud.server` module with: `image` defaulting to `ubuntu-24.04`;
  `server_type` defaulting to the existing `hcloud_default_type`
  (`inventories/group_vars/hcloud_linux/vars.yml`); `location` defaulting to
  the existing `hcloud_default_location`
  (`inventories/group_vars/hcloud/vars.yml`); and `ssh_keys` set to the
  existing vault-configured `my_ssh_key_name`
  (`inventories/group_vars/all/vars.yml`). These defaults MUST be reused via
  `include_vars` rather than duplicated. Operators may override
  `server_type`, `location`, and `image` via extra-vars.
- **FR-014**: If server creation fails after the Hetzner Cloud server has been
  created, the system MUST run a rescue step that deletes the
  partially-created server (`hetzner.hcloud.server: state: absent`) before
  failing the playbook, mirroring the Tart provider's `tart stop`/`tart
  delete` and the Docker provider's `docker stop`/`docker rm` rescue
  behavior.
- **FR-015**: This feature MUST NOT modify, read, or remove `provision.yml`,
  `provisioners/hcloud-linux.yml`, `destroy.yml`,
  `destroy-hcloud-tasks.yml`, or `inventories/hcloud.yml`. These belong to the
  pre-existing developer-server provisioning workflow and are out of scope,
  except for read-only reuse of default values from
  `inventories/group_vars/hcloud/vars.yml`,
  `inventories/group_vars/hcloud_linux/vars.yml`, and
  `inventories/group_vars/all/vars.yml` as described in FR-013.
- **FR-016**: This feature MUST NOT change the Tart or Docker providers'
  behavior, files, or output when `provider` is omitted, `tart`, or `docker`.

### Key Entities

- **Hcloud Hostname Pool**: An ordered list of Lord of the Rings place names
  defined in `playbooks/vars/hostname_pool_hcloud.yml` (variable
  `hostname_pool`). Ships with ten names; all entries are available for
  assignment. Entries are either available or in use and are allocated
  sequentially — the first name not already in
  `inventories/hcloud_autogenerated.yml` is chosen next. This pool is
  independent from, and disjoint with, the Tart and Docker hostname pools.
- **Hcloud Inventory Entry**: A record of an hcloud-provider server's
  hostname, connection details (host, port, user, SSH common-args), and group
  membership, persisted in `inventories/hcloud_autogenerated.yml` so
  automation can target the server.
- **Hetzner Cloud Server**: The disposable cloud server resource created via
  the `hetzner.hcloud.server` module, using the existing default server type,
  location, image, and registered SSH key.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An engineer with `HCLOUD_TOKEN` configured creates a disposable
  Hetzner Cloud server with a single command
  (`create-vm.yml -e provider=hcloud`); no other extra-vars are required for
  the common case.
- **SC-002**: An engineer destroys any hcloud-provider server with a single
  command specifying only the provider and hostname.
- **SC-003**: Hcloud hostname pool exhaustion and missing `HCLOUD_TOKEN` are
  both detected and reported before any Hetzner Cloud API call is made.
- **SC-004**: Concurrent server creation is not a supported use case; no
  locking mechanism is implemented. Simultaneous invocations may assign the
  same hostname. This risk is accepted for a single-operator project (YAGNI),
  matching the existing Tart and Docker providers' accepted risk.
- **SC-005**: An engineer who omits `provider`, or passes `provider=docker`,
  continues to get identical behavior, output, and file changes as before
  this feature shipped.
- **SC-006**: The hcloud provider verifies the server's SSH host key
  (`StrictHostKeyChecking=accept-new` against a project-scoped known_hosts
  file); a *changed* host key on a known IP causes the connection to fail
  loudly rather than silently re-trusting. Destroy removes the host key so a
  reassigned public IP does not raise a false-positive identification-changed
  alarm.

## Assumptions

- The hcloud hostname pool ships with ten Lord of the Rings place names
  (`rivendell`, `hobbiton`, `bree`, `edoras`, `minas-tirith`, `mordor`,
  `isengard`, `moria`, `lothlorien`, `helms-deep`), all available for
  assignment. There are no pre-existing hcloud-provider servers to reserve;
  the pool is a clean slate.
- The operator already has `HCLOUD_TOKEN` exported and the SSH private key
  matching `my_ssh_key_name` (registered with Hetzner Cloud) available to
  their local SSH agent, as required by the existing
  `provisioners/hcloud-linux.yml` workflow. No new credential mechanism is
  introduced.
- The existing developer-server resources (`hobbiton`, managed by
  `provision.yml` / `destroy.yml`) are unrelated to this feature, are not
  read, generated, or modified by it, and remain unchanged.
- If a server exists in Hetzner Cloud but is absent from
  `inventories/hcloud_autogenerated.yml`, `destroy-vm.yml -e provider=hcloud`
  treats it as unknown and fails; manual cleanup of orphaned servers is out of
  scope, matching the Tart and Docker providers' equivalent assumption.
- Default server type (`hcloud_default_type`), location
  (`hcloud_default_location`), and image (`ubuntu-24.04`) are reused from
  existing configuration rather than introducing hcloud-specific resource
  defaults, since Hetzner Cloud server types are predefined tiers rather than
  freely composable CPU/RAM/disk values.
