# Phase 0 Research: Configure Basic Profile for Tart VMs

All Technical Context fields in [plan.md](plan.md) were resolved directly from
existing repository precedent — no `NEEDS CLARIFICATION` markers remain. This
document records the decisions and the precedent each one follows.

## Decision: Orchestrator playbook structure

**Decision**: `configure-profile.yml` is a flat `import_playbook` chain
at the repository root:

```yaml
---
- import_playbook: playbooks/setup-users.yml
- import_playbook: playbooks/setup-basics.yml
- import_playbook: playbooks/setup-nodejs.yml
- import_playbook: configure-profile-roles.yml
- import_playbook: playbooks/reboot-if-required.yml
```

**Rationale**: `configure-linux.yml` (existing, repo root) already establishes
this exact pattern — a sequence of `import_playbook` statements with no
inline tasks. `configure-profile.yml` reuses the same building blocks
minus the desktop-only steps (`setup-homebrew.yml`, `setup-desktop.yml`,
`setup-keyring.yml`, `setup-desktop-apps.yml`, `restore.yml`) that FR-018
explicitly excludes for a tart VM with no desktop environment.

**Alternatives considered**:

- *Inline tasks in a single playbook*: rejected — violates Principle II
  (playbooks MUST only orchestrate roles/playbooks, no implementation logic).
- *New shared "basic" sub-playbook included by both `configure-linux.yml` and
  `configure-profile.yml`*: rejected as premature abstraction
  (Principle IV, YAGNI) — the four reused playbooks (`setup-users.yml`,
  `setup-basics.yml`, `setup-nodejs.yml`, `reboot-if-required.yml`) are
  already independently reusable `import_playbook` targets; no new
  abstraction is needed to compose them differently.

## Decision: Roles-application playbook scoped to `tart` group

**Decision**: `configure-profile-roles.yml` mirrors
`configure-linux-roles.yml`'s structure (`become: true`, `ansible_user:
{{ my_ansible_user }}`, `desktop_user_names` derived the same way) but targets
`hosts: tart` and lists only the five roles required by FR-013–FR-017:
`podman`, `ruby`, `python`, `dolt_sql_server`, `claude_code`.

**Rationale**: `configure-linux-roles.yml` targets `hosts: linux` and includes
`tmux` and `google_chrome`, which FR-018 explicitly excludes for this profile
("MUST NOT install tmux or any desktop-only tooling"). A separate playbook
with a reduced role list and a narrower host scope (`tart` instead of `linux`)
avoids either (a) modifying the existing `configure-linux-roles.yml` (which
would change behaviour for the existing `configure-linux.yml` consumer,
violating Principle XI's "single source of truth" in the other direction —
two unrelated profiles must not share one mutable role list) or (b) adding
conditional/tag-based role skipping logic to a shared file (rejected as
unnecessary complexity, Principle IV).

**Alternatives considered**:

- *Reuse `configure-linux-roles.yml` with tags to skip `tmux`/`google_chrome`*:
  rejected — `configure-linux-roles.yml` already documents a
  `not-supported-on-vagrant-arm64` tag convention for a different purpose
  (skipping roles unsupported on Vagrant ARM64, not skipping desktop-only
  roles for a basic profile); overloading or extending the tag scheme adds
  complexity disproportionate to a 5-line role list duplication.
- *Parametrize the role list via a variable*: rejected as speculative
  generalisation (Principle IV) — only two concrete role-list shapes exist
  today (`configure-linux-roles.yml`'s 7 roles vs. this feature's 5-role
  subset), and both are static lists known at write time.

## Decision: `tart` group_vars for the bootstrap account

**Decision**: New file `inventories/group_vars/tart/vars.yml`:

```yaml
---
# Tart (Linux) specific overrides
admin_user_on_fresh_system: "admin"
```

**Rationale**: `inventories/group_vars/vagrant_tart/vars.yml` already sets
`admin_user_on_fresh_system: "admin"` for the analogous `vagrant_tart` group,
with the same comment style ("Vagrant Tart (Linux) specific overrides").
`playbooks/setup-users.yml` already consumes
`{{ admin_user_on_fresh_system }}` as `ansible_user` for its initial
connection — this is the exact mechanism FR-002 requires, already wired and
already used by four other groups (`windows`, `hcloud_linux`,
`vagrant_docker`, `aws_ec2_linux`). `inventories/tart_autogenerated.yml`
(written by `create-vm.yml`) registers `ansible_user: admin` /
`ansible_ssh_pass: admin` per host directly, but `setup-users.yml` reads
`admin_user_on_fresh_system` from group_vars rather than the per-host
`ansible_user`, so the new group_vars file is required for `setup-users.yml`
to resolve the variable for hosts in the `tart` group.

**Alternatives considered**:

- *Set `admin_user_on_fresh_system` in `inventories/group_vars/all/vars.yml`*:
  rejected — `all/vars.yml` has no existing `admin_user_on_fresh_system`
  default (each platform group defines its own), and a global default would
  be incorrect for platforms whose bootstrap account differs (e.g.
  `windows`, `aws_ec2_linux` use different values per their own group_vars).
- *Hardcode `"admin"` inside `setup-users.yml`*: rejected — `setup-users.yml`
  is a shared, already-validated playbook used by every platform; hardcoding
  a tart-specific value would violate Principle XI (DRY) and break other
  groups.

## Decision: No new Galaxy collections (FR-021)

**Decision**: `requirements.yml` is left unmodified. The `ansible.posix`
collection added by an earlier PoC iteration is not reintroduced.

**Rationale**: A repo-wide check (`grep -r "ansible.posix"`) found zero usages
of any `ansible.posix` module across all roles and playbooks — it was added
speculatively and never consumed. All five development tool roles
(`podman`, `ruby`, `python`, `dolt_sql_server`, `claude_code`) and all four
reused playbooks already function with the collections currently listed in
`requirements.yml` (`community.general`, `amazon.aws`, `hetzner.hcloud`,
`ansible.windows`, `community.windows`, `containers.podman`). FR-021 and
Principle IV (YAGNI) both require avoiding unused dependencies.

**Alternatives considered**:

- *Keep `ansible.posix` "for future use"*: rejected — Principle IV
  prohibits speculative additions; an unused collection is dead weight in
  `requirements.yml` with no corresponding requirement.

## Decision: End-to-end validation strategy (Principle III)

**Decision**: `configure-profile.yml` itself is validated by running it
against a real tart VM created via `create-vm.yml` (the two-step operator
workflow documented in [quickstart.md](quickstart.md)). No new Molecule
scenario is created for the orchestrator playbooks.

**Rationale**: The five composed roles are each independently covered by a
passing `molecule/default/` scenario (Principle II is already satisfied for
each role). `configure-profile.yml` and
`configure-profile-roles.yml` contain zero implementation logic of
their own — there is nothing role-shaped to test in a container. This mirrors
how `create-vm.yml`/`destroy-vm.yml` (an orchestration-only feature, see
`specs/009-create-destroy-vm/plan.md`) were validated: end-to-end on the real
target, not via Molecule.

**Alternatives considered**:

- *New Molecule scenario for `configure-profile-roles.yml`*:
  rejected — Molecule's container driver cannot exercise the full
  `setup-users.yml` SSH-key/sudo flow realistically (no SSH daemon
  reconfiguration semantics needed there), and the role-level scenarios
  already cover the role content that `configure-profile-roles.yml`
  merely lists.
