# Contract: configure-profile.yml

## Invocation

```bash
ansible-playbook playbooks/configure-profile.yml
```

No extra-vars are required for the common case (FR-001, SC-001).

`configure-profile.yml` runs against the `tart` inventory group. The
play in `configure-profile-roles.yml` explicitly targets
`hosts: tart`; the four reused playbooks
(`playbooks/setup-users.yml`, `playbooks/setup-basics.yml`,
`playbooks/setup-nodejs.yml`, `playbooks/reboot-if-required.yml`) target
broader host groups (`all` / `linux`) but, when run via this orchestrator
against an inventory whose only relevant hosts are in `tart` (and its
superset groups `linux`/`all`), apply only to the tart VM(s).

## Preconditions

- The target VM was created by `playbooks/create-vm.yml` and is present in
  `inventories/tart_autogenerated.yml` under the `all`, `linux`, and `tart`
  groups, in its post-`create-vm.yml` state (default `admin` account only,
  `my_ansible_user` not yet created).
- The control node can reach the VM via the `admin_user_on_fresh_system`
  account (`"admin"` for the `tart` group, defined in
  `inventories/group_vars/tart/vars.yml`) using SSH key authentication: at
  create time `create-vm.yml` installs the operator's `my_ssh_public_key` into
  the VM's `admin` account, and the `inventories/tart_autogenerated.yml` entry
  has no `ansible_ssh_pass` (ADR-004).
- `my_ansible_user_password` and `my_ssh_public_key` are configured (vault),
  per the existing fail-loud defaults in `playbooks/setup-users.yml`.

## Postconditions (on success)

Per FR-003 through FR-019:

- `my_ansible_user` and every entry in `desktop_users` exist on the VM as
  members of the `sudo` group (FR-003).
- `my_ansible_user` has passwordless sudo (FR-004).
- Every configured user's SSH public key is installed for passwordless login
  (FR-005); SSH password authentication is disabled (FR-006).
- Every configured user's `.bash_profile` loads `.bashrc` on login (FR-007).
- The apt package cache is refreshed and all packages are upgraded (FR-008);
  system timezone is `Europe/Berlin` (FR-009).
- Each `desktop_users` entry has NVM installed, the current Node.js LTS
  release installed and set as default, and the global npm tools `eslint`,
  `markdownlint-cli`, `prettier`, `typescript` installed (FR-010–FR-012).
- `podman`, `ruby`, `python`, `dolt_sql_server`, and `claude_code` are
  installed and configured on the VM (FR-013–FR-017).
- Neither `tmux` nor any desktop-only tooling (e.g. a browser) is installed
  (FR-018).
- If the OS reports a pending reboot (`/var/run/reboot-required` exists), the
  VM reboots and the playbook waits for it to come back online before
  completing; otherwise the playbook completes without rebooting (FR-019).

## Failure modes

| Condition | Behaviour |
|-----------|-----------|
| `admin_user_on_fresh_system` account unreachable (e.g. VM not yet created, or already past bootstrap with the account removed) | `playbooks/setup-users.yml`'s initial connection fails explicitly; no further plays run |
| `my_ansible_user_password` or `my_ssh_public_key` not configured | `setup-users.yml` fails loudly with the existing actionable default-value error message (no silent empty-password/empty-key fallback) |
| apt upgrade fails (e.g. network unavailable) | `playbooks/setup-basics.yml`'s `apt: upgrade=safe` task fails explicitly; subsequent plays do not run |
| Any of the five role plays fails (podman, ruby, python, dolt_sql_server, claude_code) | The play fails per that role's existing error handling; `playbooks/reboot-if-required.yml` does not run |
| Reboot required but VM does not come back online within the `reboot` module's default timeout | `playbooks/reboot-if-required.yml`'s `reboot` task fails explicitly |

No new failure paths are introduced by this feature; all failure behaviour is
inherited unchanged from the reused playbooks and roles.

## Idempotency

Fully idempotent (FR-020, SC-006). A second run of
`ansible-playbook playbooks/configure-profile.yml` against an already-configured
VM, with no extra-vars, reports zero changed tasks across every imported
playbook and every applied role — each of the five development tool roles
already passes its `molecule/default/` idempotence check, and the four reused
playbooks (`setup-users.yml`, `setup-basics.yml`, `setup-nodejs.yml`,
`reboot-if-required.yml`) are themselves already idempotent (e.g. the apt
cache refresh uses `cache_valid_time`, the reboot check is gated on
`/var/run/reboot-required`, and NVM/Node/npm installs use `creates:` guards).
