# Phase 1 Data Model: AWS Windows Profile for VM Lifecycle

This feature has no database or application data model — it extends the
existing flat-YAML inventory entities used by `create-vm.yml`/
`destroy-vm.yml`/`configure-profile.yml`. The entities below are
inventory/configuration concepts, not application records.

## Profile (extended)

The existing `profile` extra-var on `create-vm.yml`, introduced by
`014-desktop-profile`.

| Field | Type | Values | Notes |
|---|---|---|---|
| `profile` | string | `basic` (default), `desktop`, `windows` (new) | `windows` selects the `windows` inventory group, the AWS Windows AMI/instance-type defaults, and the `setup-roles-windows.yml` role set. Valid only with `provider=aws`. |

**Validation rules**:

- MUST be one of `basic`, `desktop`, `windows` (FR-001).
- `profile=windows` combined with any `provider` other than `aws` MUST be
  rejected before any infrastructure action (FR-007).

**Relationships**: Orthogonal to `provider` except for the one disallowed
combination above (mirrors the existing `provider=docker profile=desktop`
disallowed combination).

## Windows Inventory Group (new)

A new dict key (`windows`) written into
`inventories/aws_autogenerated.yml`, alongside the existing `all`/`aws`
keys. Unlike `basic`/`desktop`/`linux` hosts, a `profile=windows` host is
**not** added to the `linux` group.

| Field | Type | Notes |
|---|---|---|
| group name | string | `windows`, selected when `profile == 'windows'` at create time. |
| `hosts` | dict (hostname → host vars) | Host vars include `ansible_user: Administrator` (not `ubuntu`), in addition to `ansible_host`, `ansible_port: 22`, `ansible_ssh_common_args`, and `aws_instance_id` — the same shape already used for Linux AWS hosts. |

**State transitions**:

- **Create**: host added under `all`, `aws`, and `windows` (never `linux`),
  matching `profile == 'windows'`.
- **Destroy**: host removed from `all`, `aws`, and `windows` symmetrically
  (mirrors the existing `basic`/`desktop` cleanup in
  `tasks/destroy/aws.yml`).
- A host's profile group membership is fixed at create time, unchanged by
  this feature's design (same as `basic`/`desktop`).

## AWS Windows Security Rule (widened, not new)

The existing TCP/3389 (RDP) conditional rule on the shared `ansible-sg`
security group, introduced by `014-desktop-profile` for `profile=desktop`.

| Field | Type | Notes |
|---|---|---|
| `proto` | string | `tcp` (unchanged) |
| `ports` | list[int] | `[3389]` (unchanged) |
| `cidr_ip` | string | `0.0.0.0/0` (unchanged, matching the existing 22 rule's exposure) |
| condition | — | Widened from `profile == 'desktop'` to `profile in ['desktop', 'windows']` (FR-006). Never applied for `profile == 'basic'`. |

**Relationships**: Same `ansible-sg` group already used by every AWS VM
regardless of profile — no new security group entity, no second 3389 rule
object.

## Windows Instance User-Data (new)

The PowerShell `user_data` script passed to `amazon.aws.ec2_instance` when
creating a `profile=windows` instance, ported from
`provisioners/aws-windows.yml`.

| Field | Type | Source | Notes |
|---|---|---|---|
| Administrator password | string | `windows_admin_password` (existing Vault-backed var) | MUST be defined and non-empty before any AWS API call (fail-loud edge case in spec.md). |
| Authorized SSH public key | string | `my_ssh_public_key` (existing shared var, same one used for Linux AWS hosts) | Written to `C:\ProgramData\ssh\administrators_authorized_keys` with `icacls`-restricted permissions. |

**Relationships**: Both values are existing, already-defined variables
(`inventories/group_vars/all/vars.yml`); this feature introduces no new
secret or shared variable.
