# Contract: `create-vm.yml -e provider=aws -e profile=windows`

## Invocation

```bash
ansible-playbook playbooks/create-vm.yml -e provider=aws -e profile=windows
```

## Preconditions (checked in this order, before any infrastructure action)

1. `provider` is one of `tart`, `docker`, `hcloud`, `aws` (existing
   `tasks/assert-provider-profile.yml`, unchanged).
2. `profile` is one of `basic`, `desktop`, `windows` (FR-001; extended
   assertion).
3. The combination `profile=windows` with any `provider` other than `aws` is
   rejected with an explicit error naming both values (FR-007; new
   assertion, mirrors the existing `provider=docker profile=desktop`
   rejection).
4. `windows_admin_password` is defined and non-empty (new assertion,
   fail-loud edge case in spec.md).
5. The AWS hostname-pool exhaustion check (existing, unchanged — the AWS
   pool is shared across all `provider=aws` profiles).

## Inputs

| Variable | Required | Default | Source |
|---|---|---|---|
| `provider` | no | `tart` | extra-var (unchanged) |
| `profile` | no | `basic` | extra-var (`windows` is new) |
| `windows_admin_password` | yes, when `profile=windows` | none | Vault-backed group_var (`inventories/group_vars/all/vars.yml`), already exists |

## Effects on success

1. Exactly one EC2 instance is created, running the Windows Server
   image/instance-type defaulted from
   `inventories/group_vars/aws_ec2_windows/vars.yml` (overridable via
   `aws_ami_id`/`aws_instance_type` extra-vars, same mechanism as the Linux
   branch).
2. The instance's `user_data` sets the Administrator password from
   `windows_admin_password` and authorizes the shared SSH public key
   (`my_ssh_public_key`) for SSH login.
3. The new host is registered in `inventories/aws_autogenerated.yml` under
   `all`, `aws`, and `windows` (new) — **not** under `linux` (FR-003/FR-004).
   Its `ansible_user` is `Administrator` (not `ubuntu`).
4. The shared `ansible-sg` security group allows inbound TCP 3389, in
   addition to the existing TCP 22 rule (FR-006).
5. The assigned hostname is printed (unchanged mechanism, shared AWS pool).

## Effects on failure

- `profile` not in `{basic, desktop, windows}`: fails immediately, no
  infrastructure action, no hostname consumed.
- `profile=windows` with `provider != aws`: fails immediately, no VM created,
  no hostname consumed, error names both `provider` and `profile`.
- `windows_admin_password` undefined or empty: fails immediately, before any
  AWS API call, error names the missing variable.
- All other failure modes (pool exhaustion, provider-specific rollback):
  unchanged from the existing `provider=aws` behavior.

## Out of scope / unaffected

- `profile=basic`/`profile=desktop` on any provider: unchanged (FR-012).
- `destroy-vm.yml` takes no `profile` extra-var — destroy removes a host from
  whichever profile group it was registered under at create time (see
  `destroy-vm-profile-windows.md`).
