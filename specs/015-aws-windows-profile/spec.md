# Feature Specification: AWS Windows Profile for Create, Destroy, and Configure VM Playbooks

**Feature Branch**: `015-aws-windows-profile`

**Created**: 2026-06-22

**Status**: Draft

**Input**: User description: "Implement create-vm.yml, destroy-vm.yml and
configure-profile.yml for the new provider/profile combination aws/windows.
This has been implemented in the legacy version already - you can copy the
relevant parts, but make sure to adapt to the new design!"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Create an AWS Windows Server VM as a "VM" (Priority: P1)

An engineer wants a disposable Windows Server machine — for example to run a
Windows-only desktop application — provisioned through the same unified
command already used for every other VM. They run
`create-vm.yml -e provider=aws -e profile=windows`. The system creates an EC2
Windows Server instance, makes it reachable over SSH (PowerShell shell) using
the same SSH key already used for Linux AWS VMs, opens RDP (3389) for
interactive desktop access, registers the host in a `windows` inventory group
distinct from `linux`/`basic`/`desktop`, and reports the assigned hostname on
completion.

**Why this priority**: This is the core value of the feature: it brings the
already-proven legacy Windows provisioning capability (AWS Windows Server with
SSH + RDP access) into the unified `create-vm.yml`/`destroy-vm.yml`/
`configure-profile.yml` command shape, so engineers no longer need the
separate legacy `provisioners/aws-windows.yml` entrypoint for this one
platform.

**Independent Test**: Run `create-vm.yml -e provider=aws -e profile=windows`
with AWS credentials configured. The command completes, prints a hostname,
and the new host is reachable via SSH (PowerShell) and RDP using the
registered inventory entry.

**Acceptance Scenarios**:

1. **Given** AWS credentials are configured, **When**
   `create-vm.yml -e provider=aws -e profile=windows` runs, **Then** a
   Windows Server EC2 instance is created, the next unused hostname from the
   existing AWS hostname pool is assigned, the instance is registered in
   `inventories/aws_autogenerated.yml` under a `windows` group (not `linux`),
   and the hostname is printed on completion.
2. **Given** the new instance has been created, **When** an engineer connects
   via SSH using the registered inventory entry, **Then** the connection
   succeeds with a PowerShell shell, authenticated by the same SSH key already
   used for Linux AWS VMs.
3. **Given** `provider=aws` and `profile=windows`, **When** `create-vm.yml`
   runs, **Then** the security group used by the new VM allows inbound TCP
   3389 (RDP) in addition to TCP 22 (SSH).

### User Story 2 - Destroy an aws/windows VM by hostname (Priority: P2)

An engineer is done with a Windows VM and wants to stop paying for it. They
run `destroy-vm.yml -e provider=aws -e hostname=<name>` using the hostname
that `create-vm.yml` printed. The system terminates the EC2 instance, removes
its host key, and removes it from the `windows` inventory group — using the
exact same command shape already used to destroy Linux AWS VMs.

**Why this priority**: Without reliable destroy support, every Windows VM
created through this feature becomes an ongoing, easy-to-forget cost. This is
the same lifecycle-completeness priority that drove `destroy-vm.yml` support
for the `aws` provider in the first place.

**Independent Test**: Create an aws/windows VM, note its hostname, then run
`destroy-vm.yml -e provider=aws -e hostname=<name>`. Verify the EC2 instance
is terminated and the hostname no longer appears anywhere in
`inventories/aws_autogenerated.yml`.

**Acceptance Scenarios**:

1. **Given** an aws/windows VM registered under hostname `<name>`, **When**
   `destroy-vm.yml -e provider=aws -e hostname=<name>` runs, **Then** the EC2
   instance is terminated, its known_hosts entry is removed, and `<name>` is
   removed from every group (`all`, `aws`, `windows`) in
   `inventories/aws_autogenerated.yml`.
2. **Given** one aws/linux VM and one aws/windows VM coexist in
   `inventories/aws_autogenerated.yml`, **When** the windows VM is destroyed,
   **Then** the linux VM's inventory entry and groups are left untouched.

### User Story 3 - Configure an aws/windows VM (Priority: P3)

An engineer has created an aws/windows VM and wants it ready to use, the same
way `configure-profile.yml` already makes `basic` and `desktop` VMs ready to
use. They run `configure-profile.yml` against the new host. The system
applies the Windows-specific role set already proven in the legacy stack
(RDP tuning, Chocolatey, and an AI-agent toolchain) instead of the Linux-only
plays that `basic`/`desktop` hosts receive, since those plays assume a Debian
package manager and a Bash shell that do not exist on Windows.

**Why this priority**: Without this story, a freshly created aws/windows VM
is reachable but bare — and running the existing Linux-only setup plays
against it would fail outright (they assume `apt`, `/bin/bash`, and Linux user
management). This story is lower priority than create/destroy because a bare,
reachable Windows box still has standalone value (RDP access), matching the
legacy MVP's own staged rollout.

**Independent Test**: Run `configure-profile.yml` against a host in the
`windows` inventory group. Verify the host receives the Windows role set
(RDP tuning, Chocolatey, AI-agent toolchain) and that none of the Linux-only
plays (user/account setup, apt package setup, Node.js setup) are applied to
it.

**Acceptance Scenarios**:

1. **Given** a host in the `windows` inventory group, **When**
   `configure-profile.yml` runs, **Then** the host receives the same role set
   already applied by the legacy windows configuration path, unmodified.
2. **Given** a host in the `windows` inventory group, **When**
   `configure-profile.yml` runs, **Then** none of the Linux-only plays that
   `basic`/`desktop` hosts receive are applied to it.
3. **Given** hosts in the `basic` and `desktop` inventory groups, **When**
   `configure-profile.yml` runs, **Then** their behavior is unchanged from
   today.

---

### User Story 4 - Reject profile=windows on non-AWS providers (Priority: P4)

An engineer mistakenly requests a Windows VM on a provider that does not
support it (`tart`, `docker`, or `hcloud` — Windows is documented as
AWS-only). Instead of starting a provisioning step that cannot succeed, the
command refuses immediately with an explicit, actionable error, before any
infrastructure action occurs.

**Why this priority**: Lowest priority because it is a safety guard, not new
capability — it prevents a confusing failure rather than enabling a new
outcome. It is cheap to add and mirrors the existing
`provider=docker profile=desktop` rejection already in place.

**Independent Test**: Run `create-vm.yml -e provider=tart -e profile=windows`
(or `docker`/`hcloud`). Verify the command fails immediately, before any VM
or container is created, with a message naming both the rejected provider and
`profile=windows`.

**Acceptance Scenarios**:

1. **Given** `profile=windows` is requested with any provider other than
   `aws`, **When** `create-vm.yml` runs, **Then** it fails immediately with an
   explicit error naming both values, before any VM is created.
2. **Given** `provider=aws` with `profile=windows`, **When** `create-vm.yml`
   runs, **Then** it proceeds normally (no rejection).

### Edge Cases

- What happens to the legacy entrypoints this feature is ported from
  (`provisioners/aws-windows.yml`, `configure-windows.yml`,
  `setup-roles-windows.yml`, `destroy-aws-tasks.yml`)? Unchanged, not
  removed — they remain the fallback path until a later phase proves full
  parity and retires the legacy stack (same precedent as the `desktop`
  profile feature).
- What happens to the `windows_foundation` and `win_ai_agent` roles
  themselves? Reused verbatim, unmodified — this feature wires existing,
  already-working roles into the new playbooks; it does not change role
  internals.
- What happens to the existing `profile=desktop` AWS RDP (3389) rule? It must
  keep firing for `profile=desktop` exactly as before; `profile=windows` adds
  a second, independent condition that opens the same port, never narrowing
  or removing the existing one.
- What happens to the AWS hostname pool? Shared, unchanged. `profile=windows`
  draws from the same `provider=aws` hostname pool already used by
  `profile=basic`/`desktop` AWS VMs — no separate Windows-only pool is
  introduced.
- What happens if `windows_admin_password` (the Windows Administrator
  password, already a documented Vault secret) is missing? `create-vm.yml`
  must fail loudly before any AWS API call, naming the missing variable —
  the same fail-loud guard the legacy provisioner already performs.
- What happens to SSH key authentication on the Windows host? It reuses the
  same shared SSH key (`my_ssh_public_key`/`my_ssh_key_name`) already used for
  Linux AWS VMs, injected via the instance's startup script — Windows AMIs do
  not support the cloud-init-style key injection used for the Linux AMI.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: `create-vm.yml` MUST accept `profile=windows` as a new value of
  the existing `profile` extra-var, valid only in combination with
  `provider=aws`.
- **FR-002**: When `provider=aws` and `profile=windows`, `create-vm.yml` MUST
  create an EC2 instance running the Windows Server image and instance type
  already configured for AWS Windows hosts (`inventories/group_vars/aws_ec2_windows/vars.yml`).
- **FR-003**: When `provider=aws` and `profile=windows`, `create-vm.yml` MUST
  register the new host in a `windows` inventory group inside
  `inventories/aws_autogenerated.yml`, distinct from the `linux`, `basic`, and
  `desktop` groups, with an `ansible_user` of `Administrator` and the
  PowerShell shell settings already declared in
  `inventories/group_vars/windows/vars.yml`.
- **FR-004**: When `provider=aws` and `profile=windows`, `create-vm.yml` MUST
  NOT register the new host in the `linux` inventory group.
- **FR-005**: When `provider=aws` and `profile=windows`, `create-vm.yml` MUST
  configure the new instance so it accepts SSH key authentication using the
  same shared SSH key already used for Linux AWS VMs, and sets the
  Administrator password from the existing `windows_admin_password` Vault
  variable — reproducing the approach already proven in
  `provisioners/aws-windows.yml`, adapted to the new task-file structure
  (`playbooks/tasks/create/aws.yml`).
- **FR-006**: When `provider=aws` and `profile=windows`, the AWS security
  group used by the new VM MUST allow inbound TCP 3389 (RDP), in addition to
  the existing TCP 22 (SSH) rule, using the same condition mechanism already
  used for `profile=desktop` (FR-007 of the `014-desktop-profile` feature) —
  both `profile=desktop` and `profile=windows` open 3389; `profile=basic`
  opens neither.
- **FR-007**: `create-vm.yml` MUST reject any combination of `profile=windows`
  with a provider other than `aws` immediately, with an explicit error naming
  both the rejected provider and `profile=windows`, before any VM is created.
- **FR-008**: `destroy-vm.yml` MUST destroy an aws/windows VM by hostname using
  the same command shape already used for aws/linux VMs
  (`destroy-vm.yml -e provider=aws -e hostname=<name>`), removing the EC2
  instance, its known_hosts entry, and its entry from every inventory group
  it belongs to (`all`, `aws`, `windows`) in
  `inventories/aws_autogenerated.yml`.
- **FR-009**: `destroy-vm.yml` MUST NOT alter inventory entries or groups
  belonging to any other host.
- **FR-010**: `configure-profile.yml` MUST gain a branch, scoped to the
  `windows` inventory group, that applies the existing `windows_foundation`
  and `win_ai_agent` roles (already used by the legacy
  `setup-roles-windows.yml`), verbatim and unmodified.
- **FR-011**: `configure-profile.yml` MUST NOT apply any of its existing
  Linux-only plays (`setup-users.yml`, `setup-basics.yml`,
  `setup-nodejs.yml`, or the `configure-profile-roles.yml` Linux role lists)
  to hosts in the `windows` inventory group.
- **FR-012**: This feature MUST NOT modify the behavior, output, or files
  touched by `create-vm.yml`/`destroy-vm.yml`/`configure-profile.yml` when
  `profile=basic` or `profile=desktop` is requested.
- **FR-013**: This feature MUST NOT modify the internals of the
  `windows_foundation` or `win_ai_agent` roles, nor the legacy entrypoints it
  is ported from (`provisioners/aws-windows.yml`, `configure-windows.yml`,
  `setup-roles-windows.yml`, `destroy-aws-tasks.yml`) — those remain in place,
  unchanged, as the fallback path.

### Key Entities

- **Windows Profile**: A new value (`windows`) of the existing `profile`
  extra-var, valid only with `provider=aws`, mapped to the `windows` inventory
  group and the `windows_foundation`/`win_ai_agent` role set.
- **Windows Inventory Group**: A new inventory group holding VMs created with
  `profile=windows`, distinct from `linux`, `basic`, and `desktop`, configured
  by the new `windows` branch of `configure-profile.yml`.
- **AWS Windows Security Rule**: The TCP 3389 (RDP) ingress rule, now opened
  for both `profile=desktop` and `profile=windows` AWS VMs, additive to the
  existing TCP 22 rule on the shared `ansible-sg` security group.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: An engineer provisions a working, reachable (SSH + RDP) AWS
  Windows VM with the same single-command shape already used for Linux AWS
  VMs (`create-vm.yml -e provider=aws -e profile=windows`).
- **SC-002**: An engineer destroys that VM with the same single-command shape
  already used for Linux AWS VMs (`destroy-vm.yml -e provider=aws -e hostname=<name>`),
  leaving no orphaned EC2 instance, known_hosts entry, or inventory entry.
- **SC-003**: An engineer configures that VM with the unmodified
  `configure-profile.yml` command and ends up with the same Windows role set
  (RDP tuning, Chocolatey, AI-agent toolchain) already proven by the legacy
  stack.
- **SC-004**: 100% of `profile=windows` attempts on a non-`aws` provider fail
  immediately with an explicit, actionable error, before any VM is created.
- **SC-005**: Zero `basic`-profile or `desktop`-profile VMs (on any provider)
  change behavior as a result of this feature shipping.

## Assumptions

- The `windows` and `aws_ec2_windows` group_vars files
  (`inventories/group_vars/windows/vars.yml`,
  `inventories/group_vars/aws_ec2_windows/vars.yml`), the
  `windows_admin_password`/`vault_windows_admin_password` secret, and the
  `windows_foundation`/`win_ai_agent` roles already exist in the repository
  and are reused as-is; this feature wires them into the new playbooks rather
  than creating them from scratch.
- "Windows is only supported by AWS" (FR-007) is a pre-existing, documented
  constraint (`docs/user-manual/create-vm.md`), not a new restriction
  introduced by this feature.
- The AWS hostname pool (`playbooks/vars/hostname_pool_aws.yml`) is shared
  across `profile` values for `provider=aws`; this feature does not introduce
  a Windows-specific pool.
- The shared `ansible-sg` AWS security group (already used by `basic` and
  `desktop` AWS VMs) is reused as-is; this feature does not introduce a
  separate security group for `profile=windows`.
- Real role extraction or modification of `windows_foundation`/`win_ai_agent`
  is out of scope; both roles are already implemented and proven by the
  legacy stack.
- Retiring the legacy entrypoints
  (`provisioners/aws-windows.yml`/`configure-windows.yml`/
  `setup-roles-windows.yml`/`destroy-aws-tasks.yml`) is out of scope and
  deferred to the same later legacy-retirement phase already deferred by the
  `014-desktop-profile` feature.
