# Implementation Plan: AWS Windows Profile for Create, Destroy, and Configure VM Playbooks

**Branch**: `015-aws-windows-profile` | **Date**: 2026-06-22 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/015-aws-windows-profile/spec.md`

## Summary

Introduce `profile=windows` as a new value of the existing `profile`
extra-var on `create-vm.yml`, valid only with `provider=aws`.
`playbooks/tasks/create/aws.yml` gains a Windows branch that launches a
Windows Server EC2 instance (image/instance-type defaults already declared
in `inventories/group_vars/aws_ec2_windows/vars.yml`), injects the shared SSH
key and the existing `windows_admin_password` Vault secret via instance
user-data (porting the approach already proven in
`provisioners/aws-windows.yml`), and registers the host under a new
`windows` inventory group (not `linux`) with `ansible_user: Administrator`.
`playbooks/tasks/destroy/aws.yml` gains symmetric cleanup for the `windows`
group. The existing AWS security-group TCP/3389 condition (currently
`profile == 'desktop'` only, from `014-desktop-profile`) is widened to
`profile in ['desktop', 'windows']`. `configure-profile.yml` gains an
unconditional import of the existing `setup-roles-windows.yml` (already
`hosts: windows`, already applying `windows_foundation` + `win_ai_agent`
verbatim — no new role work). Its three Linux-only plays that currently say
`hosts: all` (`setup-users.yml`, `setup-basics.yml`, `playbooks/reboot-if-required.yml`)
are re-scoped to `hosts: linux` first, so they no longer attempt to run
unsupported Ansible modules against Windows hosts — a behavior-preserving
change since every current target is already a `linux`-group host. No new
roles, no role-internals changes, no Complexity Tracking exception: this
feature is wiring, not new capability.

## Technical Context

**Language/Version**: Ansible YAML (playbooks/tasks), consistent with the
existing `tart`/`docker`/`hcloud`/`aws` provider implementations and the
`basic`/`desktop` profile orchestration playbooks in this repo.

**Primary Dependencies**: None new. Reuses the existing `windows_foundation`
and `win_ai_agent` roles, the existing `windows`/`aws_ec2_windows` group_vars,
and the existing `windows_admin_password`/`vault_windows_admin_password`
Vault secret — all already present and already exercised by the legacy
`provisioners/aws-windows.yml` → `setup-roles-windows.yml` path.

**Storage**: Flat YAML only — `inventories/aws_autogenerated.yml` (generated,
gitignored) gains a `windows` group key alongside its existing `all`/`linux`/
`aws`/`basic`/`desktop` keys. No new hostname pool (the existing
`playbooks/vars/hostname_pool_aws.yml` is shared across all `provider=aws`
profiles), no new committed inventory file, no new group_vars directory (the
`windows`/`aws_ec2_windows` group_vars this feature depends on already
exist).

**Testing**: No Molecule scenario — `create-vm.yml`/`destroy-vm.yml`/
`configure-profile.yml` are orchestration playbooks targeting dynamically
created remote hosts, not a fresh container (same precedent as
`specs/013-aws-vm-provider` and `specs/014-desktop-profile`). Unlike the
`tart`/`hcloud`/`docker` providers, Windows has **no local equivalent at all**
in this project's toolchain — Vagrant+Tart only targets Linux guests
(Constitution § Technology Stack), so there is no local Windows VM to
validate against, before or after this feature. This is a pre-existing gap
inherited from the legacy `provisioners/aws-windows.yml` path (which was
likewise validated only against real AWS), not a new one introduced here.
Validation therefore requires one real `provider=aws profile=windows`
create → configure → destroy round-trip against AWS (Constitution Principle
III's "no local equivalent" carve-out, same reasoning the AWS provider plan
used for credential/API behavior that cannot be exercised locally).

**Target Platform**: Controller = `localhost`. Managed node = one AWS EC2
Windows Server instance. No other provider is affected (`profile=windows` is
rejected for `tart`/`docker`/`hcloud` per FR-007).

**Project Type**: Single Ansible project — task-file and playbook
modifications to the existing `playbooks/` tree. No new roles; the
`windows_foundation`/`win_ai_agent` roles and the `setup-roles-windows.yml`
playbook that already applies them are reused unmodified.

**Performance Goals**: N/A (one-shot provisioning/configuration commands;
existing timeout conventions in `tasks/create/aws.yml` apply unchanged, with
Windows's already-longer boot time handled the same way the legacy
provisioner already handled it — via `wait_timeout` on `ec2_instance`).

**Constraints**: Constitution Principles I (idempotency — the new `windows`
inventory-group write uses the same `combine()` (create) /
`dict2items | selectattr` (destroy) merge pattern already proven idempotent
for `basic`/`desktop`), II (Role-First Organisation — no new role logic; this
feature is wiring two already-existing roles into already-existing
playbooks), III (Test Locally Before Cloud — no local Windows equivalent
exists; see Testing above), IV (YAGNI — `profile` gains exactly one new live
value, no speculative Windows variants beyond what the legacy stack already
proved), XI (DRY — reuses the existing `windows`/`aws_ec2_windows` group_vars
and `setup-roles-windows.yml` unchanged, no duplicated role-application
logic), XII (Fail Loud — `profile=windows` on a non-`aws` provider, and a
missing `windows_admin_password`, are both rejected before any AWS API call).
FR-001 through FR-013 in [spec.md](./spec.md) are the binding requirements.

**Scale/Scope**: 2 existing task files modified
(`tasks/create/aws.yml` — add a Windows branch; `tasks/destroy/aws.yml` — add
`windows` group cleanup), `tasks/assert-provider-profile.yml` (extend the
`profile` value list and add the non-aws+windows rejection), 3 existing
playbooks re-scoped from `hosts: all` to `hosts: linux`
(`setup-users.yml`, `setup-basics.yml`, `reboot-if-required.yml`),
`configure-profile.yml` (add one unconditional `import_playbook:
setup-roles-windows.yml`). Zero new roles, zero new top-level playbooks.

**Structure Decision**: Single Ansible project, no new top-level directories,
roles, or playbooks. The feature is a mechanical extension of the existing
`provider=aws` create/destroy task-file pattern (one more inventory-group
key, mirroring `basic`/`desktop`), one widened security-group condition, a
re-scope of three Linux-only plays from `hosts: all` to `hosts: linux`
(behavior-preserving — every current target is already in the `linux`
group), and one new unconditional import of an already-existing,
already-correctly-scoped (`hosts: windows`) playbook. No legacy file
(`provisioners/aws-windows.yml`, `configure-windows.yml`,
`setup-roles-windows.yml`, `destroy-aws-tasks.yml`, or any role) is modified
or removed.
