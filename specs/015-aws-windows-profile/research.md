# Phase 0 Research: AWS Windows Profile for VM Lifecycle

No `NEEDS CLARIFICATION` markers remain in the Technical Context — the spec
(`spec.md`) and the existing legacy implementation
(`provisioners/aws-windows.yml`, `setup-roles-windows.yml`) already resolve
every open question. This document records the concrete technical decisions
derived from them, for traceability into Phase 1 design and `tasks.md`.

## Decision: Reuse `tasks/create/aws.yml`'s shape, branch on `profile == 'windows'`

- **Decision**: Add a Windows branch inside the existing
  `tasks/create/aws.yml`, reusing the same hostname-pool / autogenerated-
  inventory / block-rescue skeleton already used for AWS Linux VMs. The
  branch differs only in: (a) AMI/instance-type sourced from
  `inventories/group_vars/aws_ec2_windows/vars.yml` instead of
  `aws_ec2_linux`, (b) an `ec2_instance` `user_data` parameter carrying a
  PowerShell script that sets the Administrator password and authorizes the
  shared SSH public key, (c) the new host's `ansible_user` set to
  `Administrator` and written into the `windows` inventory group instead of
  `linux`.
- **Rationale**: Constitution IV (YAGNI) and XI (DRY) — a parallel
  `tasks/create/aws-windows.yml` file would duplicate the hostname-pool,
  inventory-load, and security-group steps already in `tasks/create/aws.yml`.
  Branching inside the existing file keeps the one-file-per-provider
  convention intact (`profile` already varies provider behavior today, e.g.
  the 3389 rule), and is a smaller diff than introducing a second AWS task
  file.
- **Alternatives considered**: A separate `tasks/create/aws-windows.yml`,
  selected via a `provider=aws-windows` pseudo-provider — rejected; it would
  require `assert-provider-profile.yml` to treat `aws-windows` as a fifth
  provider, contradicting the spec's framing of `windows` as a `profile`
  value (consistent with how `desktop` was added as a profile, not a
  provider, in `014-desktop-profile`).

## Decision: Windows EC2 access via SSH + user-data key injection, not WinRM

- **Decision**: Port `provisioners/aws-windows.yml`'s approach verbatim: the
  instance's `user_data` PowerShell script sets the Administrator password
  (from `windows_admin_password`), enables and starts the OpenSSH Server
  service, sets PowerShell as the default SSH shell, and writes the shared
  public key (`my_ssh_public_key`) into
  `C:\ProgramData\ssh\administrators_authorized_keys` with `icacls`-restricted
  permissions. The new host connects via plain SSH (`ansible_connection: ssh`,
  inherited from `inventories/group_vars/windows/vars.yml`'s
  `ansible_shell_type: powershell`), the same connection family already used
  for every other AWS VM.
- **Rationale**: This is the connection method already proven by the legacy
  implementation (`docs/techContext.md` § Windows Server Implementation) and
  already reflected in the existing `windows`/`aws_ec2_windows` group_vars
  (`ansible_shell_type: powershell`, no `ansible_connection: winrm` anywhere
  in the repo). Reusing it avoids introducing a second connection family
  (WinRM) and its additional collection dependency
  (`community.windows`/`pywinrm`) purely for this feature, when SSH already
  works end-to-end today.
- **Alternatives considered**: WinRM (the very first, now-superseded MVP
  approach, commit `e90285b`) — rejected; superseded by the SSH-based
  approach in the repo's own history, and reintroducing it would contradict
  the already-established `windows` group_vars.

## Decision: Widen the existing 3389 condition instead of adding a second rule

- **Decision**: Change `tasks/create/aws.yml`'s security-group rule list
  comprehension from `if profile == 'desktop' else []` to
  `if profile in ['desktop', 'windows'] else []`, keeping one rule definition.
- **Rationale**: FR-006 requires TCP 3389 open for `profile=windows`,
  identically to the existing `profile=desktop` rule from
  `014-desktop-profile`. Widening the existing condition is the smallest
  change satisfying XI (DRY) — two near-identical rule blocks differing only
  in their `when`-equivalent condition would be a needless duplication.
- **Alternatives considered**: A separate rule object for `profile=windows`
  — rejected; the rule content (proto/port/cidr) is byte-for-byte identical
  to the existing desktop rule, so a second copy would drift if one were
  edited without the other.

## Decision: Re-scope three Linux-only plays from `hosts: all` to `hosts: linux`

- **Decision**: `playbooks/setup-users.yml`, `playbooks/setup-basics.yml`,
  and `playbooks/reboot-if-required.yml` each change their play's `hosts:`
  from `all` to `linux`, before `configure-profile.yml` gains any Windows
  wiring.
- **Rationale**: All three use modules that only work over a POSIX/Python
  connection (`ansible.builtin.user`, `ansible.builtin.apt`,
  `ansible.builtin.stat` against `/var/run/reboot-required`, `lineinfile`,
  `authorized_key`) and would fail outright against a `ansible_connection:
  ssh` + `ansible_shell_type: powershell` Windows host. `import_playbook`
  cannot take a `when:` condition, so the only behavior-preserving fix is to
  scope the play itself to the `linux` group — which every current target is
  already a member of, making this a no-op for existing hosts (same pattern
  `014-desktop-profile`'s FR-003 already established for
  `configure-profile-roles.yml`).
- **Alternatives considered**: Wrap each `import_playbook` in
  `configure-profile.yml` with a `when: "'windows' not in group_names"`
  guard — rejected; `import_playbook` does not support `when:` at all (it is
  evaluated at parse time, not play time), so this alternative is not
  actually available in Ansible.

## Decision: Reuse `setup-roles-windows.yml` unconditionally, no new guard needed

- **Decision**: `configure-profile.yml` gains one new line:
  `- import_playbook: ../setup-roles-windows.yml` (relative path from
  `playbooks/`), added without any `when:` condition.
- **Rationale**: `setup-roles-windows.yml` already declares `hosts: windows`.
  Ansible plays silently skip hosts outside their declared group, so no
  conditional guard is needed — unlike the `014-desktop-profile` legacy
  desktop playbooks (`setup-desktop.yml` etc.), which use `hosts: all`/
  `hosts: linux` and therefore needed an explicit `when: "'desktop' in
  group_names"` guard at the import site. `setup-roles-windows.yml` does not
  share that problem.
- **Alternatives considered**: Duplicating the `windows_foundation`/
  `win_ai_agent` role list directly inside `configure-profile.yml` instead of
  importing the existing playbook — rejected; violates XI (DRY) for no
  benefit, since `setup-roles-windows.yml` already exists, is already
  correctly scoped, and is reused unmodified by FR-010/FR-013.

## Decision: Symmetric `windows` group cleanup in `tasks/destroy/aws.yml`

- **Decision**: `tasks/destroy/aws.yml`'s `cleaned_inventory` fact gains a
  `windows` key built with the same
  `dict2items | selectattr('key', '!=', hostname) | list | items2dict`
  pattern already used for `all`/`linux`/`aws`/`basic`/`desktop`.
- **Rationale**: Destroy infers group membership from the existing inventory
  entry rather than taking a `profile` extra-var (consistent with
  `014-desktop-profile`'s precedent — `destroy-vm.yml` has no `profile`
  parameter today and this feature does not add one). Adding the `windows`
  key alongside the existing keys is the minimal, idempotent extension.
- **Alternatives considered**: Inferring which group(s) to clean generically
  from whatever keys exist in the loaded inventory, instead of a fixed list
  — rejected as out of scope; the existing code already hardcodes the
  `basic`/`desktop` keys explicitly, so this feature follows the same
  established pattern rather than refactoring it.

## Decision: Boot-readiness reuses `wait_for_connection`, no WinRM wait added

- **Decision**: The Windows branch reuses `tasks/create/aws.yml`'s existing
  `ansible.builtin.wait_for_connection` step (delegated to the new instance's
  IP), with `ansible_user: Administrator` and a longer `timeout` to
  accommodate Windows's slower boot — mirroring
  `provisioners/aws-windows.yml`'s own longer `wait_timeout` on the
  `ec2_instance` task itself.
- **Rationale**: `wait_for_connection` works for any `ansible_connection: ssh`
  target regardless of guest OS; since this feature's connection method is
  SSH (not WinRM — see decision above), no separate WinRM/RDP readiness
  check is needed.
- **Alternatives considered**: A dedicated `wait_for: port=3389` (RDP)
  readiness check before declaring the instance ready — rejected; RDP
  readiness is not required for `configure-profile.yml` to proceed (it
  connects over SSH), and the legacy implementation did not gate on RDP
  readiness either.
