# Tasks: AWS Windows Profile for Create, Destroy, and Configure VM Playbooks

**Input**: Design documents from `/specs/015-aws-windows-profile/`

**Prerequisites**: plan.md, spec.md, research.md, data-model.md, contracts/

**Tests**: Not requested in spec.md. Validation is via the spec's
"Independent Test" sections (real AWS runs per Constitution Principle III —
Windows has no local equivalent in this project's toolchain) — no Molecule
scenario applies (these are orchestration playbooks/task files, not roles).

**Organization**: Tasks are grouped by user story (US1/US2/US3/US4 from
spec.md) to enable independent implementation and testing of each.

## Format: `[ID] [P?] [Story] Description`

- **[P]**: Can run in parallel (different files, no dependencies)
- **[Story]**: Which user story this task belongs to (US1/US2/US3/US4)
- File paths are repository-relative

## Phase 3: User Story 1 - Create an AWS Windows Server VM (Priority: P1) 🎯 MVP

**Goal**: `create-vm.yml -e provider=aws -e profile=windows` creates a
reachable (SSH + RDP) Windows Server EC2 instance, registered in a `windows`
inventory group distinct from `linux`/`basic`/`desktop`.

**Independent Test**: Run
`create-vm.yml -e provider=aws -e profile=windows`. Verify the command
completes, prints a hostname, the host is reachable via SSH (PowerShell
shell, `ansible_user: Administrator`), and the host appears under `windows`
(not `linux`) in `inventories/aws_autogenerated.yml`.

### Implementation for User Story 1

- [ ] T002 [US1] Add a fail-loud assertion for `windows_admin_password`
      (defined and non-empty) in `playbooks/tasks/create/aws.yml`, run
      before any AWS API call when `profile == 'windows'` (depends on T001)
- [ ] T003 [US1] Add a `profile == 'windows'` branch in
      `playbooks/tasks/create/aws.yml` that resolves AMI/instance-type
      defaults from `inventories/group_vars/aws_ec2_windows/vars.yml`
      instead of `aws_ec2_linux` (depends on T001)
- [ ] T004 [US1] In that branch, generate the Windows `user_data` PowerShell
      script (Administrator password from `windows_admin_password`, shared
      SSH public key authorized via
      `C:\ProgramData\ssh\administrators_authorized_keys`, OpenSSH service
      enabled, PowerShell set as default SSH shell), porting
      `provisioners/aws-windows.yml`'s script verbatim (depends on T002)
- [ ] T005 [US1] Pass the generated `user_data` into the existing
      `amazon.aws.ec2_instance` task in `playbooks/tasks/create/aws.yml`,
      conditioned on `profile == 'windows'` (depends on T004)
- [ ] T006 [US1] Widen the TCP 3389 security-group rule condition in
      `playbooks/tasks/create/aws.yml` from `profile == 'desktop'` to
      `profile in ['desktop', 'windows']` (depends on T001)
- [ ] T007 [US1] In `playbooks/tasks/create/aws.yml`'s `updated_inventory`
      build, register the new host under a `windows` group key (instead of
      `linux`) when `profile == 'windows'`, with `ansible_user: Administrator`
      instead of `ubuntu` (depends on T003)
- [ ] T008 [US1] Validate end-to-end on real AWS (Constitution Principle
      III — no local Windows equivalent exists):
      `create-vm.yml -e provider=aws -e profile=windows`; confirm the
      instance launches with the Windows AMI, SSH (PowerShell,
      `Administrator`) and RDP (3389) are both reachable, and the host is
      registered under `windows` (not `linux`) in
      `inventories/aws_autogenerated.yml`. Additionally, in the same
      session, run `create-vm.yml -e provider=aws -e profile=basic` and
      confirm the widened T006 condition did NOT add a 3389 rule for this
      basic VM — `ansible-sg` still allows 3389 only because of the windows
      VM above, never because of `profile=basic` itself (SC-005 regression
      check) (depends on T002-T007)

**Checkpoint**: User Story 1 is fully functional and independently
testable — an AWS Windows VM is reachable through the unified
`create-vm.yml` command for the first time.

## Phase 4: User Story 2 - Destroy an aws/windows VM by hostname (Priority: P2)

**Goal**: `destroy-vm.yml -e provider=aws -e hostname=<name>` terminates a
Windows VM and removes it from the `windows` inventory group, using the same
command shape already used for aws/linux VMs.

**Independent Test**: Create an aws/windows VM (US1), note its hostname,
run `destroy-vm.yml -e provider=aws -e hostname=<name>`. Verify the EC2
instance is terminated and the hostname no longer appears anywhere in
`inventories/aws_autogenerated.yml`.

### Implementation for User Story 2

- [ ] T009 [US2] [P] Add a `windows` group-key cleanup (same
      `dict2items | selectattr('key', '!=', hostname) | list | items2dict`
      pattern already used for `basic`/`desktop`/`linux`) to
      `playbooks/tasks/destroy/aws.yml`'s `cleaned_inventory` build (depends
      on T001)
- [ ] T010 [US2] Validate on real AWS: destroy an aws/windows VM created in
      T008; confirm the EC2 instance is terminated, its known_hosts entry is
      removed, and it is removed from `all`, `aws`, and `windows` in
      `inventories/aws_autogenerated.yml`; confirm a coexisting aws/linux
      VM's entry is untouched (depends on T008, T009)

**Checkpoint**: User Stories 1 AND 2 both work independently — full
create/destroy lifecycle parity for aws/windows.

## Phase 5: User Story 3 - Configure an aws/windows VM (Priority: P3)

**Goal**: `configure-profile.yml` applies the `windows_foundation` +
`win_ai_agent` role set to `windows`-group hosts, and does not apply any of
the Linux-only plays to them.

**Independent Test**: Run `configure-profile.yml` against a host in the
`windows` inventory group. Verify it receives the Windows role set and that
none of `setup-users.yml`/`setup-basics.yml`/`reboot-if-required.yml`/the
`basic`/`desktop` role lists run against it.

### Implementation for User Story 3

- [ ] T011 [US3] [P] Re-scope `playbooks/setup-users.yml` from
      `hosts: all` to `hosts: linux` (behavior-preserving — every current
      target is already a `linux`-group member)
- [ ] T012 [US3] [P] Re-scope `playbooks/setup-basics.yml` from
      `hosts: all` to `hosts: linux` (same justification)
- [ ] T013 [US3] [P] Re-scope `playbooks/reboot-if-required.yml` from
      `hosts: all` to `hosts: linux` (same justification)
- [ ] T013a [US3] Validate FR-012: run `configure-profile.yml` against an
      existing `basic`- or `desktop`-group host before and after T011-T013
      land; diff the tasks/files touched — confirm identical behavior
      (depends on T011-T013)
- [ ] T014 [US3] Add `- import_playbook: ../setup-roles-windows.yml` to

      `setup-roles-windows.yml` already declares `hosts: windows`)
- [ ] T015 [US3] Validate end-to-end on the real AWS Windows host from T008

      of the Linux-only plays or role lists run against it (depends on
      T013a, T014)

**Checkpoint**: All three capability-delivering user stories (create,
destroy, configure) are independently functional.

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: N/A
- **Foundational (Phase 2)**: No dependencies beyond the existing
  codebase — BLOCKS all user stories
- **User Stories (Phase 3-6)**: All depend on Foundational completion
  (T001). US3 (configure) and US4 (reject) are independent of US1/US2 in
  code, but their *validation* tasks (T015, T017) reuse the AWS host T008
  creates, so they are listed after US1/US2 for a practical single-engineer
  sequence — a second engineer could implement US3/US4's code changes (not
  their AWS validation step) in parallel with US1/US2
- **Polish (Phase 7)**: Depends on all desired user stories being complete

### User Story Dependencies

- **User Story 1 (P1)**: Depends on Foundational only
- **User Story 2 (P2)**: Depends on Foundational (T001); its validation
  step (T010) needs a host from US1's validation (T008)
- **User Story 3 (P3)**: Depends on Foundational (T001) only for its code
  changes (T011-T014); its validation step (T015) needs a host from US1's
  validation (T008)
- **User Story 4 (P4)**: Depends on Foundational (T001) only for its code
  change (T016); its validation step (T017) cross-checks against US1's
  validation (T008)

### Parallel Opportunities

- T011, T012, T013 (the three `hosts: all` → `hosts: linux` re-scopes) can
  all run in parallel — three distinct files, no shared state
- T009 (destroy/aws.yml) can run in parallel with T002-T007 (create/aws.yml)
  and with T011-T014 (configure-profile.yml + the three re-scoped
  playbooks) — all touch disjoint files
- T016 (assert-provider-profile.yml's rejection) can run in parallel with
  T002-T014 — disjoint file
- T018 and T019 (Polish) can run in parallel

## Implementation Strategy

### MVP First (User Story 1 Only)

1. Complete Phase 2: Foundational
2. Complete Phase 3: User Story 1
3. **STOP and VALIDATE**: T008 confirms an AWS Windows VM is created and
   reachable end-to-end
4. This alone delivers most of the feature's value — a reachable Windows
   box via the unified command, matching the legacy MVP's own staged
   rollout (provisioning before configuration)

### Incremental Delivery

1. Foundational → `profile=windows` passes the generic value check
2. US1 → AWS Windows VM creation works through `create-vm.yml` (MVP)
3. US2 → destroy lifecycle parity, closing the cost-control loop
4. US3 → the VM becomes actually useful (Claude Code + tooling installed)
5. US4 → safety guard against the one invalid combination
